COVERAGE OVERVIEW

六类测试，覆盖到用户可见结果

不是用一个“全绿”数字代替质量判断，而是把规则、协作、接口、用户操作、布局和攻击面分别验证。

为什么总计是 208，而不是自动化 test() 数量？
208 是面向需求与验收拆分的场景数；一个自动化测试块可能同时断言签发、台账、总览与日志，因此代码中的 test() 数量可以小于验收场景数。
高覆盖的判断标准
同一规则从不同层次验证，并同时覆盖正常路径、异常输入、状态边界和安全攻击面；“数量多”本身不等于高质量。

例：机构 DID 停用

  • 规则层：状态变为 deactivated 且不可逆。
  • 服务层：禁止更新、轮换和签发。
  • 接口层：返回受控错误，不产生半写入。
  • 功能层：停用后从签发下拉列表消失。
  • 验证层：历史签名可正确，但整体验证失败。
  • 日志层：记录动作、对象与关联 ID。
成功路径如何验收？
不仅断言返回值，还检查持久化状态、页面刷新、台账同步和审计日志是否一致。
失败路径如何验收？
检查请求被明确拒绝、错误原因可解释、状态没有副作用，并且响应和日志不泄露敏感信息。

DID & METHODS

先验证“谁”，再验证“凭什么”

Issuer

创建机构身份，DID Document 提供 assertionMethod，私钥用于 VC 签发。

Holder

创建持有者身份，成为 credentialSubject；角色错用必须被拒绝。

DID Document

检查 verificationMethod、公钥 JWK、authentication 与 assertionMethod，且不得包含私钥。

did:example：可管理生命周期
  • 支持名称与服务地址更新。
  • 支持密钥轮换并保留历史公钥。
  • 支持不可逆停用。
  • 版本冲突必须明确拒绝。
did:key：由公钥直接派生
  • 验证 Ed25519 公钥指纹和 Base58BTC 编码。
  • 可以参与签发与验签。
  • 项目中不支持更新、轮换和停用，页面不显示这些操作。
未知 DID Method
提交未注册 Method 时必须失败，不能静默降级，也不能生成格式错误的 DID。
私钥暴露检查
公开 API、页面 DID Document 和历史密钥中均不得出现 privateJwk 或 JWK 私钥字段 d。

DID LIFECYCLE

版本、密钥与状态必须保持历史一致性

CREATEactive · v1
→
UPDATE乐观锁
→
ROTATEkey v2
→
HISTORY保留旧公钥
→
DEACTIVATE不可逆
更新与版本冲突
两个客户端都拿到版本 1：第一个更新成功后版本变成 2，第二个继续提交 expectedVersion=1 时返回冲突，防止静默覆盖。
密钥轮换与历史 VC
轮换后 keyVersion 递增，旧公钥与 verificationMethod 进入 keyHistory。旧 VC 按 proof.keyVersion 找历史公钥，新 VC 使用当前公钥。
停用后的签发限制
停用的 Issuer 或 Holder 均不得参与新签发；停用后更新、再次轮换、再次停用也必须失败。
密码学有效 ≠ 业务有效
DID 停用不会改变历史签名字节，因此历史 VC 的签名项仍可通过；但 DID 当前状态项失败，所以最终 valid=false。

VC ISSUANCE

签发测试覆盖数据、角色、时间与证明

正常签发链路

  1. 解析有效 Issuer 与 Holder。
  2. 构造 credentialSubject。
  3. 稳定序列化未签名 VC。
  4. 使用 Ed25519 私钥签名。
  5. 保存 active 台账与审计记录。

异常输入矩阵

  • Issuer / Holder 不存在或角色互换。
  • 姓名、课程为空。
  • 有效期格式错误或早于当前时间。
  • 任一 DID 已停用。
  • 失败时不产生半成品记录。
VC 结构断言
检查 @context、id、type、issuer、validFrom、validUntil、credentialSubject 和 proof；同时检查 ID 唯一、Subject 指向 Holder。
proof 结构断言
检查 verificationMethod 属于 Issuer、proofPurpose 为 assertionMethod、keyVersion 正确且 proofValue 存在。
签发后的多界面同步
凭证 JSON 预览、签发页台账、总览台账、有效 VC 数量和日志记录必须一致更新。
私钥保护边界
签名在服务端完成。返回结果可以包含公钥和签名，但不能包含 privateJwk 或私钥字段 d。

SEVEN-CHECK VERIFICATION

签名只是七项验证中的一项

格式完整性必要字段与 proofValue 是否存在
Issuer DID 解析能否找到签发方 DID Document
DID 当前状态Issuer 是否存在且未停用
密钥版本当前或历史公钥与 verificationMethod 是否匹配
Ed25519 签名来源可信且内容未被修改
有效期validFrom ≤ now ≤ validUntil
凭证当前状态本地台账状态必须为 active
最终结论七项全部通过才返回 valid=true
篡改检测覆盖
分别修改学员姓名、课程名称、Issuer 和 proofValue，验证签名或签发方检查必须失败。
失败原因可解释
结果保留每个 check 的 key、passed 和 message，页面可区分格式错误、密钥错误、签名错误、过期或撤销。

VC LIFECYCLE

状态机约束允许的每一次变化

active当前有效
suspended可恢复
replaced终态
revoked终态
expired时间终态
签名正确不等于状态有效：暂停、替换、撤销或过期的 VC 都必须在完整验证中失败。
暂停与恢复
active → suspended → active。检查时间戳、台账按钮变化，以及暂停后失败、恢复后重新通过。
替代更新
不能直接修改已签名 VC；系统签发新 ID，旧凭证标记 replaced，通过 replaces / replacedBy 保留双向关系。
撤销不可逆
active 和 suspended 均可撤销；撤销后不能恢复、暂停或替换，签名可能仍正确但状态项失败。
自动过期
当前时间超过 validUntil 时动态显示 expired；revoked 和 replaced 等明确终态不会被错误覆盖。

LEDGER, QUERY & AUDIT

结果要能被查到、解释和追溯

凭证台账

历史 VC、状态同步、完整 JSON、合法操作按钮、长 ID 安全换行。

验证记录

成功备注、单项/多项失败原因、未知原因兜底、最近活动。

结构化日志

动作、模块、级别、关联 ID、时间范围与组合筛选。

搜索与分页边界
去首尾空格、忽略大小写、多字段模糊匹配、稳定排序、51 条分页、超界页修正、空结果第一页、页大小规范化。
日志组合筛选
不同条件之间使用 AND，同一关键词在多个文本字段之间使用 OR；开始和结束时间包含边界，倒置时间范围必须拒绝。
日志清空也要留痕
取消确认时状态不变；确认后清除旧日志，但保留一条 LOG_CLEAR 摘要记录清理数量。
并发持久化回归
并发创建两个 DID 后必须保留两条记录。修复点是把“读取—修改—写入”整个事务串行化，而非只串行最终 save。

SECURITY BOUNDARIES

主动输入攻击数据，验证系统不会越界

数据保护

私钥、Token、Authorization、完整 proof、凭证和错误栈不得出现在公开响应或日志中。

输入攻击

存储型 XSS、原型污染、目录穿越和非法 JSON 均有针对性载荷。

协议加固

Content-Type、1 MB 请求限制、不支持方法、CSP、nosniff 和 Referrer-Policy。

递归日志脱敏
测试顶层、嵌套对象和数组中的 privateJwk、token、Authorization、proof 与 credential，统一替换为 [REDACTED]。
存储型 XSS
保存包含 <img onerror=...> 的名称，再通过 UI helper 渲染，断言标签被转义且不会执行。
原型污染与目录穿越
提交 __proto__ 载荷后全局对象不得被污染；编码和未编码的 ../package.json 请求都不能读取项目文件。
错误响应与副作用
危险 HTTP 方法不仅要返回 404/405，还要比较操作前后状态，证明报错请求没有偷偷修改数据。

UX & TRUSTED EVIDENCE

测试结果本身也需要可复核

用户界面与兼容性

  • 桌面五个入口均可导航。
  • 移动端无不可控横向溢出。
  • 表单、导航和点击区域保持可用。
  • 空状态、错误提示和取消确认有明确反馈。
  • 长 DID、VC ID 和失败原因不破坏布局。

永久测试证据

  • Node 与 Chromium 原始日志。
  • Git 提交、环境、耗时与退出码。
  • Playwright 报告、截图、视频与 trace。
  • 结果摘要、用例清单和文件哈希。
  • 证据被修改后可通过校验和发现。
为什么“AI 说通过”不算证据？
AI 可以生成和解释测试，但最终结论必须能追踪到测试脚本、框架退出码、原始日志、执行环境和代码版本。
失败阶段仍继续收集证据
即使 Node 阶段失败，UI 阶段仍继续执行；最终由两个阶段和证据完整性共同决定批次状态。
通过的严格条件
退出码为零、结果可解析、失败/跳过/todo 均为零且证据文件完整，才标记为可信通过。
工作区状态也被记录
存在未提交修改时仍可测试，但 metadata 必须标记“含未提交修改”，避免结果与代码版本无法对应。

CONCLUSION & BOUNDARIES

高覆盖不等于无限覆盖

本项目证明的是：在本地教学范围内，DID、密钥、VC、七项验证、生命周期、安全边界和用户操作形成了可执行、可解释、可追溯的测试闭环。

已经验证

  • DID Method 与完整生命周期。
  • Ed25519 签发、篡改检测和历史密钥。
  • VC 七项验证与状态机。
  • 台账、搜索、日志、移动布局和安全输入。
  • 本地永久测试证据链。

尚未验证的生产边界

  • 正式 W3C cryptosuite 跨系统互操作。
  • 公网 TLS、完整认证授权和渗透测试。
  • 生产级 KMS/HSM 密钥托管。
  • 压力测试、拒绝服务和多浏览器矩阵。
  • 真实 DID Resolver、链上注册和标准状态列表。
答辩结论：我们没有用“签名通过”代替完整验证，也没有用“自动化全绿”代替测试可信度；每个结论都可以回到规则、断言和证据。
