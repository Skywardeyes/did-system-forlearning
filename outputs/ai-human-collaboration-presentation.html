CENTRAL TAKEAWAY

效率交给 AI，责任仍由人工承担

AI 提高分析、实施和验证效率；人工控制目标、边界、风险和最终质量。

不是“一句话生成项目”

需求会被修正，方案会被否决，测试会因用户视角不足而补充。

不是“测试全绿即完成”

执行结果必须对应当前代码、环境与验收标准，并保留可核对证据。

不是“AI 替代负责人”

范围、业务语义、风险接受和最终交付结论必须由人工确认。

协作价值不在于消除人的责任，而在于让人的决策与 AI 的执行都能被验证、复核和回溯。

ROLE BOUNDARIES

四类角色各司其职，避免同源偏差

人工需求方与评审者

确定目标、范围、优先级和验收标准，承担最终决策。

需求分析 AI

检查现状、识别冲突、比较方案并维护文档一致性。

开发 Agent

依据批准方案实现代码，用专项和回归测试证明变更。

测试 Agent

从需求和风险独立设计正常、异常、边界、安全与 UI 场景。

人工不能只凭偏好指定技术
人工决策需要建立在技术事实和权衡上，但业务范围与风险接受仍不能交给 AI 独立决定。
测试 Agent 为什么要保持独立
如果测试只复述开发 Agent 的实现逻辑，同一错误可能同时进入代码和断言，形成“自证正确”。

STAGES 1—3 · REQUIREMENT BASELINE

需求先被质疑，再成为实施基线

阶段 1 · 生成初稿

读取源码、API、测试和现有文档，形成目标、角色、流程与待确认问题。

阶段 2 · 人工修正

补充 DID/VC 生命周期、成功与失败日志、隐私约束及优先级。

阶段 3 · 重建基线

统一范围、状态机、数据模型、API、页面、安全和完成标准。

阶段输入与输出
输入由初始目标逐步扩展为现状证据、人工意见和约束；输出从需求初稿升级为可实施、可测试的权威需求。
人工确认点
阶段 1 只确认“可讨论”；阶段 2 确认范围优先级；阶段 3 才确认它能成为技术、开发和测试的唯一基线。
为什么生命周期必须提前定义
DID 停用不可逆、VC 撤销不可恢复等语义会同时影响状态机、API、页面入口、日志与测试，不能留到编码时猜测。
回退规则
范围不清回到阶段 2；章节冲突留在阶段 3 重写；权威需求未通过前不向 Agent 下发实现任务。
05 / 13

STAGES 4—5 · DUAL-LINE DESIGN

“如何实现”和“如何证明”同时设计

阶段 4 · 技术工作线

  • 比较 DID Method、数据模型与模块边界。
  • 评估依赖、迁移、复杂度和教学展示约束。
  • 输出推荐架构与接口契约。

阶段 4 · 测试工作线

  • 从需求和风险设计正常、异常、边界与安全场景。
  • 覆盖服务、API、用户功能、UI 与端到端流程。
  • 输出测试矩阵和高风险清单。
阶段 5 · 人工批准：确认架构、Method 能力、状态机、测试层次和本地教学边界，将建议转化为可下发基线。
测试为什么不能等代码写完
测试方案提前暴露“需求无法验收”与“风险没有断言”的问题，也防止测试围绕现有实现被动补写。
方案不通过如何回退
技术不可行回到阶段 4 比较替代方案；测试与需求无法对应则回到阶段 3 澄清完成标准。

STAGE 6 · PARALLEL AGENTS

共享契约，隔离视角

开发 Agent：实现功能

依据权威需求和技术设计，以失败测试、最小实现、专项验证和分阶段提交推进代码。

代码模块集成回归修复

测试 Agent：独立证明

优先读取需求与风险，不照抄代码；构造失败路径、安全边界和真实用户操作。

测试矩阵证据缺口报告
双方共享什么
批准后的需求、API 与数据模型、状态机、错误码、测试数据边界、任务依赖和文件所有权。
双方不共享什么偏见
测试 Agent 不把开发实现当作业务真相；它验证 DID 停用不可逆、禁止新签发、历史签名计算结果与完整业务有效性之间的差异。
契约理解不一致怎么办
停止各自猜测，将分歧上报协调 AI 与人工评审者，由人工确认业务语义，再同步修订需求、设计和测试。
人工门禁
检查分工是否清晰、敏感功能是否有独立失败测试，以及两条工作线是否对齐需求但没有互相复制。
07 / 13

STAGES 7—9 · QUALITY LOOP

测试的价值是发现缺口，不是堆高数量

阶段 7

人工评审覆盖性，补充用户旅程、安全边界与页面结果。

阶段 8

AI 补充用例并执行单元、集成、API、功能、安全和 UI 测试。

失败分类

区分功能缺陷、用例错误、环境问题和需求歧义。

阶段 9

复现、追踪根因、最小修复并保留防回归断言。

证据门禁

专项与全量回归必须对应当前代码和执行环境。

人工重点检查哪些覆盖
主要用户旅程、七项完整验证、生命周期终态、安全边界、空状态、错误提示、移动布局和页面可见结果。
修复为什么要从稳定复现开始
先证明原失败，再沿数据流找根因；不能用放宽断言、删除测试或叠加猜测性补丁换取“全绿”。
08 / 13

STAGE 10 · HUMAN ACCEPTANCE

自动化达标之后，项目才进入人工验收

真实操作

创建 DID、签发 VC、查看七项验证、篡改、生命周期、选择性披露、日志和重置。

真实界面

检查桌面和移动视口、按钮能力、错误提示、台账同步和可讲解性。

真实结论

人工签署通过或不通过，记录剩余问题、已知限制和交付意见。

验收不通过时回到哪里
UI 或功能缺陷回到阶段 9；用例遗漏回到阶段 7—8；需求偏离真实目标回到阶段 2—3。
AI 在验收中的边界
AI 可以准备环境、数据、清单和证据导航，但不能替代人工宣布项目验收通过。

PROJECT CASES · REQUIREMENT CORRECTION

真实冲突推动需求变得更准确

DID Method 选型纠偏

原需求希望同一 DID 轮换密钥，但 did:key 由公钥派生，换公钥会得到新 DID。

结论：用 did:example 演示更新、轮换、停用；保留 did:key 演示自认证与不可更新。

日志需求逐步完整化

“查看成功和失败记录”被拆解为审计日志与系统日志，再补充级别、关联 ID、组合筛选、独立存储、留存上限、清空摘要和递归脱敏。

结论：一句需求演进为可实施、可测试的完整规范。

为什么不是用代码技巧绕过 Method 冲突
概念上矛盾的需求必须先修正规范；如果实现掩盖标准边界，演示得到的只是错误认知。
日志为何必须同时覆盖成功与失败
只有失败日志无法重建完整业务轨迹；只有成功日志则无法定位安全拒绝和系统异常，两者都需关联到同一次请求。
10 / 13

PROJECT CASES · TEST & DEFECT FEEDBACK

测试、缺陷与页面反馈反向完善工程

测试覆盖由人工反向补全

API 能暂停 VC，不代表用户能在台账操作并看到状态同步；因此补充用户功能、空状态、错误提示和移动端用例。

并发写入缺陷

并发创建两个 DID 后只保留一条。根因是只串行化最终写入，没有串行化“读取—修改—写入”事务。

真实浏览器验收

逻辑测试通过后仍发现隐藏单选框和日志筛选栏撑开页面，由响应式修复和布局回归用例共同闭环。

并发修复保留了什么证据
集成测试保留并发回归用例，证明修复针对数据完整性根因，而不是偶然改变执行时序。
为什么浏览器检查不可替代
单元测试证明逻辑行为，真实浏览器证明用户看见的最终布局与交互；两者验证对象不同。

BOUNDARIES & CONCLUSION

人工不可替代的决策，决定协作是否可信

必须由人工承担

  • 项目目标、服务对象、范围与优先级。
  • DID 停用、VC 撤销、隐私披露等业务语义。
  • 风险是否可接受、页面是否符合使用预期。
  • 已知限制和最终验收结论。

AI / Agent 不得越界

  • 未经授权扩大范围或用技术偏好替代业务决策。
  • 删除有价值断言换取测试全绿。
  • 用一次运行代替完整验证。
  • 把本地教学实现宣传为生产级标准系统。

最终交付的不是一段 AI 文本，而是一条可验证、可复核、可回溯的工程链路。

决策可确认执行可验证问题可回退
