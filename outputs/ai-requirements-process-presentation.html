STARTING POINT

DID / VC 主流程

创建身份、签发凭证和验证签名。

生命周期

DID 更新与停用，VC 暂停、恢复、替代和撤销。

分页与排序

数据量增大后分页，最新记录优先。

AI 的第一步不是写需求
先读取源码、接口、测试和既有文档，确认系统真实状态，防止把已经存在的能力再次规划，或提出与架构冲突的要求。
四类能力如何区分
  • 当前已实现能力；
  • 计划扩展能力；
  • 可以继续沿用的实现；
  • 需要补充的新状态、接口和测试。
关键原则：基于现有项目生成需求，而不是脱离代码写一份通用 DID/VC 方案。

AI 负责

  • 检索现状与冲突。
  • 拆解问题与备选方案。
  • 分析权衡和潜在边界。
  • 同步需求、设计和测试。

用户负责

  • 确认业务目标和优先级。
  • 决定关键状态规则。
  • 选择技术与产品方案。
  • 承担最终验收判断。
为什么不能让 AI 擅自确定规则？
例如“DID 停用后历史 VC 是否整体有效”属于信任策略，不是算法自动给出的唯一答案，必须由项目目标和用户确认决定。
什么时候回溯修改需求？
实施发现标准冲突、测试暴露遗漏或真实页面出现问题时，先判断需求或设计是否有误，再同步修订代码和文档。
03 / 11

LIFECYCLE RULES

状态不是字段，而是受约束的规则

DID 生命周期

创建 → 正常使用 → 信息更新 → 密钥轮换 → 停用

  • 停用不可逆，之后不能更新或轮换。
  • 更新与轮换增加版本号。
  • 历史公钥保留，历史私钥不公开。
  • 停用后不能参与新的 VC 签发。

VC 生命周期

active ↔ suspended
active/suspended → replaced
active/suspended → expired
active/suspended → revoked

  • 暂停可以恢复，终态不可恢复。
  • 更新通过重新签发完成。
  • 新旧 VC 保存替代关系。
为什么 VC 不能原地修改？
VC JSON 已被数字签名覆盖，修改声明会破坏签名。更新必须签发新 ID，并将旧 VC 标记为 replaced。
为什么保留历史公钥？
密钥轮换前签发的 VC 需要使用当时的公钥验证；删除旧公钥会导致历史凭证失去可验证性。
业务状态与密码学结果
签名有效只证明内容未被篡改且对应某把私钥；暂停、撤销、过期或 DID 停用仍会让业务验证失败。
测试如何映射状态机？
每条允许转换验证成功结果，每条非法转换验证明确拒绝、状态不变和日志可追踪。

新增的架构能力

组合场景

未知 Method 如何处理？
必须明确拒绝，不能静默退回默认 Method，也不能让 DID 前缀和适配器不一致。
05 / 11

LIST CONTRACT

“支持搜索分页”被细化为可测试契约

搜索

主要业务字段模糊匹配，去除首尾空格，英文忽略大小写。

稳定排序

业务时间倒序；时间相同再按记录 ID 倒序。

分页

默认 10 条，可选 10 / 20 / 50，变化后回到第一页。

空状态

区分原始数据为空与搜索无结果，给出不同提示。

页码越界规则
当前页超过总页数时自动修正为最后一个有效页；空结果保持第 1 页而不是第 0 页。
边界数据量
分页边界：0、1、10、11、20、21、50、51 条，分别覆盖空集、刚好一页、跨页和页大小边界。
为什么需要稳定排序？
如果多条记录时间相同却没有第二排序键，多次请求可能产生不同顺序，导致翻页重复或遗漏。
覆盖哪些列表？
DID 列表、VC 台账、验证记录和结构化日志都采用统一查询契约，并在页面验证搜索、清除、分页和错误保留。
06 / 11

TEST STRATEGY

比较三种组织方式，再选择混合方案

分层测试矩阵

定位精确，适合验证规则和接口，但容易缺少用户完整旅程。

完整业务场景

贴近真实使用，但失败时定位较慢，底层边界可能不充分。

按页面功能划分

便于人工验收，但无法独立证明密码学与服务规则。

用户确认的最终方案：分层测试矩阵 + 端到端主流程。
服务 / API 层
验证状态机、签名、搜索、排序、分页、错误码和敏感信息保护。
前端交互层
验证按钮状态、表单、筛选、分页、空状态、错误提示和不可逆操作确认。
端到端层
从 DID 创建、VC 签发和七项验证一直走到暂停、恢复、替代、撤销或停用。
统一用例字段
编号、模块、优先级、层级、前置条件、数据、步骤、预期结果及自动化映射，让需求能够追踪到断言。
07 / 11

LOGGING EVOLUTION

一句“查看记录”，演进成完整日志规范

提出需求查看操作成功和失败记录
澄清范围审计日志 + 系统日志
定义结构级别、模块、动作、关联 ID
加入安全递归脱敏和独立存储
同步交付需求、设计、测试、README

查询与留存

  • info / warn / error 三级及文字标识。
  • 文本、类型、结果、级别、模块、时间组合筛选。
  • 独立 data/logs.json。
  • 最多保留最新 5,000 条。

隐私与清理

  • 私钥、proof、完整 VC、请求体和令牌递归脱敏。
  • 重置演示数据不删除日志。
  • 清空后保留 LOG_CLEAR 摘要。
  • 成功和失败都记录。
业务审计日志
覆盖 DID、VC、验签和演示重置等操作，使用 correlationId 串联同一次请求。
系统运行日志
覆盖非法 JSON、未知接口、请求超限和内部异常，并避免向响应暴露堆栈与本地路径。
08 / 11

FROM REQUIREMENTS TO DELIVERY

需求必须能转换成Agent 可执行步骤

编写失败测试 → 确认测试因功能缺失而失败 → 编写最小实现 → 运行专项测试 → 运行全量回归 → 分阶段提交 → 真实页面验收 → 合并主分支
为什么先看测试失败？
证明测试确实能发现缺失功能，而不是在实现完成后写一个永远通过、无法证明有效性的断言。
为什么先专项、再全量？
专项测试快速定位当前改动，全量回归检查它是否破坏 DID、VC、日志、存储或页面的其他能力。
独立 Git 工作树
实施修改在隔离工作树中进行，避免污染主工作区；完成后由用户选择合并、PR、保留或放弃。
权威需求文档
产品目标、范围、角色、Method 矩阵、状态机、模型、API、页面、安全、兼容与完成标准被固化；与旧文档冲突时以新基线为准。

REAL BROWSER FEEDBACK

单元测试通过，页面仍可能错乱

诊断视口

1440、1280、1024、768、390px

真实浏览器逐页检查导航、表单、筛选栏、面板和横向溢出。

发现的根因

  • 隐藏单选框继承 input { width:100% }。
  • 日志筛选栏固定八列最小宽度。
  • 逻辑测试无法证明最终布局正确。
DID 页面横向溢出
隐藏 radio 被全局输入框样式拉满宽度，虽然不可见，却仍撑开布局。修订为明确的小尺寸。
日志筛选栏溢出
由固定八列改为桌面四列、平板两列、手机一列，使中等桌面宽度也能正常收缩。
导航与防回归
移动端为功能入口提供等宽列，增加布局测试，并在真实浏览器中检查每个视图的横向溢出。
它如何反馈回需求？
把“页面不溢出、不同视口可操作、真实浏览器验收”写入正式完成标准，而不是只做一次临时 CSS 修复。
测试代码证明逻辑行为；真实浏览器证明用户看到的最终行为。两者不能互相替代。

方法总结
  1. 基于现有项目，而非凭空生成。
  2. 关键规则由用户逐项确认。
  3. 需求、设计、测试、实现和验收保持追踪。
  4. 冲突先修订规范，再修改代码。
最终结论
形成的是一套能够直接指导开发 Agent、测试 Agent 和验收人员工作的需求与交付体系，而不是一份孤立的 AI 回复。
