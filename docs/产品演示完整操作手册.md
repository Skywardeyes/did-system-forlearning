# “信证台”产品演示完整操作手册

> 适用场景：课程答辩中的产品演示环节  
> 建议时长：主流程 7 分钟，预留 1 分钟机动  
> 演示主线：载入数据 → 查看 DID → 签发 VC → 正常验证 → 篡改失败 → 暂停/恢复 → 替换/撤销 → 查看日志

---

## 一、演示目标与口径

本次演示不是只展示静态页面，而是证明以下闭环真实运行：

1. Issuer 和 Holder 拥有各自的 DID；
2. Issuer 使用 Ed25519 私钥签发 VC；
3. Verifier 对格式、签发方、DID 状态、密钥版本、签名、有效期和 VC 状态进行七项验证；
4. 凭证内容被篡改时，数字签名验证失败；
5. 凭证暂停、恢复、替换、撤销后，验证结果随业务状态变化；
6. 操作过程形成经过脱敏的结构化审计日志。

汇报时应明确：本项目是本地教学演示系统，参考 W3C DID Core 和 VC Data Model 2.0 的核心结构，但不声称具备生产级互操作、链上注册或正式密钥托管能力。

### 步骤 1：重置并载入演示数据（00:00–00:30）

操作：

1. 点击左下角“重置并载入演示数据”；
2. 在确认框点击“确定”；
3. 等待右下角提示“演示数据已准备完成”。

预期页面数据：

- DID 身份：2；
- 已签发凭证：1；
- 有效凭证：1；
- 验证次数：0。

系统自动创建的示例：

| 对象 | 示例内容 |
|---|---|
| Issuer | 可信学习中心，角色为 `issuer`，默认使用 `did:example` |
| Holder | 张晓明，角色为 `holder`，使用 `did:key` |
| VC | 数字身份与可信凭证训练营结业凭证 |

讲解词：

> 我先重置演示环境，使每次汇报都从相同数据起点开始。系统自动创建发行方和持有人两个 DID，并由发行方签发一张结业凭证。这里的操作会真实写入本地 JSON 数据，而不是播放预制动画。

对应代码：

```javascript
// src/vc-service.js
const issuer = await this.createDid({
  name: '可信学习中心',
  role: 'issuer'
});
const holder = await this.createDid({
  name: '张晓明',
  role: 'holder',
  method: 'key'
});
```

### 步骤 2：说明运行总览（00:30–01:15）

操作：

1. 停留在“运行总览”；
2. 从左到右指向四个统计卡片；
3. 指向“可信凭证闭环”：Issuer → Issue VC → Holder → Verifier；
4. 指向页面下方凭证台账中的示例 VC。

讲解词：

> 当前系统中有两个 DID 和一张有效 VC。业务闭环由四个角色动作组成：发行方创建并签名凭证，持有人保存凭证，验证方不需要接触发行方私钥，只需要根据 DID Document 中的公钥进行独立验证。

不要在此处展开：搜索、分页、所有状态按钮。这些属于扩展能力，主流程应优先保证正常验证和异常验证演示完整。

### 步骤 3：查看 DID 与 DID Document（01:15–02:00）

操作：

1. 点击左侧“DID 身份”；
2. 找到“可信学习中心”，点击“DID Document”；
3. 指出 `id`、`verificationMethod`、`publicKeyJwk`、`authentication` 和 `assertionMethod`；
4. 关闭详情窗口；
5. 指出张晓明使用 `did:key`，可信学习中心使用 `did:example`。

讲解词：

> DID 是身份标识，DID Document 是用于解析身份能力和验证方法的公开文档。页面和普通 API 只返回公钥，不返回私钥。`did:example` 用于演示可更新、可轮换、可停用的注册型标识；`did:key` 直接由公钥推导，因此不可更新。

DID Document 关键结构示例：

```json
{
  "id": "did:example:实际随机标识",
  "verificationMethod": [
    {
      "id": "did:example:实际随机标识#key-1",
      "type": "JsonWebKey2020",
      "controller": "did:example:实际随机标识",
      "publicKeyJwk": {
        "kty": "OKP",
        "crv": "Ed25519",
        "x": "实际公钥值"
      }
    }
  ],
  "authentication": ["did:example:实际随机标识#key-1"],
  "assertionMethod": ["did:example:实际随机标识#key-1"]
}
```

提醒：现场不要逐字朗读随机 DID 和公钥，只指出字段含义。

### 步骤 4：签发一张新 VC（02:00–02:50）

操作：

1. 点击左侧“凭证签发”；
2. 发行方选择“可信学习中心”；
3. 持有人选择“张晓明”；
4. 填写以下示例：

| 字段 | 输入值 |
|---|---|
| 学员姓名 | 张晓明 |
| 完成日期 | 当天日期 |
| 课程名称 | 软件工程与可信数字身份实践 |
| 有效期至 | 明年同日 |

5. 点击“签名并签发 VC”；
6. 等待“VC 已使用 Issuer 私钥签发”；
7. 在右侧 JSON 中指出 `issuer`、`credentialSubject`、`validUntil` 和 `proof`。

讲解词：

> 页面提交签发请求后，后端首先校验 Issuer、Holder、角色和有效期，再对稳定序列化后的 VC 内容进行 Ed25519 签名，最后原子写入本地存储。`proofValue` 是签名结果，`verificationMethod` 指明验签应使用哪个密钥版本。

凭证核心结构示例：

```json
{
  "@context": ["https://www.w3.org/ns/credentials/v2"],
  "type": ["VerifiableCredential", "TrainingCompletionCredential"],
  "issuer": "did:example:实际发行方标识",
  "credentialSubject": {
    "id": "did:key:实际持有人标识",
    "name": "张晓明",
    "course": "软件工程与可信数字身份实践"
  },
  "validUntil": "实际有效期",
  "proof": {
    "type": "EducationalEd25519Signature2026",
    "verificationMethod": "did:example:实际发行方标识#key-1",
    "proofValue": "实际签名值"
  }
}
```

对应调用链：

```text
POST /api/credentials
  → VcService.issueCredential()
  → signCredential()
  → JsonStore 原子保存
  → LogService 写入 VC_ISSUE
```

### 步骤 5：执行正常验证（02:50–03:50）

操作：

1. 点击左侧“凭证验证”；
2. 点击“载入最新凭证”；
3. 点击“执行完整验证”；
4. 指向右侧“验证通过”；
5. 依次概括七项检查；
6. 向下指出验证记录台账新增一条成功记录。

预期七项结果：

| 序号 | 检查项 | 正常结果 |
|---|---|---|
| 1 | 格式完整性 | 必要字段和 proof 存在 |
| 2 | Issuer DID 解析 | 能找到签发方 |
| 3 | DID 当前状态 | Issuer 未停用 |
| 4 | 密钥版本 | 能找到当前或历史公钥 |
| 5 | Ed25519 签名 | 签名与内容一致 |
| 6 | 有效期 | 当前时间处于有效期内 |
| 7 | VC 当前状态 | 本地记录为 active |

讲解词：

> 验证不是只调用一次密码学函数。系统会分别执行七项检查，只有全部通过，最终的 `valid` 才为 true。这样能够区分“签名错误”“签发方停用”“凭证过期”和“凭证被撤销”等不同失败原因。

核心汇总代码：

```javascript
const result = {
  valid: checks.every((item) => item.passed),
  credentialId: credential.id,
  checkedAt: new Date().toISOString(),
  checks
};
```

### 步骤 6：篡改姓名并再次验证（03:50–04:35）

操作：

1. 保持在“凭证验证”；
2. 点击“模拟篡改姓名”；
3. 可在左侧 JSON 中指出姓名后出现“（已修改）”；
4. 再次点击“执行完整验证”；
5. 指出总体结果变为“验证失败”；
6. 指出“Ed25519 签名”检查失败，其他与状态相关的检查仍可通过。

讲解词：

> 我只修改已签发凭证中的姓名，没有重新生成 proof。虽然 JSON 格式仍然正确，Issuer 也存在，但签名对应的原始字节已经改变，所以 Ed25519 验签失败。这证明系统检测的是内容完整性，不是简单比较页面文字。

篡改动作等价于：

```javascript
credential.credentialSubject.name = '张晓明（已修改）';
```

关键提醒：下一步操作生命周期之前，必须点击“载入最新凭证”，恢复未经篡改的 JSON。

---

### 步骤 7：暂停、验证失败并恢复（04:35–05:35）

操作：

1. 点击左侧“运行总览”；
2. 在最新凭证所在行点击“暂停”；
3. 返回“凭证验证”；
4. 点击“载入最新凭证”，再点击“执行完整验证”；
5. 指出签名检查通过，但“VC 当前状态”失败，状态为 `suspended`；
6. 返回“运行总览”，点击同一凭证的“恢复”；
7. 回到验证页，重新载入并验证，确认恢复为通过。

讲解词：

> 暂停没有改变凭证内容，因此数字签名仍然正确；但业务状态已经不是 active，最终验证仍然失败。恢复后同一张凭证又可以通过。这说明密码学真实性和业务有效性是两个不同维度。

状态转换规则：

```javascript
const VC_TRANSITIONS = {
  active: new Set(['suspended', 'replaced', 'revoked']),
  suspended: new Set(['active', 'replaced', 'revoked']),
  replaced: new Set(),
  expired: new Set(),
  revoked: new Set()
};
```

---

### 步骤 8：替代重签或撤销（05:35–06:25）

主流程推荐替代重签，因为它能展示新旧凭证的追溯关系。当前页面将该操作按钮简写为“更新”。

操作：

1. 在“运行总览”的最新有效凭证行点击“更新”（该按钮执行替代重签，不是原地覆盖）；
2. 在弹窗输入：`软件工程与可信数字身份实践（修订版）`；
3. 确认台账出现一张新凭证；
4. 原凭证状态变为 `replaced`，新凭证状态为 `active`；
5. 点击两张凭证的“查看”，指出旧凭证的 `replacedBy` 与新凭证的 `replaces` 形成追溯关系。

讲解词：

> 替换不是覆盖旧凭证，而是签发一个新 ID，并保留新旧记录之间的双向关系。旧凭证进入 replaced 终态，新凭证继续有效，因此能够追溯历史。

如果现场剩余不足 30 秒，可改为撤销：

1. 点击有效凭证的“撤销”；
2. 确认警告；
3. 重新载入并验证；
4. 指出签名仍通过，但 VC 状态检查失败且不可恢复。

---

### 步骤 9：查看结构化日志（06:25–07:00）

操作：

1. 点击左侧“日志中心”；
2. 类型选择“操作审计”；
3. 模块依次选择 `VERIFY` 或 `VC`；
4. 指出 `VC_ISSUE`、`VC_VERIFY`、`VC_SUSPEND`、`VC_RESUME`、`VC_REPLACE` 或 `VC_REVOKE`；
5. 点击一条日志的“详情”；
6. 指出 `correlationId`、`action`、`success`、`targetId` 和脱敏后的上下文。

讲解词：

> 每次核心操作都会生成结构化日志，`correlationId` 用于关联同一次请求。日志只记录定位问题所需的元数据，凭证正文、proof、token、cookie、password 和 private 字段会在持久化前递归脱敏。

脱敏字段示意：

```javascript
const sensitiveKeys = [
  'private', 'secret', 'token', 'password',
  'authorization', 'cookie', 'proofValue', 'credential'
];
```

收尾话术：

> 至此，我们完成了从身份建立、凭证签发、正常验证，到篡改检测、状态管理和审计追踪的完整闭环。页面上的每个结果都由真实 API、Ed25519 算法、业务状态机和本地持久化共同产生。

## 四、可选扩展演示

### 4.1 DID 更新与乐观并发控制

1. 打开“DID 身份”；
2. 在“可信学习中心”卡片点击“更新”；
3. 输入：`可信学习中心（更新后）`；
4. 确认版本号增加。

讲解：更新请求携带 `expectedVersion`；如果页面版本落后于存储版本，服务返回冲突，避免后写请求静默覆盖先写请求。

### 4.2 DID 密钥轮换与历史凭证验证

1. 对 `did:example` Issuer 点击“密钥轮换”；
2. 确认 `keyVersion` 增加；
3. 验证轮换前签发的旧 VC；
4. 旧 VC 仍应通过签名检查。

讲解：旧公钥进入 `keyHistory`，VC proof 中保存的 `keyVersion` 用于定位历史公钥。

### 4.3 DID 停用

此操作不可恢复，只在所有其他演示结束后执行：

1. 对 `did:example` Issuer 点击“停用”；
2. 确认警告；
3. 重新验证其签发的 VC；
4. “DID 当前状态”应失败。

不要尝试停用 `did:key`；系统会明确提示它不支持更新、轮换和停用。

### 4.4 搜索与分页

- DID 搜索：输入“可信学习中心”；
- VC 搜索：输入“张晓明”或课程名；
- 验证记录搜索：输入凭证 ID 的一部分；
- 日志筛选：组合类型、成功/失败、级别、模块和时间范围；
- 数据超过一页后可切换每页 10、20、50 条。

## 五、PowerShell API 备用演示

当页面异常但服务仍可访问时，可以用以下命令证明后端功能。以下操作会修改本地演示数据，正式执行前先重置。

### 5.1 检查服务与重置数据

```powershell
$base = 'http://127.0.0.1:4173'
$demo = Invoke-RestMethod -Method Post -Uri "$base/api/demo/reset" -ContentType 'application/json' -Body '{}'
$demo | ConvertTo-Json -Depth 8
```

### 5.2 查看系统状态

```powershell
$state = Invoke-RestMethod -Uri "$base/api/state"
$state | Select-Object `
  @{Name='DID数量'; Expression={$_.dids.Count}}, `
  @{Name='VC数量'; Expression={$_.credentials.Count}}, `
  @{Name='验证记录'; Expression={$_.verificationLogs.Count}}
```

### 5.3 验证原始凭证

```powershell
$verifyBody = @{ credential = $demo.credential.credential } | ConvertTo-Json -Depth 20
$validResult = Invoke-RestMethod -Method Post -Uri "$base/api/verify" -ContentType 'application/json' -Body $verifyBody
$validResult.checks | Format-Table label, passed, detail -AutoSize
$validResult.valid
```

预期最后输出：

```text
True
```

### 5.4 篡改副本并验证

先通过 JSON 往返创建副本，避免修改 `$demo` 中的原始对象：

```powershell
$tampered = $demo.credential.credential | ConvertTo-Json -Depth 20 | ConvertFrom-Json
$tampered.credentialSubject.name = '张晓明（已修改）'
$tamperedBody = @{ credential = $tampered } | ConvertTo-Json -Depth 20
$tamperedResult = Invoke-RestMethod -Method Post -Uri "$base/api/verify" -ContentType 'application/json' -Body $tamperedBody
$tamperedResult.checks | Format-Table label, passed, detail -AutoSize
$tamperedResult.valid
```

预期：总体为 `False`，Ed25519 签名检查为失败。

### 5.5 暂停、恢复和撤销

```powershell
$credentialId = $demo.credential.id

# 暂停
Invoke-RestMethod -Method Post -Uri "$base/api/credentials/$([uri]::EscapeDataString($credentialId))/suspend" -ContentType 'application/json' -Body '{}'

# 恢复
Invoke-RestMethod -Method Post -Uri "$base/api/credentials/$([uri]::EscapeDataString($credentialId))/resume" -ContentType 'application/json' -Body '{}'

# 撤销：不可恢复，最后执行
Invoke-RestMethod -Method Post -Uri "$base/api/credentials/$([uri]::EscapeDataString($credentialId))/revoke" -ContentType 'application/json' -Body '{}'
```

$logs.items | Select-Object occurredAt, level, module, action, success, targetId | Format-Table -AutoSize

## 六、常见故障与现场处理

| 现象 | 原因 | 处理方法 |
|---|---|---|
| 浏览器无法访问 | 服务未启动或终端已关闭 | 回到项目目录执行 `npm start` |
| 端口 4173 被占用 | 旧服务未结束 | 关闭旧终端，或执行 `$env:PORT=4175; npm start` 后访问 4175 |
| 页面数据与讲稿不一致 | 上次演示数据未清理 | 点击“重置并载入演示数据” |
| 篡改后后续验证一直失败 | 文本框仍是篡改副本 | 点击“载入最新凭证”恢复原始 VC |
| 最新凭证状态不适合演示 | 已被替换、撤销或过期 | 重置演示数据，重新签发新 VC |
| 找不到暂停/恢复按钮 | 当前凭证已进入终态 | 重新签发 active 凭证；终态不能恢复 |
| did:key 没有更新按钮 | 这是设计约束 | 说明 did:key 与密钥绑定，不支持生命周期操作 |
| UI 测试不能运行 | Chromium 未安装 | 演示前执行 `npx playwright install chromium` |
| JSON 无法验证 | JSON 被手工改坏 | 点击“载入最新凭证”，不要现场修 JSON 标点 |
| 投影下图表或文字太小 | 浏览器缩放或分辨率不合适 | 使用 `Ctrl+0` 回到 100%，必要时 `Ctrl++` 放大 |

### 现场降级顺序

1. 页面正常：执行完整 7 分钟流程；
2. 个别按钮异常：刷新页面，数据仍保存在本地 JSON；
3. 页面异常但 API 正常：使用第五章 PowerShell 命令；
4. 服务无法恢复：展示事先生成的测试证据和 HTML 汇报中的系统设计图，并明确说明现场环境问题；
5. 不要在台上临时安装 Node.js、修改源码或删除数据文件。
