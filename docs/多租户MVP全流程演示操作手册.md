# DID/VC 多组织多租户 MVP 全流程演示操作手册（历史方案）

> 本文描述的“个人空间 + 多组织空间 + 成员邀请 + 手工 Challenge”流程已经停用，仅保留为设计演进记录。当前可执行流程请使用 [MVP 角色简化与碰一碰演示](MVP角色简化与碰一碰演示.md)。

> 适用版本：MySQL Schema V11、多租户统一账号、个人钱包自托管、组织审核、版本化凭证模板、动态字段 VC、多 VC 组合披露、组合验证台账和本地 EVM DID 锚定。

## 1. 文档用途

本手册用于现场完整演示当前系统。操作人员可以从头依次执行，不需要临时推断下一步。

完整演示覆盖：

1. 统一自然人账号注册；
2. 自动个人空间和待审核组织空间；
3. 平台管理员审核组织；
4. 组织角色与 `tenant_id` 隔离；
5. 个人钱包本地生成 Holder 私钥和 `did:key`；
6. 公开 Holder DID 登记与组织关联；
7. 机构 Issuer DID 创建与 KMS 托管；
8. 组织创建、发布和版本化管理凭证模板；
9. 多家组织按各自模板签发动态字段 VC，钱包接收并领取；
10. 完整 VC 验证与篡改检测；
11. 钱包从多张 VC 中分别选择字段并生成一次组合出示；
12. 验证方逐张验证、Holder 整体签名、Challenge 防重放与组合验证台账；
13. VC 暂停、恢复和不可逆撤销；
14. 组织成员邀请与最小权限授权；
15. Issuer DID 本地 EVM 链上锚定；
16. 审计与敏感明文访问证据。

```text

```

不要勾选“注册后同时提交组织入驻申请”。

点击“创建账号”。

预期结果：

- 自动进入“平台管理员的个人空间”；
- 当前角色只有个人空间角色；
- 暂时看不到“平台治理后台”。

### 6.2 由部署人员授予平台角色

回到项目 CMD：

```cmd
npm run platform:grant-admin -- platform@example.com
```

预期输出：

```json
{
  "userId": "...",
  "role": "platform_admin"
}
```

网页中退出并重新登录 `platform@example.com`，左侧应出现“平台治理后台”。

### 6.3 演示讲解词

> 平台管理员权限不能由注册用户自行勾选，否则任何人都可以自封为平台管理员。我们把平台角色与租户角色分开，并由部署人员通过受控命令授予。平台管理员可以审核组织，但不会自动获得任何组织的 Issuer 私钥或业务数据权限。

## 10. 第五阶段：注册学生自然人账号

在无痕窗口打开：<http://127.0.0.1:5173/login>

注册：

```text

邮箱：student@example.com
```

预期结果：系统只自动创建：


个人空间用于账号、组织关系和公开 Holder DID 管理，并提供“打开个人钱包”入口；完整 VC、选择性披露和 Holder 签名只在钱包完成。个人空间不拥有 Issuer、Verifier 或租户审计能力。

## 11. 第六阶段：钱包本地生成 Holder DID

在学生浏览器打开：<http://127.0.0.1:5176/#identity>

在“创建本地 DID”区域填写：

```text
身份名称：演示学生钱包身份
```

点击“本地生成密钥与 DID”。

预期出现类似：

```text
did:key:z6Mk...
```

记录完整 Holder DID，后续组织关联时使用。

### 11.1 安全观察点

公开登记包只应包含：

- DID；
- DID Document；
- Ed25519 公钥 JWK；
- authentication 和 assertionMethod。

不应包含：

- Holder 私钥；
- 账号密码；
- 恢复材料；
- 已持有 VC；
- 随机盐。

### 11.2 演示讲解词

> Holder 的 Ed25519 私钥由浏览器 Web Crypto 本地生成，以不可导出的 CryptoKey 保存到 IndexedDB。平台只能取得公开 DID Document，不能读取私钥，也不能代替学生对 Challenge 签名。

## 12. 第七阶段：把钱包 DID 绑定到学生个人空间

在钱包顶部“登录个人账号，登记 Holder DID”区域填写：

```text
信证台 API 地址：http://127.0.0.1:4173
账号邮箱：student@example.com
账号密码：DemoPassword123
```

点击“登录并绑定到个人空间”。

预期提示：

```text
公开 DID 已绑定到个人空间；Holder 私钥未上传。
```

这一操作写入当前 V11 数据模型中的 `v2_user_holder_dids`，该表是公开 Holder DID 目录，不是密钥库。

## 13. 第八阶段：学校创建 Issuer DID

回到学校组织工作台，确认当前空间：

```text
组织｜可信数字学院
```

进入“DID 身份”，填写：

```text
机构名称：可信数字学院签发身份
DID Method：did:example
服务地址：留空
```

点击“创建机构 DID”。

预期生成：

```text
did:example:<UUID>
```

卡片应显示：

```text
role = issuer
status = active
keyCustody = issuer_managed_kms
```

讲解：

> Issuer 私钥与 Holder 私钥采用不同托管模型。Holder 私钥由个人钱包自托管；Issuer 私钥属于组织签发基础设施，由服务端 KMS 加密保存，业务接口只能请求签名，不能直接导出私钥。

## 14. 第九阶段：学校关联学生公开 Holder DID

仍在“DID 身份”页面，在“已在个人钱包发布的 Holder DID”输入框粘贴学生的完整：

```text
did:key:z6Mk...
```

点击“从公开目录关联到当前组织”。

预期结果：DID 列表增加 Holder 记录：

```text
role = holder
method = key
keyCustody = holder_self_custody
```

讲解：

> 组织只是把学生公开 DID Document 关联到自己的租户，方便选择签发对象。这个过程没有复制私钥。相同公开身份可以被多个可信组织识别，但各组织签发的 VC 和审计记录仍由 tenant_id 隔离。

## 15. 第十阶段：创建版本化模板并签发第一张动态 VC

学校进入“凭证签发”。页面上半部分是凭证模板注册表，下半部分是动态签发表单。

### 15.1 创建“大学毕业证明”模板草稿

填写模板基本信息：

```text
模板名称：大学毕业证明
凭证类型：UniversityGraduationCredential
```

依次配置字段：

| 字段键 | 中文名称 | 类型 | 必填 | 示例值 |
|---|---|---|---|---|
| `studentName` | 学生姓名 | 文本 | 是 | 演示学生 |
| `degreeLevel` | 学历层次 | 枚举 | 是 | 本科、硕士、博士 |
| `major` | 专业 | 文本 | 是 | 计算机科学与技术 |
| `graduationDate` | 毕业日期 | 日期 | 是 | 当天日期 |
| `studentNumber` | 学号 | 文本 | 否 | 20260001 |

点击“创建模板草稿”，再在模板卡片中点击“发布模板”。

预期结果：

- 模板状态由 `draft` 变为 `published`；
- 模板显示 `V1` 和凭证类型；
- 发布后的版本不能原地修改；
- 同名重新创建会生成 V2，历史 V1 仍可验证。

讲解：

> 模板用于约束组织可以签发哪些字段、字段类型和必填规则。创建草稿时系统对稳定 JSON 计算 SHA-256 `schemaHash`，发布后冻结该版本。模板 ID、版本和摘要会进入 VC 与 Issuer 签名覆盖的 SD-JWT payload，验证方不能只看字段值，还要确认它属于可信模板版本。

### 15.2 使用模板签发毕业 VC

在动态签发表单中填写：

```text
Issuer：可信数字学院签发身份
Holder：演示学生钱包身份
凭证模板：大学毕业证明 · V1
学生姓名：演示学生
学历层次：本科
专业：计算机科学与技术
毕业日期：当天日期
学号：20260001
有效至：一年后
```

点击“使用 Issuer KMS 签发”。

预期结果：

- 弹窗显示本次签发的完整动态 VC；
- VC 摘要列表新增一条 `active` 凭证；
- `credentialSchema` 包含模板 ID、V1 和摘要；
- `credentialSubject` 包含本次动态字段；
- 系统同步创建 `wallet-vc-package-v2` 待领取 offer；
- 列表仍不直接展示学生姓名、学号等受保护正文。

观察 `proof`：它包含 Issuer 验证方法、密钥版本和 Ed25519 签名，但没有 Holder 私钥。

### 15.3 第二家组织签发职业资格 VC（完整组合演示必做）

为了完整体现“跨组织组合”，使用 `cert@example.com` 重复第 7—14 节：

1. 注册“数字技能认证中心”组织并由平台审核；
2. 创建该组织的 Issuer DID；
3. 关联同一个学生 Holder DID；
4. 创建并发布模板：

```text
模板名称：职业资格证明
凭证类型：ProfessionalQualificationCredential
```

模板字段：

| 字段键 | 中文名称 | 类型 | 必填 | 示例值 |
|---|---|---|---|---|
| `qualificationName` | 资格名称 | 文本 | 是 | 软件设计师 |
| `qualificationLevel` | 资格等级 | 枚举 | 是 | 初级、中级、高级 |
| `issuedDate` | 取得日期 | 日期 | 是 | 当天日期 |
| `certificateNumber` | 证书编号 | 文本 | 否 | CERT-2026-001 |

向同一 Holder 签发第二张 VC。

第二家组织及模板建议在正式计时前准备，但第二张 VC 尽量现场签发。这样能够证明两张 VC 的 Issuer DID、模板和租户来源不同，验证方仍可通过公共信任投影逐张解析，而不是把所有数据塞进一张“大证书”。如果跳过本节，只能证明“多凭证组合”，不能宣称现场演示了“跨组织组合”。

## 16. 第十一阶段：学生钱包收到并领取两张 VC

回到钱包，进入“凭证收件箱”。钱包每 15 秒自动检查；也可以点击“立即检查”。

应看到学校和认证机构分别发来的待领取凭证。对两条消息依次点击“领取并导入”。

每次领取的后台过程是：

1. 钱包请求一次性领取 Challenge；
2. 钱包使用 Holder 本地私钥签名；
3. 后端使用公开 DID Document 验证签名；
4. Challenge 被原子消费；
5. v2 钱包包自动导入 IndexedDB；
6. “本地凭证库”出现对应凭证。

进入“本地凭证库”，确认至少有：

- 大学毕业证明；
- 职业资格证明。

讲解：

> 后端不会因为请求者知道 Holder DID 就交付 VC。领取者必须证明自己控制对应私钥。v2 钱包包还带有字段中文名、类型、模板版本和路径元数据，因此钱包无需预先写死“姓名、课程、日期”，就能动态生成字段选择界面。

## 17. 第十二阶段：完整 VC 验证与篡改检测

验证组织进入“凭证验证”。

### 17.1 授权载入

点击“授权载入”。系统载入当前组织最新 VC，并提示“已授权载入，明文访问已审计”。明文只保存在当前 Vue 组件内存，离开页面时清空。

如果返回 403，说明第 9 节尚未授予 `credential_data_reader`。

### 17.2 执行完整验证

点击“执行完整验证”。预期检查包括：

- 凭证结构和动态 `credentialSubject`；
- Issuer DID 及其当前状态；
- 签名密钥版本与 Ed25519 签名；
- 有效期；
- VC 生命周期状态；
- Holder DID 状态。

### 17.3 篡改检测

直接在 JSON 中修改任意动态字段，例如把 `major` 改为“金融学”，但保留原 `proof`，再次执行完整验证。

预期结果：Ed25519 签名检查失败，整体验证失败。

讲解：

> VC 可以作为 JSON 显示和传输，但明文不等于可以任意修改。Issuer 签名覆盖凭证核心内容；攻击者保留 proof、修改任何动态字段，验签都会失败。HTTPS解决传输保密，数字签名解决来源真实性和内容完整性，两者职责不同。

## 18. 第十三阶段：多 VC 组合披露与 Holder 绑定验证

### 18.1 验证方生成一次性 Challenge

验证组织进入“钱包证明验证”，填写：

```text
验证方域名：hr.example.com
```

点击“生成一次性 Challenge”，复制 Challenge 和域名。页面会显示过期时间。

### 18.2 Holder 从两张 VC 中分别选择字段

学生钱包进入“选择性披露”。先在搜索框按签发方、凭证名称或凭证 ID 查找，在下拉框选择并点击“添加凭证”；可重复添加多张。页面只展开已添加凭证的动态字段，不会一次铺开全部本地 VC。

凭证标题采用“签发方名称·凭证名称”，例如：

```text
上海大学·大学毕业证明
数字技能中心·职业资格证明
```

分别添加两张演示凭证后，推荐勾选：

```text
大学毕业证明：学历层次、专业、毕业日期
职业资格证明：资格名称、资格等级
```

不要勾选：

```text
学生姓名、学号、证书编号、取得日期
```

填写：

```text
Verifier Challenge：刚复制的 Challenge
Verifier 域名/标识：hr.example.com
```

点击“本地生成组合披露证明”，复制完整 JSON。

确认顶层类型是：

```text
WalletBoundMultiSdJwtPresentation2026
```

其中 `verifiableCredentials` 应有两项，每项只带该 VC 的 Issuer JWT 和本次勾选的 Disclosure；`holderProof` 是 Holder 对整个组合、Challenge、domain 和创建时间的一次整体签名。

### 18.3 验证方逐张验证组合

回到“钱包证明验证”，粘贴组合 JSON，点击：

```text
逐张验证并核验 Holder 组合签名
```

预期总体结果为“组合出示验证通过”，并显示两条逐凭证结果。每张凭证分别检查：

- Issuer DID 和历史公钥能否从公共信任数据解析；
- Issuer SD-JWT 的 EdDSA 签名；
- Disclosure 与 `_sd` 摘要；
- 模板 ID、版本、`schemaHash`、凭证类型和字段声明；
- 有效期和当前凭证状态；
- SD-JWT `sub` 是否都等于当前 Holder DID。

整组还要检查：

- 凭证 ID 不重复；
- Holder 对完整组合的本地 Ed25519 签名；
- Challenge 和 Domain 匹配；
- Challenge 首次使用并对整组原子消费。

### 18.4 查看组合验证台账

在同一页面下方查看“组合验证台账”。最新一条记录应显示：

- `presentationId`；
- Holder DID；
- 总体 `valid`；
- 凭证数量为 2；
- 每张凭证的 Issuer、类型、结果和披露路径。

台账不会复制保存未披露字段值。

### 18.5 防重放与整组失败演示

不做任何修改，再次提交同一份组合证明。

预期结果：验证失败，因为 Challenge 已被消费。

如需演示“任意一张失败导致整组失败”，生成新 Challenge 后，撤销其中一张 VC，再重新生成并提交组合证明。预期该凭证状态检查失败，整组结果也失败。

讲解：

> 验证方只看到本次跨凭证选择的声明。每项声明仍保留所属 VC、Issuer 和模板来源，不能把 A 机构字段伪装成 B 机构字段。Issuer 签名证明声明来源；所有凭证的 `sub` 与 Holder 整体签名证明由同一个持有人主动组合出示；一次性 Challenge 阻止旧组合被重复播放。

## 19. 第十四阶段：VC 生命周期

回到“凭证签发”的 VC 摘要列表。

### 19.1 暂停

点击“暂停”。

预期状态：

```text
suspended
```

再次验证时，密码学签名仍可能正确，但凭证整体无效。离线钱包仍可能组装本地证明，因为它不应被要求持续向平台报告持有内容；验证方查询最新公开状态后必须拒绝该凭证。

### 19.2 恢复

点击“恢复”。

预期状态恢复为：

```text
active
```

再次验证恢复有效。

### 19.3 撤销

点击“撤销”，确认不可逆提示。

预期状态：

```text
revoked
```

撤销后不能恢复。完整验证应失败；包含该 VC 的多凭证组合也应整组失败，其他未撤销 VC 不受影响。

讲解：

> 签名回答“内容是否由该 Issuer 签发且未被修改”，生命周期回答“现在是否仍被认可”。二者必须同时检查。撤销后不必重写历史记录，验证方读取最新状态即可拒绝旧凭证。

## 20. 第十五阶段：组织邀请和角色授权

学校进入“组织与成员”。

为了演示，可以提前再注册一个账号：

```text
姓名：学校验证员
邮箱：verifier@example.com
密码：DemoPassword123
```

学校负责人输入：

```text
受邀邮箱：verifier@example.com
```

点击“生成一次性邀请令牌”。

复制令牌，登录验证员账号，在“收到的邀请令牌”中粘贴并接受。

预期初始角色只有：

```text
organization_member
```

学校负责人加载成员后，可以只授予：

```text
verifier_operator
```

不要授予 Issuer 或租户管理员，以展示最小权限。

安全检查：

- 邀请令牌使用 256 位随机数；
- 数据库只保存 SHA-256 哈希；
- 明文只在创建时返回一次；
- 接受邀请的登录邮箱必须与受邀邮箱一致；
- 系统禁止移除最后一个 `tenant_admin`。

## 21. 第十六阶段：本地 EVM 链上 DID 锚定

该部分建议演示前完成部署，正式讲解时只操作“上链登记/同步”和“查询链上状态”。

### 21.1 启动本地链

另开终端：

```cmd
npm run chain:node
```

保持终端运行。

### 21.2 部署 DidRegistry 合约

再开终端：

```cmd

```

```text
data/chain-deployment.json
```

后端会自动从该文件读取合约地址，因此通常不需要手工复制 `BLOCKCHAIN_CONTRACT_ADDRESS`。

### 21.3 配置 `.env`

```dotenv
BLOCKCHAIN_ENABLED=true

BLOCKCHAIN_CHAIN_ID=31337
```

修改后必须重新启动后端：

```cmd
npm start
```

### 21.4 上链登记 Issuer DID

学校进入“DID 身份”，找到活跃 Issuer DID，点击：

```text
上链登记 / 同步
```

预期弹窗显示：

- 交易哈希；
- 区块号；
- 合约地址；
- DID 哈希；
- DID Document 哈希；
- 链 ID；
- 链上版本。

### 21.5 可选：链上停用

1. 在平台中先停用 Issuer DID；
2. 再点击“写入链上停用”；
3. 查询链上状态，应显示 `deactivated = true`。

## 22. 第十七阶段：审计证据

学校进入“审计中心”，重点展示：

- 请求关联 ID；
- 当前 `tenant_id`；
- 操作人 ID；
- DID 创建和生命周期；
- VC 签发与状态变更；
- 模板创建、发布和版本变化；
- 验证成功和失败；
- Challenge 创建和消费；
- 组合验证 `presentationId` 及逐凭证结果；
- 敏感明文访问目的；
- 脱敏后的错误信息。

解释：

> 普通 VC 列表查询不会解密正文。只有独立的 credential_data_reader 角色提供明确目的时，系统才受控解密，并坚持先写审计、后返回明文；如果审计写入失败，明文读取也会失败关闭。

## 23. 推荐现场顺序与时间

### 23.1 时间是否够用

如果从零开始现场注册四个账号、审核两家组织、创建两套模板并完成全部功能，建议预留 **25—35 分钟**。不要把这套“从零搭建流程”硬塞进 12 分钟。

汇报中的产品演示建议提前准备：平台管理员、两家已审核组织、两个 Issuer DID、已关联的 Holder DID，以及两个已发布模板。现场保留“各签发一张 VC、钱包领取、组合披露、验证与防重放”作为核心真实操作。这样 12 分钟较稳妥。

### 23.2 12 分钟推荐产品演示

| 时间 | 内容 |
|---:|---|
| 0:00—0:50 | 统一账号、个人/组织空间、审核前后权限对比 |
| 0:50—1:40 | 钱包展示 Holder DID 与本地不可导出私钥边界 |
| 1:40—3:00 | 展示两家组织的 Issuer DID、模板 V1 与动态字段 |
| 3:00—4:20 | 两家组织分别现场签发毕业 VC 和资格 VC |
| 4:20—5:20 | 钱包收到消息并领取两张 v2 凭证 |
| 5:20—6:20 | 完整 VC 验证和动态字段篡改失败 |
| 6:20—8:10 | 从两张 VC 分别勾选字段并生成组合出示 |
| 8:10—9:30 | 验证方逐张验证并展示组合验证台账 |
| 9:30—10:10 | 重复提交，演示 Challenge 防重放 |
| 10:10—11:10 | 撤销一张 VC，使用新 Challenge 演示整组失败 |
| 11:10—12:00 | 租户隔离、审计证据与生产边界总结 |

### 23.3 8 分钟压缩演示

该版本同样要求两家组织、DID、模板和 Holder 关联已经准备完成；8 分钟内不现场注册和审核第二家组织。

必须现场操作：

1. 组织审核前后权限变化；
2. 钱包本地 Holder DID；
3. 展示已发布模板，并由两家组织各签发一张动态 VC；
4. 钱包领取两张 VC；
5. 从两张 VC 组合选择字段；
6. 逐张验证、组合台账与防重放；
7. 撤销后整组验证失败。

只讲不操作：

- 成员邀请；
- 敏感访问角色；
- 链上停用；
- 平台管理员授予命令。

### 23.4 演示结束后如何重新开始

当前系统没有提供“一键删除所有多租户生产数据”的按钮，这是有意避免误删审计、状态和组织数据。`/api/v2/demo/reset` 的含义是追加一组本地演示数据，不是清空数据库。

推荐采用以下可重复演示策略：

1. 账号邮箱、组织名和模板名增加场次后缀，例如 `school-02@example.com`、`大学毕业证明-第二场`；
2. 下一场使用新的 Holder 钱包身份，避免旧 IndexedDB 凭证和旧 Challenge 混入；
3. 浏览器开发者工具中仅清理钱包站点 `http://127.0.0.1:5176` 的 IndexedDB，或者直接使用新的浏览器配置文件；
4. 不重复使用已消费的 Challenge；每次生成新的 Challenge 和组合证明；
5. 不要为了重演而直接修改 `v2_credentials.status`、删除审计记录或手工恢复已撤销凭证；
6. 如确实需要全新数据库，应使用单独的演示数据库重新执行迁移和初始化，不要清空仍需保留的开发数据库。

演示前只想快速补一组本地固定数据时，可以在组织工作台使用演示重置能力；但动态模板、第二家组织和钱包本地私钥仍应按本手册准备。

## 25. 常见故障与现场处理

| 现象 | 原因 | 处理 |
|---|---|---|
| `Could not read package.json` | CMD 不在项目目录 | 执行 `cd /d D:\Github\did-system-forlearning` |
| `Cannot find package mysql2` | 未安装依赖 | 执行 `npm install` |
| 后端提示数据库不可用 | MySQL 未启动或 `.env` 错误 | 启动 MySQL，检查 DB_HOST、DB_USER、DB_NAME |
| Schema 版本不支持 | 未执行最新迁移 | 执行 `npm run migrate` |
| 模板无法用于签发 | 模板仍是 `draft` 或已 `retired` | 在模板卡片发布草稿，签发时选择 `published` 版本 |
| 动态字段不出现 | 未选择模板，或模板字段未发布 | 重新选择已发布模板，确认模板字段列表 |
| 页面登录后仍无平台后台 | 授权后会话未刷新 | 退出并重新登录平台账号 |
| 组织审核通过后仍无签发菜单 | 当前令牌还是旧空间角色 | 切换个人空间再切回组织，或重新登录 |
| “授权载入”返回 403 | 缺少敏感读取角色 | 在组织成员页授予 `credential_data_reader` |
| 钱包 `Failed to fetch` | 后端未启动、地址错误或旧进程 | 确认 API 地址为 4173，并重启后端 |
| 钱包收不到 VC | Holder 未关联、VC 未签发给该 DID | 核对完整 did:key 和 VC Holder |
| 钱包领取失败 | DID 不匹配或 Challenge 过期 | 重新检查收件箱，生成新 Challenge |
| v2 钱包包无法导入 | 前后端或钱包仍是旧进程 | 重启后端和 `wallet:dev`，确认包格式为 `wallet-vc-package-v2` |
| 组合证明无法生成 | 没有勾选字段、超过 10 张 VC 或 50 个字段 | 每张所选 VC 至少勾选一项，并减少选择数量 |
| 一张 VC 通过但组合失败 | Holder 不一致、另一张 VC 无效或整体签名/Challenge 失败 | 查看页面逐凭证结果和总体检查项 |
| 同一证明第二次验证失败 | 正常防重放行为 | 重新生成 Challenge 和钱包证明 |
| 区块链显示尚未就绪 | 节点、合约或配置缺失 | 启动 chain:node、执行 chain:deploy、重启后端 |
| 链上停用按钮不可用 | 尚未平台停用或尚未上链 | 先上链登记，再平台停用，最后写链上停用 |

## 26. 应急降级方案

如果现场网络或链节点异常：

1. 区块链部分改为展示 `data/chain-deployment.json` 的非敏感字段和已有截图；
2. 不影响统一账号、钱包、VC、选择性披露和生命周期演示；
3. 明确链上锚定是附加信任层，不是 VC 签名验证的前置条件。

如果钱包通知未及时出现：

1. 点击“立即检查”；
2. 仍失败时，在组织凭证列表点击“生成钱包交付包”；
3. 钱包“本地凭证库”保留手动导入作为演示备用路径。

如果平台审核账号不可用：

1. 使用已审核的演示组织继续主流程；
2. 用架构图解释 `pending → approved → role grant`；
3. 不要现场直接修改数据库伪造审核状态。

## 27. 演示结束总结词

> 这套 MVP 的信任源点不是管理后台替用户保管所有密钥，而是组织通过 KMS 控制 Issuer DID，个人通过本地钱包控制 Holder DID。不同组织可以用版本化模板签发结构完全不同的 VC；个人不必把所有事实合并成一张证书，而是从多张 VC 中挑选本次需要的字段。验证方通过公共 DID、公钥历史、模板摘要和凭证状态逐张验证，再验证 Holder 对整个组合与一次性 Challenge 的签名。平台负责组织准入、租户隔离、状态和审计，但不能替 Holder 签名，也不会把未披露值写入组合台账。区块链仍只锚定公开 DID 状态。当前系统已形成从模板、签发、领取、组合出示、跨组织验证到撤销的可运行闭环，同时明确保留移动端安全区、正式 SD-JWT VC 配置、联盟链治理等生产化工作。
