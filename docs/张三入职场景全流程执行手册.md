# 张三入职场景全流程执行手册

## 1. 演示目标

本手册基于当前项目的真实实现，完成以下业务闭环：

```text
上海大学 ──签发本科毕业证明──┐
                              ├─→ 张三的信证钱包 ─→ 组合选择性披露 ─→ 格尔软件验证
华山医院 ──签发 A 类体检报告─┘
```

最终要证明：

1. 上海大学和华山医院分别拥有自己的组织账号与 Issuer DID；
2. 两家组织分别维护自己的凭证模板；
3. 张三在个人钱包中创建并掌控 Holder DID；
4. 两家组织分别向张三签发一张 VC；
5. 张三领取两张 VC，并只选择其中部分字段；
6. 两张 VC 的部分字段被组合成一份选择性披露证明；
7. 证明被定向发送给格尔软件股份有限公司；
8. 格尔软件一次完成 Issuer、VC 状态、披露字段、Holder 签名和防重放验证。

本流程不演示区块链，避免上链环境影响主体业务演示。

---

## 2. 当前系统中的角色对应

| 参与者 | 角色 | 使用产品 | 主要操作 |
|---|---|---|---|
| 上海大学 | Issuer | 信证台 | 创建 Issuer DID、创建并发布模板、签发本科毕业证明 |
| 华山医院 | Issuer | 信证台 | 创建 Issuer DID、创建并发布 A/B 类体检模板、签发 A 类体检报告 |
| 格尔软件股份有限公司 | Verifier | 信证台 | 接收张三定向投递的证明并验证 |
| 张三 | Holder | 信证钱包 | 创建 Holder DID、领取 VC、组合字段、出示证明 |

注意：

- Holder 只使用信证钱包，不登录信证台；
- Issuer 和 Verifier 只使用信证台；
- Holder 私钥在浏览器 IndexedDB 中以不可导出的 `CryptoKey` 保存，不上传信证台；
- 组织注册后，当前 MVP 会直接授予演示所需的签发和验证权限，不需要额外执行审核；
- 格尔软件只做验证时，不必创建 Issuer DID；验证方通过自己的组织账号和租户身份接收定向证明。

## 4. 启动系统

### 4.1 打开三个终端

三个终端都先进入项目目录：

```powershell
cd D:\Github\did-system-forlearning
```

### 4.2 终端一：检查数据库并启动后端

```powershell
npm install
npm run db:check
npm start
```

预期看到后端监听 `http://127.0.0.1:4173`。

如果数据库迁移未完成，先停止后端并执行：

```powershell
npm run migrate
npm run db:check
npm start
```

### 4.3 终端二：启动信证台

```powershell
npm run frontend:dev
```

访问：<http://127.0.0.1:5173>

### 4.4 终端三：启动信证钱包

```powershell
npm run wallet:dev
```

访问：<http://127.0.0.1:5176>

### 4.5 启动后的检查

1. `5173` 显示“信证台”登录页；
2. `5176` 显示“信证钱包”登录页；
3. 两个页面提交表单时没有 `Failed to fetch`；
4. 本流程不要清除 `5176` 的浏览器站点数据，否则张三本地 DID 和私钥会丢失。

建议把信证台和信证钱包分别放在两个浏览器窗口中，演示切换更清晰。

## 7. 第三阶段：上海大学创建本科毕业证明模板

### 7.1 关于模板名称

“大学生结业证明”和“本科毕业证明”在业务含义上不同，而且当前系统会直接把“模板名称”显示为钱包中的凭证名称。

因此本演示将“大学生结业/毕业证明业务模板”具体命名为“本科毕业证明”。如果填写“大学生结业证明”，最终钱包标题也会显示“上海大学·大学生结业证明”。

### 7.2 创建模板

进入左侧“凭证签发”，在“组织自定义凭证模板”中填写：

| 字段 | 填写内容 |
|---|---|
| 模板名称 | 本科毕业证明 |
| 凭证类型 | UniversityGraduationCredential |

逐项创建以下字段。每增加一个字段，点击“添加字段”。

| 顺序 | 字段键 | 中文名称 | 类型 | 必填 | 枚举选项 |
|---:|---|---|---|---|---|
| 1 | subjectName | 学生姓名 | 文本 | 是 | — |
| 2 | studentNumber | 学号 | 文本 | 是 | — |
| 3 | degreeLevel | 学历层次 | 枚举 | 是 | 本科、硕士、博士 |
| 4 | major | 专业 | 文本 | 是 | — |
| 5 | graduationStatus | 毕业状态 | 枚举 | 是 | 已毕业、已结业、未完成 |
| 6 | graduationDate | 毕业日期 | 日期 | 是 | — |
| 7 | degreeName | 学位名称 | 文本 | 是 | — |
| 8 | certificateNumber | 证书编号 | 文本 | 是 | — |

枚举字段的操作方式：

1. 类型选择“枚举”；
2. 在“枚举选项”输入一个选项；
3. 点击“新增选项”；
4. 把所有选项逐项加入后再创建模板。

点击“创建模板草稿”。模板卡片出现后，继续点击“发布模板”。

预期结果：模板状态由 `draft` 变为 `published`。

注意：模板发布后不能直接修改。如果填错，应重新创建新版本，不要继续使用错误模板。

回到信证钱包“凭证收件箱”，等待自动提醒或点击“立即检查”。

点击“领取并导入”。

进入“本地凭证库”，预期看到两行：

```text
上海大学·本科毕业证明
华山医院·A类体检报告
```

如果只看到一张：

1. 确认钱包当前 DID 是“张三·求职身份”；
2. 回到收件箱点击“立即检查”；
3. 确认医院签发时选择的是同一个 Holder DID；
4. 确认医院已接受张三的 Holder 申请。

## 16. 第十二阶段：张三生成跨两张 VC 的选择性披露证明

进入钱包“选择性披露”。

### 16.1 选择验证组织

| 字段 | 填写内容 |
|---|---|
| 搜索验证组织 | 格尔软件股份有限公司 |
| 接收并验证证明的组织 | 格尔软件股份有限公司 |

必须先选择格尔软件。系统生成的一次性 Challenge 会绑定格尔软件组织，其他组织不能验证这份投递。

### 16.2 添加本科毕业证明

在“搜索凭证”中输入：

```text
上海大学
```

选择“上海大学·本科毕业证明”，点击“添加凭证”。

建议勾选：

```text
学生姓名
学历层次
专业
毕业状态
毕业日期
```

不要勾选：

```text
学号
证书编号
```

这样可以证明张三已从上海大学本科毕业，同时不暴露内部学号和证书编号。

### 16.3 添加 A 类体检报告

在“搜索凭证”中输入：

```text
华山医院
```

选择“华山医院·A类体检报告”，点击“添加凭证”。

建议勾选：

```text
体检类别
综合结论
入职适岗结论
报告有效期
```

不要勾选：

```text
受检人姓名
报告编号
血常规结果
胸部影像结果
传染病筛查
服务价格
```

两张凭证由同一个 Holder DID 持有，并由张三本地私钥对组合证明签名。因此体检报告不必重复披露姓名，也能证明它与毕业证明属于同一个 Holder。

### 16.4 生成证明

点击“本地生成组合披露证明”。

预期提示类似：

```text
组合披露证明已在本地生成，共包含 2 张凭证，将发送给“格尔软件股份有限公司”。
```

此时证明只在钱包本地生成，尚未投递。

可以展开“查看证明技术数据”，演示其中包含：

- 两张凭证的选择性披露材料；
- 已披露字段；
- Issuer 签名相关材料；
- 张三 Holder DID；
- 张三钱包的 Holder 组合签名；
- 格尔软件绑定的 Challenge 和 Domain。

未勾选字段的原值不会进入组合披露证明。

## 18. 第十四阶段：格尔软件验证组合证明

回到格尔软件的“钱包证明验证”页面。

确认左侧出现：

```text
已收到一份选择性披露证明
组合凭证：2 张
```

点击“验证”。

### 18.1 预期总结果

页面显示：

```text
验证通过：来源、状态、字段摘要与 Holder 签名均有效。
```

右侧“逐项验证报告”应全部通过。

### 18.2 验证程序实际检查的内容

| 检查项 | 要证明什么 |
|---|---|
| 表示格式 | 组合证明的结构可以被安全解析 |
| Challenge/Domain | 证明是为格尔软件本次验证生成，不是旧证明重放 |
| Holder DID | 两张 VC 的持有人与当前出示者一致 |
| Holder 签名 | 张三确实控制本地 DID 私钥 |
| Issuer DID | 上海大学、华山医院的 DID 可以解析 |
| Issuer 签名 | 两张 VC 分别由对应机构签发且内容未被篡改 |
| VC 生命周期 | 两张 VC 均未暂停、撤销或过期 |
| 模板摘要 | 披露字段仍属于原始已发布模板 |
| 披露摘要 | 已披露值与签发时承诺的摘要一致 |

### 18.3 逐张凭证结果

页面应出现两张凭证的独立结果：

```text
UniversityGraduationCredential · valid
ClassAMedicalExaminationCredential · valid
```

格尔软件能看到的业务内容应为：

```text
张三
本科
计算机科学与技术
已毕业
2026-06-30
A类
合格
适合入职
2026-10-10
```

格尔软件不应看到：

```text
学号 2022012345
毕业证书编号 SHU-GRAD-2026-0001
体检报告编号 HSH-A-2026-0001
血常规详细结果
胸部影像详细结果
传染病筛查结果
体检服务价格
任何私钥
```

### 18.4 验证台账

验证完成后，下方“组合验证台账”新增一条记录，包含：

- 验证时间；
- `valid` 结果；
- 张三 Holder DID；
- 凭证数量 `2`；
- 每张凭证的类型、结果和披露路径。

这条台账用于证明格尔软件在什么时间验证了什么，而不是把张三的完整原始 VC 再保存一份。

## 19. 演示时的完整口播串词

> 张三入职格尔软件时，不再携带毕业证复印件和完整体检报告。上海大学和华山医院先在信证台建立自己的机构 DID，并分别配置本科毕业证明和体检报告模板。张三在个人钱包里创建自己的 DID，私钥始终保存在自己的设备上。两家机构接受张三的公开 DID 后，各自签发一张可验证凭证。张三领取凭证后，不需要把完整材料交给企业，而是从毕业证明中选择姓名、学历、专业和毕业状态，从体检报告中只选择 A 类、合格、适合入职和有效期，组合成一份证明。最后，他选择格尔软件作为接收方，碰一下完成投递。格尔软件一次验证两家机构的签名、凭证状态和张三的 Holder 签名，既解决跨机构信任，也减少企业对无关个人信息的收集。

## 20. 常见问题与排查

### 20.1 钱包组织下拉框找不到某个组织

1. 确认该组织已经在信证台完成注册；
2. 点击钱包中的“刷新组织列表”；
3. 清空组织搜索框后重新输入完整名称；
4. 确认后端仍在 `4173` 端口运行。

### 20.2 机构找不到张三

1. 确认张三把同一个 DID 发送给了该机构；
2. 在“DID 身份”点击 Holder 申请区域的“刷新”；
3. 点击“接受并关联”；
4. 再进入“凭证签发”搜索张三。

### 20.3 签发按钮不可用

检查以下内容：

- Issuer DID 是否已经创建且状态为 `active`；
- Holder 申请是否已接受；
- 模板是否已发布为 `published`；
- 是否填写了所有必填字段；
- “有效至”是否晚于当前时间。

### 20.4 钱包没有收到凭证

1. 确认签发日志已经出现该凭证；
2. 确认签发时选择的是“张三·求职身份”；
3. 在钱包确认当前使用的也是“张三·求职身份”；
4. 在收件箱点击“立即检查”；
5. 不要清除钱包 IndexedDB。

### 20.5 “本地生成组合披露证明”失败

检查：

- 是否选择了格尔软件股份有限公司；
- 是否已经添加两张凭证；
- 每张需要使用的凭证是否至少勾选一个字段；
- VC 是否为 `active` 且未过期；
- 当前钱包 DID 是否与 VC 的 Holder DID 一致。

### 20.6 格尔软件没有收到“碰一下”证明

1. 确认钱包目标组织选择的是格尔软件股份有限公司；
2. 确认已经先生成证明，再点击“碰一下，就验证！”；
3. 保持格尔软件停留在“钱包证明验证”页面，等待约 2.5 秒；
4. 确认格尔软件账号没有退出；
5. 重新生成证明再投递，不要重复使用已消费的 Challenge。

### 20.7 验证失败

根据右侧失败项判断：

- `signature`：凭证或披露内容可能被修改；
- `credential status`：VC 可能已暂停、撤销或过期；
- `issuer DID`：签发机构 DID 可能已停用或无法解析；
- `holder signature`：证明不是由对应 Holder 私钥生成；
- `challenge/domain`：证明投错组织、已经验证过或已过期；
- `disclosure digest`：披露值与签发时承诺的摘要不一致。
