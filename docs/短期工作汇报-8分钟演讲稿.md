## 2:00—3:10　生产级数据层：02

数据层采用分层结构：入口层负责 V2 API、JWT、HTTPS 和响应不缓存；领域服务处理 DID、VC、披露与验证；数据访问层包含 Repository、Unit of Work、KMS 和验证方 Challenge 台账；最下方是 MySQL V7。

Repository 隔离 SQL 与业务规则，业务服务不能随意操作数据库。Unit of Work 保证多表修改全部提交或全部回滚，例如签发时，凭证、披露材料和状态事件必须保持一致。

敏感字段使用 AES-256-GCM 加密；私钥只能在 KMS 边界内用于签名；查询必须携带租户条件，并用乐观锁防止并发覆盖。

迁移也不是“一刀切”：先 Preflight 检查，再迁移，经过 Dual 双模式和 Reconcile 对账，最后切到 V2-only，降低历史数据损坏和链路中断风险。

## 3:10—4:35　安全整改：03

安全整改首先澄清一个概念：DID Document 中的 DID 和公钥本来就应公开，因为验证方需要公钥验签；公开公钥不等于公开私钥。

真正需要保护的是私钥、VC 个人声明、选择性披露盐和敏感材料。

整改前，VC 列表会解密正文，管理员刷新页面就可能批量看到姓名和课程。整改后，列表 SQL 不查询加密正文，只返回 ID、状态、有效期、关联关系和版本号。

读取完整 VC 必须调用 `content-access` 接口：检查 `credential_data_reader` 角色；校验固定用途；只解密指定凭证；记录操作人、凭证、用途和时间；最后返回明文。

解密和审计在同一事务。审计失败就回滚且不返回明文。这是“失败关闭”：宁可本次不可用，也不能产生无法追责的访问。

同时，`tenant_admin` 不再自动继承签发、持有、验证和敏感读取权限，实现了管理权与数据读取权分离。

【操作：下滑到“Vue 3 前后端分离”】

## 4:35—5:45　Vue 3 前后端分离：04

第三阶段采用同一仓库、两个独立工程、生产同源的方案。浏览器运行 Vue 3、Router 和 Pinia；Nginx 提供静态文件并把 `/api` 转发给 Node；Node 在生产环境设置 `SERVE_FRONTEND=false`，只负责 API。

已迁移总览、DID、签发、完整验证、选择性披露和审计六个页面。旧 `public` 页面暂时作为迁移回退，功能对齐后再删除。

前端也控制敏感数据：Token 只在模块内存，不进入 Pinia、LocalStorage 或 SessionStorage；Pinia 只保存摘要；完整 VC 仅在授权后进入当前组件，离页即清空。SD-JWT 和披露证明采用同样策略。

因此，这不只是更换框架，也是在浏览器侧落实最小数据原则。

测试结果是：Node 219 项全部通过；Vue 构建通过；真实 Chromium 端到端新增钱包自托管全流程通过；MySQL 已升级到 V7，V2-only 冒烟确认完整 VC、SD-JWT、列表保护、敏感访问审计和 Challenge 台账均正常。

## 6:45—7:35　边界与下一步：06

答辩反馈指出一个更根本的边界：Holder DID 私钥必须由个人控制，不能由服务端 KMS 托管。数据库加密、KMS 隔离和 HTTPS 都不能替代“私钥在个人钱包本地”的控制权。

这项更正已经完成钱包 MVP：个人钱包使用浏览器 Web Crypto 在本地生成不可导出的 Holder Ed25519 私钥和 `did:key`；签发平台只接收 DID 与公开 Document，不创建、不保存 Holder 私钥；机构 HSM/KMS 仅管理 Issuer 私钥。钱包导入 VC 交付包后，在本地选择披露字段，并对 Challenge、验证方域名和 SD-JWT 材料进行 Holder 签名；平台再验证 Issuer 签名、VC 状态与 Holder 签名。验证方会先生成一次性 Challenge，平台只保存其 SHA-256 哈希，首次成功验证原子消费，第二次重放失败。正式 DID Registry、跨设备恢复和标准 SD-JWT Holder Binding 仍属于下一阶段。
