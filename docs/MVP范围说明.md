# DID/VC 本地演示系统 MVP 范围说明

> 本文保留最初 MVP 的立项背景。当前扩展阶段的唯一权威需求基线为 [产品需求说明书-DID-VC生命周期.md](产品需求说明书-DID-VC生命周期.md)。当本文与产品需求说明书冲突时，以产品需求说明书为准。

## 1. 项目定位

本项目为课程结业任务中的本地演示系统，目标是实现 DID 与 VC 的核心业务闭环，包括：

- 创建 DID
- 签发 VC 凭证
- 查看 VC 凭证
- 验证 VC 凭证
- 撤销 VC 凭证
- 完成测试、人工验收与交付总结

本项目不以生产环境上线为目标，不涉及真实身份核验、链上 DID 注册、生产级密钥托管、跨机构互操作和商业化部署。

## 2. 演示场景

系统以“培训营结业证书”为业务场景。

培训机构作为 VC 发行方，为完成课程的学员签发一张可验证的结业凭证。学员可以查看自己的凭证。验证方可以提交该凭证，系统验证凭证是否由指定机构签发、内容是否被篡改、是否过期、是否已撤销。

## 3. 核心角色

| 角色 | 说明 |
|---|---|
| Issuer 发行方 | 培训机构，负责创建机构 DID 并签发 VC |
| Holder 持有人 | 学员，拥有个人 DID 和结业 VC |
| Verifier 验证方 | 验证 VC 是否真实有效 |
| Admin 管理员 | 管理演示数据、课程、签发记录和撤销状态 |

MVP 阶段可以不做复杂账号权限系统，可通过页面入口或接口参数区分角色。

## 4. MVP 与扩展功能范围

在原有创建、签发、验证和撤销闭环之上，当前扩展阶段增加：

- 创建时可选 `did:example` 或 `did:key`，默认 `did:example`；
- `did:example` 支持信息更新、密钥轮换和不可逆停用；
- `did:key` 支持创建、解析、签名和验签，但明确不支持更新、轮换或停用；
- 两种 Method 可以分别作为 Issuer 或 Holder 进行混合签发；
- VC 支持暂停、恢复、更新/重新签发、自动过期和撤销；
- DID、VC 和验签记录支持模糊搜索、10/20/50 分页和默认最新在前；
- 验签增加 DID 状态、密钥版本和 VC 生命周期状态检查。

### 4.1 DID 创建与查看

系统应支持创建本地演示用 DID。

验收标准：

- 能创建 Issuer DID
- 能创建 Holder DID
- 每个 DID 关联一组公私钥
- 能查看 DID Document
- DID Document 至少包含：
  - `id`
  - `verificationMethod`
  - `publicKey`
  - `authentication` 或 `assertionMethod`

### 4.2 VC 签发

系统应支持发行方为学员签发培训结业 VC。

VC 内容至少包含：

- 凭证 ID
- 凭证类型
- issuer DID
- holder DID / subject DID
- 学员姓名
- 课程名称
- 完成日期
- 签发时间
- 有效期
- 签名 proof

验收标准：

- 能选择或输入学员信息
- 能选择或输入课程信息
- 能生成一份完整 VC JSON
- VC 中包含 issuer、credentialSubject、proof
- proof 由 Issuer 私钥生成
- 签发记录能被保存和查看

### 4.3 VC 查看

系统应支持查看已签发 VC。

验收标准：

- 能展示 VC JSON
- 能展示凭证状态：有效 / 已撤销 / 已过期
- 能复制或导出 VC JSON，用于验证页面测试

### 4.4 VC 验证

系统应支持验证方提交 VC 并返回验证结果。

验证项至少包括：

| 验证项 | 说明 |
|---|---|
| 格式验证 | VC JSON 是否结构完整 |
| 签名验证 | 内容是否被篡改 |
| Issuer 验证 | issuer DID 是否存在且可解析 |
| 有效期验证 | VC 是否在有效期内 |
| 撤销验证 | VC 是否已被撤销 |

验收标准：

- 原始 VC 验证通过
- 修改学员姓名后验证失败
- 修改课程名称后验证失败
- issuer DID 不存在时验证失败
- 过期 VC 验证失败
- 已撤销 VC 验证失败
- 验证结果能展示每一项检查是否通过

### 4.5 VC 撤销

系统应支持撤销已签发 VC。

验收标准：

- 能将指定 VC 状态改为已撤销
- 已撤销 VC 再次验证时失败
- 系统能展示撤销状态

### 4.6 记录与日志

MVP 应至少保留基础记录。

验收标准：

- 能查看已创建 DID
- 能查看已签发 VC 列表
- 能查看 VC 当前状态
- 可选：保存验证记录

## 5. 非功能范围

MVP 需要满足：

| 类型 | 要求 |
|---|---|
| 运行方式 | 本地运行即可 |
| 数据存储 | SQLite、JSON 文件或本地数据库均可 |
| 安全性 | 私钥不展示在普通页面中 |
| 可演示性 | 能按固定脚本完整演示 |
| 可测试性 | 有成功和失败测试用例 |
| 可交付性 | 有 README、测试说明、验收说明 |

## 6. 明确不包含内容

MVP 不包含以下内容：

- 不上链
- 不接入真实身份认证
- 不做真实钱包 App
- 不做生产级 KMS / HSM 密钥托管
- 不做复杂权限系统
- 不做跨机构互认
- 不保证完整 W3C 生产级互操作
- 不实现 BBS+、零知识证明或正式 SD-JWT VC
- 已实现加盐 SHA-256 声明摘要与 Ed25519 摘要清单签名的教学版选择性披露，但不宣称标准互操作或匿名性

这些内容可作为后续扩展方向写入交付总结。

## 7. 推荐技术边界

建议采用：

| 模块 | 推荐方案 |
|---|---|
| DID 方法 | `did:example` 与 `did:key`；能力差异见权威需求文档 |
| 签名算法 | Ed25519 |
| VC 格式 | 参考 W3C VC Data Model 的 JSON 结构 |
| 签名实现 | 教学版 JSON 签名 proof |
| 后端 | Node.js / Python 均可 |
| 前端 | Web 页面 |
| 数据库 | SQLite 或 JSON 文件 |
| 部署 | 本地启动 |

说明文字建议写入文档：

> 本系统参考 W3C DID 与 VC 数据模型，实现教学演示版 DID 创建、VC 签发、VC 验证和 VC 撤销流程。签名、撤销、密钥管理采用本地简化实现，主要用于展示可信凭证的核心原理和业务闭环。

## 8. 演示验收流程

人工验收时按以下步骤执行：

1. 启动本地系统
2. 创建 Issuer DID
3. 创建 Holder DID
4. 创建课程信息
5. 为 Holder 签发培训结业 VC
6. 查看 VC JSON
7. 将原始 VC 提交验证，结果应为通过
8. 修改 VC 中的学员姓名，再次验证，结果应为失败
9. 修改 VC 中的课程名称，再次验证，结果应为失败
10. 撤销该 VC
11. 再次提交原始 VC 验证，结果应为失败
12. 查看签发记录和凭证状态
13. 对照验收清单确认交付完成

## 9. MVP 验收清单

```text
[ ] 系统可在本地启动
[ ] 能创建 Issuer DID
[ ] 能创建 Holder DID
[ ] 能查看 DID Document
[ ] 能签发培训结业 VC
[ ] VC 包含 issuer、credentialSubject、proof
[ ] 能查看和复制 VC JSON
[ ] 原始 VC 验证通过
[ ] 篡改学员姓名后验证失败
[ ] 篡改课程名称后验证失败
[ ] 过期 VC 验证失败
[ ] 撤销 VC 后验证失败
[ ] 能查看 VC 签发记录
[ ] 能查看 VC 当前状态
[ ] README 包含启动方式和演示流程
[ ] 测试报告包含成功与失败场景
[ ] 交付总结说明已完成内容、限制和改进方向
```

## 10. 成功标准

当系统能够在本地完成以下闭环时，即认为 MVP 达到结业任务要求：

> 创建 Issuer DID → 创建 Holder DID → 签发培训结业 VC → 查看 VC → 验证 VC 成功 → 篡改 VC 验证失败 → 撤销 VC 验证失败 → 输出测试与验收材料。

以上是最初 MVP 的最小可验收范围。当前系统的完整完成定义以产品需求说明书第 17 节和 `测试用例-DID-VC生命周期.md` 为准。
