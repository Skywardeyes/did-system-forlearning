# DID/VC 本地演示系统：苏格拉底式项目问答

> 用途：用于两人答辩前的自我追问、分角色模拟与技术复盘。
>
> 回答原则：先给结论，再说明当前代码如何实现，最后明确教学边界或改进方向。不要把“当前实现”说成“行业标准生产方案”。

---

## 一、概念与目标

### 1. 你们要解决的核心问题是什么？传统 PDF 证书为什么不够？

传统 PDF 或图片证书主要供人阅读，验证方往往需要肉眼识别、电话回访或人工查系统，难以自动判断签发来源、内容是否被篡改、证书是否过期或撤销。项目将培训结业证书建模为可验证凭证：Issuer 对 VC 签名，Verifier 自动检查身份、密钥、签名、有效期和状态，从“看证书”转为“验证证书”。

### 2. DID、DID Document、VC 分别回答什么问题？

- DID 回答“要定位哪个数字主体”；
- DID Document 回答“如何验证这个主体”，其中包含公开密钥和验证方法；
- VC 回答“哪个 Issuer 对哪个主体声明了什么事实”。

在本项目中，VC 的 `issuer` 指向 Issuer DID，`credentialSubject.id` 指向 Holder DID，`proof.verificationMethod` 指出验签公钥的位置。

### 3. DID 能证明现实中这个人是谁吗？

不能自动证明真实姓名、身份证或法人资格。DID 首先证明的是“某次签名或操作由控制对应私钥的一方完成”。现实身份需要可信机构另行核验，并用 VC 将“某 DID”与“某学生、员工或组织”的事实关联起来。本项目不做真实身份核验。

### 4. 为什么数字签名正确，不代表凭证当前有效？

签名只证明 VC 自签发后没有被修改，并且可由对应 Issuer 公钥验证；它不负责回答凭证是否被撤销、暂停、替换或超过有效期。因此系统将签名与状态拆开：签名可能通过，但 VC 状态不是 `active` 时，总体仍为 `valid=false`。

### 5. Issuer、Holder、Verifier 分别是谁？它们一定是三个不同主体吗？

Issuer 是签发声明的机构，例如培训中心；Holder 是接收和出示凭证的学员；Verifier 是核验凭证的企业或系统。它们是业务角色，不要求物理上一定是三个人或三个机构；但在典型信任模型中，Verifier 应独立于 Issuer，才能形成有意义的外部核验。

## 二、需求与业务规则

### 6. 为什么 VC 需要多种生命周期状态？

因为“内容有没有被改”与“业务上是否仍被接受”不同。`active` 表示可用，`suspended` 表示临时不可用但可恢复，`revoked` 表示确认撤销，`replaced` 表示被新凭证替代，`expired` 表示超过有效期。状态机让不同失效原因可解释、可审计。

### 7. 为什么暂停不等同于撤销后重签？

暂停通常用于临时核查，原凭证内容没有变化，后续可能恢复；撤销是终态。若每次临时暂停都撤销并重签，会产生新 ID、新签名和重复凭证，也丢失“同一凭证曾被暂停后恢复”的业务历史。

### 8. 什么情况下替换凭证，什么情况下撤销凭证？

当正文事实需要更正或更新，例如姓名、课程、完成日期填写错误，应签发新 VC，并将旧 VC 标记为 `replaced`，通过 `replaces` 与 `replacedBy` 建立追溯关系。确认原凭证不应继续被接受、且不需要新凭证替代时，使用 `revoked`。

### 9. Holder DID 被停用后，系统为什么禁止新签发？

当前代码在签发时检查 Issuer 和 Holder 的本地 DID 状态；任一方为 `deactivated` 都会拒绝签发。这是业务规则，而不是 Ed25519 的必然结果：密钥可能仍能工作，但系统选择不向已停用主体发放新凭证。若业务需求不同，也可以改变该规则，但必须同步测试和文档。

### 10. Issuer DID 停用后，历史 VC 为什么签名仍可能正确却整体无效？

停用 DID 不会回写或篡改以前已签名的 VC，所以旧签名仍可用历史或当前公钥验证。系统的 DID 状态检查会失败，因此总体无效。这体现了“密码学完整性”和“业务可接受性”的分离。

## 三、总体与详细设计

### 11. 为什么分离 HTTP、领域、密码学、存储和日志？

分离后，页面和 HTTP 层只处理请求与响应，领域层维护 DID/VC 规则，密码模块集中处理签名和摘要，存储模块负责 JSON 文件，日志模块负责脱敏与审计。这样职责清晰、便于独立测试，也能避免把私钥、状态判断和 UI 代码混在一起。

### 12. 领域层具体负责什么？为什么不把生命周期判断写在前端？

领域层 `VcService` 负责创建 DID、签发和验证 VC、生成披露证明、状态转换、密钥轮换等业务规则。前端可以隐藏不允许的按钮，但前端可被绕过；状态机必须由后端领域层强制执行，才能保证 API 调用同样不能进行非法转换。

### 13. `did:example` 与 `did:key` 的本质差别是什么？

`did:example` 是本项目自定义的本地注册型教学标识：注册表保存 DID Document、状态和密钥历史，因此可更新、轮换和停用。`did:key` 由公钥直接派生，标识与公钥绑定，不依赖注册表，因而不支持原地更新、轮换和停用。

### 14. 为什么 `did:key` 不支持更新、轮换和 DID 停用？

`did:key` 的方法内标识来自公钥指纹。公钥改变就会得到新的 DID，因此没有“保持同一 DID 但更换公钥”的位置；同时它没有独立注册表发布“已停用”状态。需要撤销的是具体 VC 时，可通过独立 VC 状态记录完成。

### 15. `did:example:` 后面的随机标识是公钥吗？

不是。它是公开的、用于定位本地注册记录的唯一标识符。验证方通过该 DID 找到 DID Document，再从 `verificationMethod.publicKeyJwk` 取得真正的 Ed25519 公钥。相比之下，`did:key` 的方法内标识才是由公钥编码得到的。

### 16. 为什么密钥轮换后要保留历史公钥？为什么不公开历史私钥？

旧 VC 的 `proof` 记录了签发时的密钥版本和验证方法。轮换后需要历史公钥才能验证旧 VC；历史私钥只会扩大泄露面，既不参与验签，也不应通过 API 或页面公开。当前项目在公开投影中删除当前和历史私钥。

## 四、密码学与验证

### 17. VC JSON 为什么可直接显示？签名和加密有什么区别？

项目中的 `proofValue` 是 Ed25519 签名，不是密文。签名让任何人都能读取 JSON 并验证它是否由 Issuer 签发、是否被改；加密则让没有解密密钥的人无法阅读内容。当前本地系统未实现 VC 内容加密、HTTPS 或端到端加密。

### 18. 为什么需要 `stableStringify()`？

签名针对字节序列。两个 JSON 对象即使语义相同，若对象键顺序不同，普通序列化可能产生不同字节，导致验签失败。`stableStringify()` 递归排序对象键并保持数组顺序，使同一业务内容得到确定的待签名字节。它是项目教学规范化方案，不等同于正式 W3C canonicalization。

### 19. 七项验证分别防止什么问题？

格式检查防止缺字段；Issuer 解析确认签发方存在；DID 状态检查签发方是否被停用；密钥版本检查防止错误公钥或轮换后找不到旧密钥；签名检查篡改与伪造；有效期检查时间失效；VC 状态检查暂停、替换、撤销等业务失效。任何一项失败均不接受凭证。

### 20. 验证方通过什么路径找到 Issuer 公钥？

验证方读取 VC 的 `issuer`，在当前项目的本地 JSON 注册表中找到对应 DID，再根据 `proof.verificationMethod` 与 `keyVersion` 解析当前或历史公钥。生产环境会替换为相应 DID Method 的公开解析机制、受信任注册表或解析器。

### 21. 本地 JSON 注册表与跨机构 DID 解析服务有什么区别？

本地注册表只在本系统内有效，由同一应用维护，适合离线教学演示。跨机构解析需要公开可访问、可治理、可用性和安全性更高的 DID Method 基础设施，并处理信任、更新、撤销、缓存和互操作问题。

## 五、选择性披露与隐私

### 22. 选择性披露隐藏了什么，又没有隐藏什么？

当前方案不把未选字段的原值和盐放入 `disclosedClaims`，例如只公开课程和完成日期时，不公开姓名原值。它没有隐藏字段路径、`credentialId`、Issuer、有效期和摘要清单；这些信息仍可能暴露数据结构或形成关联线索。

### 23. 随机盐为什么必须存在？

若摘要只是 `SHA-256(value)`，像 `Completed`、常见姓名等低熵值可被攻击者枚举并比对。项目为每个字段生成独立随机盐，摘要覆盖“字段路径、盐和值”，未披露字段不提供盐，从而提高字典猜测难度。

### 24. 企业只要求证明已完成培训，学生应提交什么？企业能看到什么？

学生应只选择 `credentialSubject.achievement`，必要时再选择课程字段。企业会看到选中字段的路径、值和盐，以及 Issuer DID、凭证 ID、有效期、摘要清单和 Issuer 对清单的签名；不会看到未选字段的原值与盐。当前实现不能做到“只给出一个是/否结论而完全不暴露字段值”。

### 25. 为什么当前方案不是零知识证明？

零知识证明要求验证方除了结论外不获得额外秘密信息。当前证明会公开所选字段的值和盐，并携带稳定摘要清单、字段路径、凭证 ID 等信息；它实现了部分字段不外发和公开字段防篡改，但不具备零知识条件证明或证明随机化。

### 26. 为什么当前方案多次出示可能被关联？

同一凭证会携带稳定的 `credentialId`、Issuer、有效期、相同摘要清单和相同签名。即使披露字段不同，多个验证方也可能通过这些稳定标识判断它们来自同一凭证。

### 27. SD-JWT 相比教学版方案主要增强什么？BBS+ 又增强什么？

SD-JWT 提供标准化的 JWT、Disclosure 和可选 Holder key binding 机制，便于钱包和验证器互操作。项目现已实现 RFC 9901 的核心 Issuer-signed JWT、`_sd` 摘要和 Disclosure 紧凑表示，但尚未实现 Holder key binding、钱包本地保管或 SD-JWT VC 应用配置。BBS+ 支持 Holder 从原始多消息签名生成随机化派生证明，密码学证明本身难以与原始签名或其他展示关联，当前项目未实现。

### 28. BBS+ 是否意味着完全匿名？

不意味着。BBS+ 主要提供派生证明的密码学不可关联性；独特的披露字段、固定 Holder 标识、撤销查询、网络 IP、设备指纹、业务登录和验证方串通仍可关联身份。完整匿名是密码学、协议、网络和业务治理共同决定的结果。

### 29. 为什么没有直接接入 BBS+，且 SD-JWT 只实现核心格式？

这是范围与风险取舍，不是把简化实现误称为标准实现。BBS+ 需要专门密码学库和严格实现审计；SD-JWT 的完整生产应用还要处理 Holder binding、钱包、受众与 nonce、信任链、状态服务和互操作。课程项目使用 Node 内置 Ed25519 实现 RFC 9901 核心格式，同时明确保留这些未完成边界；BBS+ 则不在没有经审计密码库的情况下冒险自研。

## 六、工程质量与测试证据

### 30. Node 测试与 UI 测试分别说明什么？

Node 测试覆盖算法、领域规则、存储、API、安全边界等底层与服务行为；Chromium UI 测试覆盖真实页面导航、表单、按钮、台账和响应式流程。两者互补：UI 成功不能证明密码学或状态机正确，单元测试通过也不能保证用户能完成流程。

### 31. 测试全部通过能证明系统没有 Bug 吗？

不能。它只能说明在已定义的用例、环境与断言下，程序符合预期。项目应将结论表述为“已覆盖场景通过”，而非“系统绝无缺陷”；生产环境还需要压力测试、渗透测试、依赖审计、威胁建模和持续监控。

### 32. 什么是可验证测试证据链？

全量测试记录保存原始 Node/UI 日志、环境版本、Git 状态、结果统计、用例清单和浏览器报告，再生成证据清单与 SHA-256 校验值。这样可检查批次文件是否被新增、删除或修改，比只展示一张“测试通过”截图更可追溯。

### 33. 为什么测试记录要包含 Git 状态、环境版本、原始日志和校验值？

因为测试结论离不开“测试的是哪份代码、在哪个环境、原始输出是什么”。Git 状态能说明是否有未提交改动；环境版本帮助复现；原始日志保留失败细节；校验值证明留存证据未被静默修改。

### 34. 如何证明未披露字段没有通过普通 API 泄露？

代码通过 `publicCredential()` 从普通凭证响应移除 `disclosureMaterial`，其中才存有全部字段的值与盐；API 与安全测试会断言普通状态、凭证等响应不包含未披露字段随机盐或完整披露材料。仍应明确：完整 VC 本身若被发送，当然会包含其正文；选择性披露的前提是不发送完整 VC。

### 35. 测试文档是否存在旧统计数字？为什么是答辩风险？

存在。README、当前汇报和演讲稿使用 Node 130/130、Chromium UI 44/44；但《测试与人工验收》《测试缺陷报告》等历史材料仍保留 114/114、6/6、120/120 等旧基线，汇报 HTML 顶部还出现过 208/208 的不同口径。统计数字不一致会削弱“可验证证据”的可信度，答辩前必须统一为同一最终证据批次及其命令。

## 七、安全、边界与生产化

### 36. 普通 API 不返回私钥，是否就代表私钥安全？

不是。当前公开投影能防止普通页面和 API 直接泄露私钥，但私钥仍以明文 JWK 保存在本地 `data/store.json`。这适合本机教学，生产环境应使用 KMS、HSM、安全钱包、访问控制、加密存储和密钥轮换策略。

### 37. 原子写入和 Promise 队列解决了什么，又不能解决什么？

临时文件写入后 `rename` 降低半写 JSON 风险；`JsonStore.update()` 队列可把“读取—修改—保存”串行化，避免多个受保护业务写操作互相覆盖。它不提供多进程/多机器分布式事务、数据库索引、崩溃恢复、权限隔离或高并发能力。

### 38. 验证记录与日志写入是否一定不会并发覆盖？

当前不能做出“完全不会”的承诺。`verifyCredential()` 和 `verifyDisclosurePresentation()` 先在队列外读取状态，再调用 `save` 写入验证台账；`LogStore.append()` 也先读取再排队替换。并发下可能出现两个请求都基于同一旧快照写回，导致其中一条记录丢失。现有并发测试主要覆盖 DID 创建的 `JsonStore.update()` 路径，尚未覆盖这些读改写路径。生产或进一步完善时，应将验证台账追加和日志追加改为同一队列中的原子更新，并增加并发回归测试。

### 39. 当前系统为什么不适合直接部署到公网？

系统没有账号认证、授权、HTTPS、生产级密钥托管、数据库事务、限流、跨机构信任、标准状态服务和完整安全运维。它默认只监听 `127.0.0.1`，目的就是缩小教学演示的攻击面，而不是提供公网服务。

### 40. 生产化优先改什么？

优先顺序取决于风险，但通常先补齐密钥与访问控制：KMS/HSM、认证授权、HTTPS、审计和最小权限；再替换本地 JSON 为数据库和可靠事务；随后接入正式 DID 解析、状态机制和标准 cryptosuite。若选择性披露是核心需求，应迁移到正式 SD-JWT VC 或经过审计的 BBS+ 实现。

### 41. 为什么 DID/VC 不替代 IAM、OAuth、RBAC？

IAM、OAuth 与 RBAC 擅长在资源方系统内做账号、会话、角色和策略执行。DID/VC 擅长表达可携带的身份、资格或授权证明，尤其适合跨机构传递。最终是否允许访问某个资源，仍应由资源方 IAM、API 网关和策略引擎决定。

### 42. 哪些跨机构场景下 DID/VC 更有价值？

当学校、企业、培训机构、供应商、平台等多个独立组织需要反复验证同类资格，并且不希望每一对组织都重复建账号、同步档案和人工核验时，VC 可减少重复证明。固定的两家长期合作机构，直接使用 OIDC/SAML 联邦和 IAM 映射通常更简单。

### 43. DID 在 AI Agent 安全中能做什么，不能做什么？

DID 可为 Agent 提供可验证身份、密钥轮换、组织归属与授权凭证，帮助 API 网关审计“哪个 Agent 在何时以何种资格调用工具”。它不能自动防御提示注入、错误推理、恶意检索内容或已获授权后的不合理操作；还需要工具参数校验、策略引擎、隔离、限额、人工审批和行为监控。

## 八、反思与答辩追问

### 44. 最容易被误解为生产能力的功能是什么？

选择性披露、DID 生命周期和“安全存储”最容易被过度理解。应主动说明：当前选择性披露不是 BBS+/正式 SD-JWT；`did:example` 是本地教学标识；私钥只是避免通过普通 API 泄露，并非 KMS/HSM；本地状态不是跨机构可解析状态服务。

### 45. 如果老师说“这只是本地 JSON，不算 DID”，如何回答？

可以承认它不是跨机构去中心化网络，但说明 DID Core 定义的是标识和 DID Document 的数据模型，具体创建和解析由 DID Method 决定。本项目的 `did:example` 用本地注册表演示可更新身份及其生命周期，`did:key` 演示公钥直接派生身份；准确定位是 DID 教学实现，而非生产 DID 网络。

### 46. 如果老师说“选择性披露只是自己写了个摘要”，如何回答？

应承认它不是正式 SD-JWT/BBS+，但说明它不是普通隐藏字段：每条声明使用独立随机盐和 SHA-256 摘要，Issuer 对摘要清单进行 Ed25519 签名，Verifier 重算公开字段摘要并验签，因此能检测公开字段篡改并避免未选字段原值外发。它的价值是教学性部分声明验证；不足是可关联、字段结构泄露且没有零知识能力。

### 47. 使用 AI 辅助开发后，如何证明自己理解并验证了代码？

AI 输出只作为候选方案。团队将需求写成状态机、接口契约和测试断言；关键代码可逐段解释；每次修改通过 Node、API、安全和 UI 测试复核；并保留缺陷修复、测试证据和人工演示记录。最终对功能范围、取舍和结论负责的是项目成员，而不是 AI。

### 48. 如果只能保留一个项目亮点，选什么？

建议选择“将可信凭证验证拆成可解释、可测试的完整闭环”。七项验证说明为什么有效或失败，生命周期说明签名正确不等于业务有效，选择性披露说明最小化数据共享，测试证据链说明结论可复核。这样比单独强调某一个密码学名词更能体现软件工程价值。

## 最后复习：四句必须说准确的话

1. **签名正确不代表凭证当前有效。**
2. **DID 证明密钥控制关系，不自动完成真实身份核验。**
3. **当前选择性披露隐藏未选字段原值，但不是零知识证明、不是 BBS+、也不是正式 SD-JWT。**
4. **本项目是可运行、可解释、可测试的本地教学系统，不宣称生产级互操作或安全能力。**
