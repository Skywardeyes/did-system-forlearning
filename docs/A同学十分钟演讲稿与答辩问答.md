各位老师好，我们的项目是“信证台：DID/VC 本地演示系统”。

## 00:35–02:10｜需求分析

[画面：需求分析——问题定义与项目目标]

首先是需求分析。

传统电子证书通常是一张图片或普通文件。它可以被人查看，却很难让验证方独立判断四个问题：第一，证书究竟由谁签发；第二，证书内容有没有被修改；第三，签发者的身份当前是否有效；第四，证书是否已经暂停、过期、被替换或者撤销。

因此，我们把“看证书”转化为“验证证书”，并将这些疑问落实成可执行、可测试的软件需求。

[下滑到业务角色与用例图]

系统包含四类角色。

Issuer，也就是发行方，负责创建机构 DID、维护身份并签发 VC；Holder，也就是持有人，接收并展示自己的凭证；Verifier，也就是验证方，提交 VC 并查看逐项验证结果；演示管理员负责重置环境、查询台账和审计日志。

围绕这些角色，我们确定了九类核心功能：DID 创建与查看、DID 生命周期、VC 签发、七项完整验证、VC 生命周期、查询分页、结构化日志、一键演示数据，以及教学版选择性披露。

[下滑到功能需求和非功能需求]

除了功能正确，我们还提出了五项主要非功能要求。

第一是可运行性，系统不依赖数据库、区块链和外部网络；第二是安全性，普通 API 不返回私钥，日志写入前进行递归脱敏；第三是可靠性，通过更新队列和原子文件替换降低并发覆盖与半写文件风险；第四是可测试性，算法、业务规则、API、安全边界和 UI 都有对应测试；第五是可解释性，验证失败不能只返回一个 false，而要明确告诉用户失败发生在哪一项。

这里也要说明项目边界：我们参考 DID Core 和 VC Data Model 2.0 的核心结构，但这是教学实现，不进行真实身份核验，不接入真实链上注册，也不宣称兼容正式生产级密码套件。

## 02:10–03:40｜总体设计

[画面：系统总体架构图]

第二阶段是总体设计。

我们采用本地 B/S 架构。最上层是浏览器表现层，由原生 HTML、CSS 和 JavaScript 构成；浏览器通过 HTTP 和 JSON 调用 `server.js` 提供的接口；业务层由 `VcService`、DID Method 注册表、密码学模块、查询模块和日志服务组成；最下层通过 `JsonStore` 和 `LogStore` 分别保存业务状态与结构化日志。

一次签发操作会从页面进入 HTTP 路由，再由领域服务校验 Issuer、Holder、字段和有效期，然后调用 Ed25519 签名，最后保存凭证并记录审计日志。

[下滑到模块职责与代码映射]

在代码结构上，我们没有把全部逻辑写入一个文件。`server.js` 负责 HTTP 协议和路由；`vc-service.js` 负责 DID、VC 和生命周期规则；`crypto.js` 负责 Base58、稳定序列化、加盐 SHA-256 摘要以及 Ed25519 签名验签；`store.js` 负责原子存储和私钥公开隔离；日志模块负责脱敏、查询和持久化。

这样划分的主要价值是职责清晰，每一层都可以独立测试。

技术选型方面，Node.js 20 已经提供 HTTP 和 Ed25519 能力；JSON 文件易于查看和重置；原生前端减少了答辩现场对网络、数据库服务和复杂构建工具的依赖。当然，这种选择适合单机教学演示，不等同于生产系统架构。

[下滑到“详细设计”]

## 03:40–05:40｜详细设计

[画面：逻辑 ER 图]

第三阶段是详细设计。

虽然项目实际使用 JSON 文件而不是关系型数据库，我们仍然通过逻辑 ER 图表达领域对象之间的关系。

核心对象包括 DID 身份、历史密钥、凭证记录、完整 VC 验证记录、披露证明验证记录和结构化日志。一个 DID 可以作为发行方签发或持有多张 VC；一张 VC 可以产生多次完整验证和披露验证记录；DID 轮换密钥后，旧公钥进入历史密钥集合；凭证替换则通过 `replaces` 和 `replacedBy` 保留新旧凭证的追溯关系。

[下滑到 did:example 与 did:key 对比]

项目支持两种 DID Method。

`did:example` 是注册型、可变化的教学标识，支持更新、密钥轮换和不可逆停用；`did:key` 由 Ed25519 公钥经过 multicodec 前缀和 Base58btc 编码得到，标识与密钥直接绑定，因此不支持更新、轮换和停用。

这两种方法让我们能够同时演示“可维护身份”和“公钥衍生身份”的差异。

[下滑到 VC 生命周期状态机]

VC 的初始状态是 active。active 可以暂停为 suspended，suspended 可以恢复为 active；active 或 suspended 都可以进入 replaced 或 revoked；超过有效期后进入 expired。

其中暂停是可逆状态，而 replaced、revoked 和 expired 是终态。这个状态机由代码中的 `VC_TRANSITIONS` 明确限制，不能从撤销状态重新恢复。

这里有一个非常重要的设计结论：密码学签名正确，不代表凭证当前有效。例如，一张已撤销的 VC 内容没有变化，所以签名仍可能正确，但它的业务状态已经失效，最终验证必须失败。

[下滑到七项验证规则]

因此，我们没有把“验证”简化成一次验签，而是拆成七项：格式完整性、Issuer DID 解析、DID 当前状态、密钥版本和验证方法、Ed25519 签名、有效期以及 VC 当前状态。

七项检查分别回答不同问题，最终只有全部通过，系统才返回 `valid=true`。这样用户能够清楚区分凭证是被篡改、已经过期，还是被业务撤销。

在完整 VC 验证之外，我们提供两种选择性披露方式。第一种是教学版摘要证明：系统为每个声明生成随机盐和 SHA-256 摘要，由 Issuer 对摘要清单签名。第二种是 RFC 9901 核心 SD-JWT：Issuer 使用 EdDSA 签发 JWT，JWT 中保存 `_sd` 摘要，Holder 只拼接本次选中的 Disclosure。两种方式都可只公开课程和完成日期，并结合 Issuer、有效期和 VC 状态判断结果；成功与失败都会进入披露证明验证台账。需要说明的是：教学版协议不是 BBS+ 或零知识证明；SD-JWT 实现也尚未加入 Holder key binding、钱包本地生成或正式 SD-JWT VC 应用配置。

## 05:40–07:15｜编码实现

[画面：VC 签发与验证时序图]

第四阶段是编码实现。

从时序图可以看到，用户在页面填写发行方、持有人和课程信息后，浏览器调用创建凭证接口。`VcService` 先读取身份和状态，完成业务校验，再调用密码学模块使用 Issuer 私钥签名。签名结果写入 `proofValue`，凭证通过原子写入保存，同时记录 `VC_ISSUE` 审计日志。

验证时，系统读取 Issuer DID、当前或历史密钥以及 VC 状态，再使用 DID Document 中的公钥验签，汇总七项结果，并保存验证记录和 `VC_VERIFY` 日志。

[下滑到 stableStringify 代码]

这里重点说明两个工程问题。

第一个是确定性签名。数字签名针对的是字节序列。同一个 JSON 对象如果键顺序不同，普通序列化可能产生不同字节，导致验签不稳定。因此我们实现了 `stableStringify`，递归排序对象键，同时保持数组原有顺序，使相同业务内容产生确定的签名载荷。

[下滑到原子写入代码]

第二个是存储可靠性。系统不会直接覆盖正式 JSON 文件，而是先写入临时文件，再通过 rename 整体替换；同时使用 Promise 更新队列串行执行“读取、修改、保存”，避免两个并发请求读取同一旧状态后互相覆盖。

[下滑到公开投影、日志脱敏和 HTTP 安全边界]

安全方面，`publicDid()` 会移除当前私钥和历史私钥，普通凭证 API 也不会返回未披露字段的随机盐和完整披露材料；日志中的敏感字段会在持久化前脱敏；HTTP 层还限制 JSON 内容类型、1 MB 请求体大小、静态目录访问范围，并设置基础安全响应头。

这些措施不能把教学系统变成生产系统，但能够建立清晰的安全边界。

## 四、答辩老师提问模拟与推荐回答

## 第一组：项目概念

### 问题 1：DID、DID Document 和 VC 分别是什么？

推荐回答：

> DID 是主体的去中心化标识符，回答“是谁”；DID Document 是这个标识的公开说明文档，包含验证方法和公钥，回答“如何验证他”；VC 是发行方向持有人作出的可验证声明，回答“谁向谁证明了什么”。在我们的系统中，Issuer 使用私钥签发 VC，Verifier 根据 Issuer DID 找到 DID Document，再使用其中的公钥验签。

### 问题 2：为什么这个系统需要 DID，直接保存一个公钥不行吗？

推荐回答：

> 如果只保存公钥，我们只能得到一段密钥材料，缺少稳定的主体标识、验证方法 ID、密钥用途和生命周期表达。DID 把主体与 DID Document 关联起来，使 VC 可以通过 issuer 引用身份，再从 verificationMethod 中解析公钥。教学场景下直接保存公钥也能完成一次验签，但不能清晰表达身份解析和密钥轮换等问题。

### 问题 3：Ed25519 在项目中解决什么问题？

推荐回答：

> Ed25519 用于数字签名。Issuer 使用私钥对稳定序列化后的 VC 内容签名，Verifier 使用 DID Document 中的公钥验证。它证明内容与对应私钥之间的关系，并检测签发后的内容修改。但它不能单独判断 VC 是否撤销或过期，所以还要结合 DID 状态、有效期和 VC 状态。

### 问题 4：数字签名和加密有什么区别？

推荐回答：

> 加密主要解决内容保密，数字签名主要解决来源认证和完整性。我们的 VC JSON 是可读的，proofValue 不是对 VC 的加密，而是签名。任何人都可以读取凭证，但只有持有 Issuer 私钥的一方能够生成可由公钥验证的签名。

### 问题 5：为什么签名通过，凭证仍然可能无效？

推荐回答：

> 签名只说明凭证内容没有被修改，并且对应某个私钥。凭证仍可能已经暂停、撤销、替换或过期，Issuer DID 也可能已停用。因此我们把验证拆成七项，签名只是其中一项，七项全部通过才返回有效。

## 第二组：需求与设计

### 问题 6：为什么你们选择 B/S 架构？

推荐回答：

> B/S 架构便于在浏览器中演示，同时保留前后端 HTTP 契约。表现层、API、业务层和持久化层可以分别测试。部署时只需要启动本地 Node.js 服务，不需要安装桌面客户端，也不依赖外部服务器，适合答辩环境。

### 问题 7：为什么不用数据库，而使用 JSON 文件？

推荐回答：

> 这是根据教学演示范围作出的取舍。JSON 文件可直接查看、容易重置、不需要数据库服务，能够降低现场环境风险。为了提高可靠性，我们增加了更新队列和临时文件加 rename 的原子写入方式。但它不具备生产数据库的事务、索引、权限和并发能力，因此我们明确把系统限制在单机教学场景。

### 问题 8：没有关系数据库，为什么还画 ER 图？

推荐回答：

> ER 图在这里表达的是逻辑数据模型，而不是物理 SQL 表。即使数据保存在 JSON 中，DID、历史密钥、凭证记录、验证记录和结构化日志之间仍然存在一对多和替换引用关系。我们在图上明确标注了“逻辑 ER 图，实际保存为 JSON”。

### 问题 9：为什么同时实现 `did:example` 和 `did:key`？

推荐回答：

> 两者用于展示不同类型的 DID。`did:example` 是注册型教学标识，支持更新、密钥轮换和停用；`did:key` 由公钥指纹直接生成，标识与密钥绑定，因此不可变。两种 Method 让我们能够比较可维护身份与公钥衍生身份的设计差异。

### 问题 10：凭证替换为什么不直接修改旧 VC？

推荐回答：

> VC 已经被数字签名，直接修改正文会破坏签名，也会丢失历史。项目在替换时签发一个新 ID，把旧凭证设为 replaced，并用 replaces 和 replacedBy 建立双向追溯关系。这样既保留旧证据，也能明确新凭证是对哪张凭证的替代。

### 问题 11：为什么撤销状态放在本地记录，而不写进 VC？

推荐回答：

> 已签发的 VC 正文不能随意修改，否则签名失效。撤销、暂停和恢复属于签发后的业务状态，所以保存在外层 CredentialRecord 中。验证时既检查 VC 的密码学签名，也查询本地状态。生产系统通常会使用可公开解析的状态列表或注册表，我们的实现是本地教学版。

## 第三组：代码实现

### 问题 12：为什么不能直接使用 `JSON.stringify()` 签名？

推荐回答：

> 数字签名针对字节序列，而 JSON 对象键顺序可能不同。两个业务内容相同的对象如果序列化顺序不同，可能得到不同字节。我们使用 stableStringify 递归排序对象键，数组保持原顺序，使相同内容得到确定的签名载荷。需要说明的是，这属于项目的教学规范化方法，不是正式 W3C canonicalization 实现。

### 问题 13：私钥存在哪里？是否安全？

推荐回答：

> 私钥保存在本地被 Git 忽略的 `data/store.json` 中，普通 API 和页面通过 `publicDid()` 删除当前和历史私钥后再返回。这个设计可以防止普通公开接口直接泄露私钥，但本地文件没有加密，因此不适合生产。生产环境应使用密钥管理服务、硬件安全模块或受控密钥库。

### 问题 14：密钥轮换后，旧 VC 如何继续验证？

推荐回答：

> VC proof 中保存 verificationMethod 和 keyVersion。轮换时，系统把旧公钥、旧验证方法 ID、版本和停用时间保存到 keyHistory。验证旧 VC 时，服务根据 proof 中的版本解析当前或历史公钥，因此旧凭证仍然可以验签。

### 问题 15：你们如何避免 JSON 文件并发写入丢数据？

推荐回答：

> 我们使用 Promise 更新队列，把“读取—修改—保存”事务串行化，避免两个请求同时读取旧状态后互相覆盖。保存时先写临时文件，再 rename 替换正式文件，降低半写文件风险。但这是单进程内的保护，不等同于多进程数据库事务。

### 问题 16：你们的日志脱敏是否绝对安全？

推荐回答：

> 不能说绝对安全。当前实现会递归识别 private、secret、token、password、authorization、cookie、proofValue、credential 和请求体等敏感字段并替换，同时有自动化测试验证常见情况。但字段命名变化、敏感信息混入普通文本等情况仍可能漏检。生产环境还需要数据分类、白名单日志结构和集中审计策略。

### 问题 17：为什么服务只监听 `127.0.0.1`？

推荐回答：

> 因为系统定位是本机教学演示。只监听回环地址可以避免默认暴露到局域网，缩小攻击面。如果未来需要多人访问，就必须增加认证授权、HTTPS、部署隔离、密钥托管和更严格的安全评估，而不应该只修改监听地址。

## 第四组：测试与质量

### 问题 18：130 项 Node 测试和 44 项 UI 测试分别测什么？

推荐回答：

> Node 测试覆盖单元、集成、API、功能和安全层，包括稳定序列化、Ed25519、加盐摘要、选择性披露、状态机、存储并发和隐私边界等。UI 测试使用 Chromium 覆盖六个入口、身份、签发、完整验证、选择性披露与验证台账、生命周期、日志和响应式布局。两者关注点不同，不能用 UI 测试代替底层算法测试。

### 问题 19：如何证明测试记录没有被事后修改？

推荐回答：

> 每个测试批次会保存 Git 提交、工作区状态、环境、阶段退出码、原始日志、用例清单和浏览器报告，然后生成 evidence-manifest 和 SHA-256 校验文件。验证命令会重新计算并检查文件缺失、新增和内容修改。这提供本地完整性证据，但不是带第三方时间戳的不可抵赖证明。

### 问题 20：如何保证披露证明没有偷偷包含姓名？

推荐回答：

> 生成接口只把用户选中字段的路径、值和盐放入 `disclosedClaims`；未选字段的原值和盐不会进入证明，只有摘要留在经过 Issuer 签名的摘要清单中。普通状态与凭证 API 会通过公开投影移除完整披露材料。自动化测试还直接断言只公开课程时，返回 JSON 不包含姓名和未公开字段的盐。需要主动说明：摘要清单仍包含字段路径，因此它隐藏的是字段值，不保证隐藏数据结构，也不提供不可关联性。

### 问题 20A：你们的选择性披露与 BBS+ 有什么区别？

推荐回答：

> 我们采用加盐 SHA-256 声明摘要和 Ed25519 摘要清单签名，实现了未选字段不进入证明、公开字段可验证和篡改可检测。但它不是零知识派生证明，多次展示仍可能通过摘要清单关联，也不是正式 SD-JWT VC。BBS+ 可以由 Holder 从多消息签名派生随机化证明，隐私能力更强，但 Node 原生不支持，需要审慎引入第三方密码学库，所以本阶段没有冒充实现。

### 问题 21：你们发现过什么真实缺陷？

推荐回答：

> 例如 JSON Store 最初需要处理并发更新覆盖问题。测试构造并发创建场景后发现，如果每个请求独立读取旧状态再写回，就可能丢失更新。我们把整个更新事务放入 Promise 队列串行执行，并通过集成测试回归。其他修正还包括 Base58 边界、Content-Type 和请求大小限制、路径穿越防护以及分页一致性。

### 问题 22：只测 Chromium 是否足够？

推荐回答：

> 对当前课程答辩的目标浏览器来说够用，但不能据此宣称跨浏览器兼容。我们的材料明确说明 UI 自动化只覆盖 Chromium。如果进入生产，应补充 Firefox、WebKit、不同操作系统、辅助技术和更多移动设备测试。

## 第五组：标准、边界与改进

### 问题 23：你们的 VC 是否完全符合 W3C 标准？

推荐回答：

> 不能这样表述。我们的数据结构参考 VC Data Model 2.0 的核心字段，DID Document 也参考 DID Core，但完整 VC 的 `EducationalEd25519Signature2026`、`eddsa-stable-json-demo-2026`，以及披露证明的 `EducationalSelectiveDisclosureProof2026` 都是教学协议，不是正式注册的 Data Integrity cryptosuite，也不是正式 SD-JWT 或 BBS+。项目没有完成生产级规范化、跨机构解析和互操作认证。

### 问题 24：这个系统为什么不需要区块链？

推荐回答：

> DID 和 VC 的核心概念并不要求所有实现都必须使用区块链。DID Method 决定标识如何创建和解析。我们的 `did:example` 使用本地注册表，`did:key` 直接由公钥推导。区块链可以作为某些 Method 的注册与状态基础设施，但不是完成本项目教学目标的必要条件。

### 问题 25：如果要投入生产，最先需要改什么？

推荐回答：

> 第一，替换本地私钥存储，使用 KMS 或 HSM；第二，使用正式标准的 cryptosuite 和规范化算法；如果需要生产级选择性披露，则迁移到正式 SD-JWT VC 并考虑 Holder key binding，或采用经过审计的 BBS+ 实现；第三，引入持久化数据库、事务、备份和访问控制；第四，建立真实 DID 解析与凭证状态机制；第五，增加账号权限、HTTPS、审计平台、隐私保护和跨浏览器兼容测试。生产化不是简单把 JSON 换成数据库，而是整体安全模型升级。

### 问题 26：项目最大的创新或价值是什么？

推荐回答：

> 我们的价值不在于发明新的密码算法，而在于把 DID/VC 的抽象概念变成一个完整、可解释、可操作、可测试的教学闭环。特别是把验证拆成身份、密钥、签名、时间和状态七个层次，并用状态机、分层测试和证据链说明“为什么可信、为什么失败”。

### 问题 27：你个人在这个项目中学到了什么？

推荐回答：

> 技术上，我理解了数字签名只能证明来源和完整性，不能代替身份状态、有效期和撤销检查；工程上，我学会用需求、状态机、接口契约和测试把抽象规则落到代码；在 AI 协作方面，我认识到 AI 可以提高分析和实现速度，但最终结论必须由我们通过源码阅读、运行结果和测试证据确认。

### 问题 28：为什么停用 Holder 的 DID 后，系统仍可能允许向他签发凭证？

推荐回答：

> 因为当前业务规则重点校验 Issuer 是否有资格签名，Holder DID 主要作为凭证主体标识；Holder 被停用并不会让 Issuer 的私钥或签名能力失效。因此，这不是密码学验签错误，而是当前教学系统的业务策略选择。若需求规定“不得向已停用主体签发”，就应在签发服务中增加 Holder 状态校验，并补充 API、功能和 UI 回归测试。答辩时应区分“当前代码允许什么”和“生产业务应该限制什么”。

### 问题 29：选择性披露中的随机盐到底解决了什么问题？

推荐回答：

> 如果只计算 `SHA-256(字段值)`，姓名、等级、日期等取值空间较小的字段可以被攻击者枚举猜测：把候选值逐个求摘要，再与清单对比。加入足够随机且每个声明独立的盐后，攻击者不知道盐，就不能直接使用预计算表反推字段值。盐不负责签名，也不隐藏已公开字段；它主要提高未披露字段抵抗字典猜测和彩虹表攻击的能力。

### 问题 30：Holder 能不能自己伪造一个披露证明？

推荐回答：

> Holder 可以任意拼接 JSON，但无法在不知道 Issuer 私钥的情况下伪造有效的摘要清单签名。验证方会先验证 Issuer 对摘要清单的 Ed25519 签名，再使用披露字段的路径、值和盐重新计算摘要并与已签名清单比对，同时检查 Issuer、有效期和 VC 状态。因此，单纯修改公开值、盐或摘要都会导致验证失败。不过当前方案不包含正式 SD-JWT 的 Holder key binding，不能等同于证明“当前出示者一定就是原 Holder”。

### 问题 31：为什么还要建立披露证明验证台账？它会不会反而泄露隐私？

推荐回答：

> 台账用于记录何时验证、验证了哪份证明、总体是否通过以及各项检查结果，便于审计、故障定位和课堂展示；成功和失败都要记录，才能形成完整证据链。但台账本身也属于敏感数据，不能无节制保存披露值、随机盐或完整证明。当前实现通过公开投影和日志脱敏降低泄露风险；生产环境还需要最小化字段、访问控制、保存期限、加密和删除策略。

### 问题 32：Node 130 项和 UI 44 项全部通过，能否证明系统没有漏洞？

推荐回答：

> 不能。测试通过只能说明当前代码在已定义的用例、环境和断言下符合预期，不能穷尽所有输入、并发条件和攻击方式。我们的可信度来自多层测试、反例测试、真实浏览器流程和可校验的测试证据，但结论必须限定为“已覆盖场景通过”。若进入生产，还需要威胁建模、代码审计、依赖审计、渗透测试和持续监控。

### 问题 33：既然使用 AI 开发，你们如何避免自己并不理解生成的代码？

推荐回答：

> 我们把 AI 输出视为候选方案，而不是最终事实。每次关键修改都要回到三个依据：需求和状态机是否一致，源码能否逐段解释，自动化测试和实际操作是否支持结论。对于 Ed25519、DID 生命周期和选择性披露等关键部分，我们还专门形成逐行解释文档，并主动记录教学实现与正式标准之间的差距。答辩时我们只陈述自己能够从代码和证据中说明的内容。

## 五、老师连续追问模拟

### 模拟一：关于“去中心化”

老师：你们的数据都保存在本地，为什么还能叫 DID？

A：

> DID 的结构和解析方式由 DID Method 决定，并不意味着所有 DID 都必须上链。我们的 `did:example` 是本地教学注册表，主要演示可更新身份；`did:key` 则由公钥直接推导，不依赖中心注册表。我们会把它称为 DID 教学实现，而不会宣称已经构建跨机构去中心化网络。

老师：那你们的 `did:example` 和普通数据库 ID 有什么区别？

A：

> 仅从存储位置看，它确实由本地注册表维护；区别在于它按照 DID 的结构组织，并关联 DID Document、verificationMethod、authentication 和 assertionMethod，用于凭证签发与验证。它的价值是教学表达和统一验证模型，不是网络去中心化程度。

### 模拟二：关于签名与撤销

老师：撤销后为什么不让签名直接失败？

A：

> 因为撤销没有修改原始 VC，原签名在密码学上仍然正确。如果为了让签名失败而修改 VC，反而会破坏原始证据。正确做法是分别保留签名结论和状态结论：签名回答内容是否被改，状态回答凭证当前是否可接受。

老师：那验证方必须访问你们的本地系统吗？

A：

> 在当前教学实现中，是的，状态来自本地注册记录。生产环境需要可公开解析的状态列表、状态服务或其他跨机构机制，这正是当前项目明确的边界之一。

### 模拟三：关于测试可信度

老师：测试都是你们自己写的，怎么证明系统真的正确？

A：

> 自编测试不能形成数学意义上的绝对证明，但可以通过多层独立验证提高可信度。我们既测底层算法，也通过 HTTP 黑盒测试接口，再通过 Chromium 模拟用户旅程；同时覆盖篡改、非法状态、私钥泄露和路径穿越等反例。测试证据还绑定 Git 状态、运行环境和原始日志。我们把结论限定为“在这些明确场景和环境下通过”，而不是宣称没有任何缺陷。

### 模拟四：关于 AI 使用

老师：这个项目是不是主要由 AI 写的？

A：

> AI 参与了需求整理、方案比较、代码草案、测试补充和失败分析，但功能目标、范围边界、方案取舍、源码审核、真实运行和最终结论由我们负责。我们没有把 AI 输出直接视为正确答案，而是通过测试失败、代码检查和人工判断形成迭代闭环。B 同学后面会展示具体协作过程和证据。

老师：如果 AI 生成了错误代码，你们怎么发现？

A：

> 首先把需求转化为状态机、接口契约和测试断言；其次运行分层测试和真实 UI 流程；再次检查安全边界和测试证据。如果输出与需求或测试不一致，就根据失败信息定位并修改。AI 输出只是候选方案，测试和人工审核才决定是否进入项目。

## 六、回答问题的表达原则

### 推荐结构

面对技术问题时，采用三段式：

1. 先给结论；
2. 再说明当前项目怎么做；
3. 最后主动说明边界或生产改进方向。

示例：

> 结论是签名正确不等于凭证有效。我们的项目分别检查 Ed25519 签名和本地 VC 状态，所以撤销后签名可以通过，但总体结果失败。生产环境还需要公开可解析的状态机制，而我们的实现是本地教学版。

### 避免使用的表述

- “这个绝对安全”；
- “完全符合所有 W3C 标准”；
- “DID 就是区块链身份”；
- “用了 Ed25519 就不会被攻击”；
- “130 项测试证明没有 Bug”；
- “AI 自动完成了整个项目”；
- “JSON 存储与数据库没有区别”。

### 不确定时的回答方式

> 这个问题涉及我们当前范围之外的生产实现。就现有代码而言，我们实现的是……；如果扩展到生产环境，还需要进一步验证……。我不想在没有证据的情况下给出绝对结论。

这种回答比猜测更专业。

## 七、A 同学最后背熟的九个结论

1. DID 回答“是谁”，DID Document 回答“如何验证”，VC 回答“证明了什么”；
2. Ed25519 证明来源和内容完整性，不负责判断撤销与过期；
3. 七项验证全部通过才返回 `valid=true`；
4. `did:example` 可更新，`did:key` 与公钥绑定、不可变；
5. stableStringify 用于获得确定的签名字节，但不是正式 W3C 规范化实现；
6. JSON 原子写入和更新队列适合单机教学，不等同于数据库事务；
7. 测试数字必须区分当前复核结果和历史执行记录；
8. 项目参考标准核心结构，但 proof、状态服务和密钥管理仍属于教学实现。
9. 选择性披露采用加盐摘要和 Issuer 签名，不是 BBS+ 或正式 SD-JWT VC。
