# DID/VC 本地演示系统开发方案与任务拆解

> 当前扩展实现必须遵循 [产品需求说明书-DID-VC生命周期.md](产品需求说明书-DID-VC生命周期.md)。本文原有“仅使用 did:key、仅 active/revoked”的描述已由双 Method 与完整生命周期方案替代。

## 1. 方案目标

围绕培训营结业证书构建单机 Web 应用，完成 Issuer、Holder 和 Verifier 之间的最小可信闭环。系统强调可解释、可演示、可测试，不追求生产部署或跨机构互操作。

## 2. 总体架构

```text
浏览器操作台
  ├─ DID 身份管理
  ├─ DID Method 能力与生命周期管理
  ├─ VC 签发与台账
  ├─ VC 生命周期管理
  ├─ 日志中心（审计日志与系统日志）
  └─ VC 七项逐项验证
          │ HTTP / JSON
Node.js 本地服务
  ├─ DID Method Registry 与两个 Adapter
  ├─ DID/VC 领域服务和状态机
  ├─ 搜索、稳定排序和分页查询
  ├─ 结构化 LogService 与递归脱敏
  ├─ Ed25519 加密模块
  └─ 业务 JSON 与独立 logs.json 存储
```

系统采用原生 Node.js HTTP 服务和原生 Web 前端，无第三方运行依赖。该选择降低了课程现场对网络、包管理器和数据库服务的依赖。

## 3. 核心技术决策

| 主题 | 决策 | 理由 |
|---|---|---|
| DID 方法 | `did:example` + `did:key` | 前者演示完整生命周期，后者演示无注册表能力边界 |
| Method 架构 | Registry + Example/Key Adapter | 隔离创建、解析和生命周期差异，VC 服务不硬编码 Method |
| 密钥算法 | Ed25519 | 密钥短、签名快，适合数字签名教学 |
| VC 结构 | 参考 VC Data Model 2.0 | 包含 context、type、issuer、subject、有效期和 proof |
| 签名载荷 | 移除 proof 后的稳定键序 JSON | 保证同一 JSON 对象得到确定的签名字节 |
| proof 类型 | 教学专用类型 | 避免宣称兼容未完整实现的正式 cryptosuite |
| VC 状态 | active/suspended/replaced/expired/revoked | 展示签名有效不代表凭证仍可使用 |
| 查询 | 服务端模糊搜索、稳定倒序、分页 | 支持数据增长并保持统一列表行为 |
| 日志架构 | 统一结构化 `LogService` | 集中字段、级别、关联 ID、脱敏、留存和失败兜底规则 |
| 日志存储 | 独立 `data/logs.json`，最多 5,000 条 | 与业务状态隔离并避免无界增长 |
| 数据存储 | JSON 文件 | 可查看、易重置，足以满足单用户本地演示 |
| 权限模型 | 页面工作区区分角色 | MVP 不引入账号与认证复杂度 |

## 4. 数据模型

### DID 身份

- 内部 ID、名称、角色、Method、状态、版本及审计时间
- `did:example` 或 `did:key` 标识符
- DID Document
- 当前公私钥 JWK，私钥仅本地存储
- `did:example` 历史密钥版本和服务地址
- 公开能力声明；公开投影递归移除私钥

### VC 记录

- VC ID 和完整 credential JSON
- 签发时间
- `active`、`suspended`、`replaced`、`expired` 或 `revoked` 状态
- 暂停、恢复、替代和撤销时间
- `replaces`、`replacedBy` 关联

### 验证日志

- 验证时间、凭证 ID、总体结果
- 未通过的检查项

### 结构化日志

- ID、发生时间、关联 ID
- audit/system 类型和 info/warn/error 级别
- 模块、稳定 action、成功标记
- 对象类型、ID、名称
- 错误码、用户可读摘要、脱敏上下文
- 独立文件保存，最多保留最新 5,000 条

## 5. 日志设计

```text
HTTP 请求生成 correlationId
        │
        ├─ 业务服务 ──> audit log（成功或失败）
        └─ API/存储 ──> system log（预期警告或异常）
                         │
                         ▼
                   LogService
             校验 → 递归脱敏 → 限量持久化
                         │
                         ▼
                   data/logs.json
```

- 业务服务只描述 action、目标与结果，不直接操作日志文件。
- API、JSON 解析和存储边界负责系统日志。
- 日志写入失败不改变原业务响应，通过 `console.error` 兜底。
- info/warn/error 在页面分别使用绿色/黄色/红色，并同时显示级别文字。
- 查询支持文本、类型、成功状态、级别、模块和时间范围组合过滤。
- 重置演示数据不清日志；清空后保留 `LOG_CLEAR` 摘要。

## 6. 验证规则

验证按七项独立执行，并以全部通过作为最终有效条件：

1. 格式完整性：必要字段存在。
2. Issuer DID：本地注册表可解析签发方及公钥。
3. DID 当前状态：Issuer 未停用。
4. 密钥版本：proof 指定的当前或历史验证方法可解析。
5. Ed25519 签名：凭证内容与签发时一致。
6. 有效期：当前时间处于 `validFrom` 与 `validUntil` 之间。
7. VC 状态：本地记录存在且处于 active。

## 8. 安全与限制

- 页面和状态 API 会移除 `privateJwk`，避免普通操作暴露私钥。
- 本地数据目录已加入 `.gitignore`，防止演示私钥进入版本库。
- 系统仅监听 `127.0.0.1`，默认不对局域网提供服务。
- 本地 JSON 文件没有加密，不应保存真实个人信息或生产密钥。
- 教学 proof 不等同于正式 W3C Data Integrity 实现。
- 日志不得包含当前或历史私钥、proofValue、完整 VC、完整请求体、token 或认证信息。
