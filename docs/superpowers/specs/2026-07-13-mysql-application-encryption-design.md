# 本地 MySQL 与应用层加密改造设计

## 1. 背景与目标

当前系统使用本地 JSON 文件持久化 DID、VC、验证记录和日志。JSON 文件以明文写入，DID 私钥、VC 个人声明及选择性披露材料可能随文件泄露，不满足敏感数据保密要求。

本次改造将运行时存储迁移到本地 MySQL，并在应用层使用 AES-256-GCM 对敏感字段进行认证加密。系统引入软件模拟 KMS，使业务代码通过密钥编号请求签名，不直接读取或返回私钥。MySQL 配置或主密钥缺失时，应用必须拒绝启动，不得回退到 JSON 明文存储。

本项目仍是教学演示系统。本次改造改善静态数据保密、密钥隔离和审计能力，但不宣称达到真实云 KMS 或 HSM 的硬件安全等级。

## 2. 范围

### 2.1 包含范围

- 使用本地 MySQL 取代 `data/store.json` 和 `data/logs.json` 的运行时持久化。
- 增加数据库结构初始化与版本管理。
- 增加 AES-256-GCM 加密模块和软件模拟 KMS。
- 加密 DID 当前私钥、历史私钥、VC 敏感声明、选择性披露材料和 SD-JWT Disclosure 原始材料。
- 保留 DID、VC、验证、撤销、暂停、恢复、替换、分页、审计和选择性披露等现有行为。
- 增加启动配置校验、密文完整性校验、日志脱敏和安全测试。
- 提供不包含真实秘密的环境变量模板。

### 2.2 不包含范围

- 真实云 KMS、HSM 或操作系统硬件安全模块接入。
- MySQL 安装和管理员级初始化自动化。
- 现有 JSON 业务数据自动迁移。
- 多租户权限模型、用户登录或互联网公开部署。
- 数据库磁盘全盘加密和操作系统加固。

## 3. 方案选择

采用应用层字段加密，而非仅依赖 MySQL 磁盘加密或将完整业务对象加密为单一字段。

应用层字段加密能够在数据库文件、逻辑备份或数据库账号泄露时继续保护敏感字段；同时将 DID、凭证编号、状态和有效期等非敏感索引字段独立保存，维持查询、分页和生命周期管理能力。

## 4. 总体架构

```text
HTTP API / VcService
        |
        +--> MySqlStore
        |      +-- DID、VC、状态、验证记录
        |      +-- 加密后的敏感声明和披露材料
        |
        +--> LocalKms
               +-- 生成 Ed25519 密钥
               +-- 加密私钥后交给 MySqlStore 保存
               +-- 按 keyId 读取并临时解密私钥
               +-- 在模块内部完成签名，只返回签名
```

`VcService` 不再以 `privateJwk` 作为长期业务对象字段。签发 VC、生成选择性披露证明和 SD-JWT 时，服务向 `LocalKms` 提交 `keyId` 与待签名数据。`LocalKms` 完成解密和签名后仅返回签名结果。

## 5. 数据分类与保护

### 5.1 必须加密

- Ed25519 当前私钥和所有历史私钥。
- VC 中的姓名、课程、成绩、完成日期、成就等个人声明。
- 教学版选择性披露的盐值、原始声明和生成材料。
- SD-JWT Disclosure 的盐值及原始材料。
- 后续新增且被标记为个人信息、认证秘密或密钥材料的字段。

### 5.2 可明文保存

- DID、公开 JWK、验证方法及密钥版本。
- VC ID、Issuer DID、Holder DID。
- VC 状态、生效时间、失效时间、替换关系。
- proof、签名值和公开摘要。
- 创建和更新时间。
- 已脱敏的验证记录与审计记录。

日志不得记录完整 VC、私钥、主密钥、数据库密码、Token、选择性披露原始材料或解密后的个人声明。

## 6. 加密设计

### 6.1 算法与数据格式

- 对称加密算法：AES-256-GCM。
- 主密钥长度：32 字节，以 Base64 形式从环境变量载入。
- 每次加密生成新的 12 字节密码学随机 IV，禁止复用 IV。
- 每条密文保存 `ciphertext`、`iv`、`authTag`、`keyId` 和 `encryptionVersion`。
- 二进制字段统一使用 Base64 编码后写入数据库。
- 采用与记录类型及记录 ID 绑定的附加认证数据（AAD），防止密文被复制到其他记录后仍可解密。

### 6.2 主密钥管理

主密钥不得写入 MySQL、源码、日志、测试证据或 Git。应用仅在启动时从 `KMS_MASTER_KEY` 读取。`KMS_MASTER_KEY_ID` 用于标记密文使用的主密钥版本，为后续主密钥轮换保留接口。

本次实现不自动轮换主密钥，但数据结构必须允许同时识别不同 `keyId`。配置中不存在密文所需主密钥时，系统返回受控错误，不得跳过认证或使用空密钥。

### 6.3 解密失败

密文、IV、认证标签或 AAD 被修改时，AES-GCM 认证必须失败。系统记录不含敏感内容的错误代码和目标记录 ID，不得返回底层堆栈、密文或主密钥信息。

## 7. 软件模拟 KMS

`LocalKms` 对外提供明确接口：

- `generateSigningKey()`：生成 Ed25519 密钥，将私钥加密保存并返回 `keyId` 与公钥。
- `sign(keyId, payload)`：在模块内部读取、解密并使用私钥，只返回签名。
- `rotateSigningKey(currentKeyId)`：生成新版本并将旧密钥标记为退役。
- `getPublicKey(keyId)`：返回公开密钥材料。

禁止提供导出解密私钥的公共接口。私钥明文只作为签名调用期间的局部变量存在，不写入磁盘、数据库、日志或 API 响应。

## 8. MySQL 数据模型

数据库至少包含以下表：

- `schema_migrations`：数据库结构版本。
- `dids`：DID 元数据、公开文档、当前密钥编号、版本和状态。
- `kms_keys`：密钥编号、加密私钥、公开 JWK、版本、状态和时间。
- `credentials`：可查询元数据、加密声明、proof、生命周期状态和替换关系。
- `verification_logs`：完整 VC 验证结果的脱敏摘要。
- `disclosure_verification_logs`：选择性披露验证结果的脱敏摘要。
- `audit_logs`：系统和业务操作审计记录。

需要查询、排序或建立关联的字段使用独立列；结构复杂但不敏感的公开数据可保存为 MySQL JSON 列；敏感对象必须先加密，再以文本或二进制列保存。

所有写操作使用事务。DID 更新、密钥轮换、VC 状态变更和替换必须保持现有版本冲突与原子性语义。

## 9. 配置与启动行为

提供 `.env.example`，仅包含空值或非秘密默认值：

```dotenv
DB_HOST=
DB_PORT=3306
DB_NAME=
DB_USER=
DB_PASSWORD=
DB_SSL=false
KMS_MASTER_KEY=
KMS_MASTER_KEY_ID=local-master-v1
```

以下任一情况发生时，应用必须以非零状态退出：

- 必填数据库配置缺失。
- `KMS_MASTER_KEY` 不是合法 Base64 或解码后不是 32 字节。
- MySQL 连接或认证失败。
- 数据库结构未初始化或版本不受支持。

启动错误可以指出缺失配置名、数据库不可达或结构版本错误，但不得输出密码、主密钥或完整连接串。系统禁止自动退回 JSON 存储。

## 10. 数据库权限与本地部署要求

- 应用使用专用 MySQL 账号，不使用 `root`。
- 应用账号只获得目标数据库必要的 `SELECT`、`INSERT`、`UPDATE` 和 `DELETE` 权限。
- MySQL 仅监听本机地址，除非用户明确改变部署架构。
- 数据库备份视为敏感资产，限制访问并验证恢复流程。
- 生产数据和主密钥不得用于自动化测试。

## 11. 兼容性与迁移

原有 `JsonStore` 和 `LogStore` 不再作为正式运行路径。测试可以使用隔离的测试数据库或明确的内存测试替身，但不得导致正式应用回退到 JSON。

现有 `data/store.json` 和 `data/logs.json` 不自动导入。若后续需要迁移，应新增一次性迁移命令：读取旧数据、加密敏感字段、在单个迁移事务中写入 MySQL，并输出不含敏感信息的迁移摘要。

## 12. 测试与验收标准

### 12.1 配置和启动

- 数据库配置为空时应用拒绝启动。
- 主密钥缺失、Base64 无效或长度错误时应用拒绝启动。
- 数据库连接失败或结构版本错误时应用拒绝启动。
- 启动错误不泄露任何秘密。

### 12.2 静态数据保护

- MySQL 中不得出现私钥 JWK 的 `d` 参数明文。
- MySQL 中不得出现测试姓名、课程、成绩及 Disclosure 原文。
- 相同明文加密两次应产生不同 IV 和密文。
- 篡改密文、IV、认证标签或记录绑定信息后必须解密失败。

### 12.3 功能回归

- DID 创建、更新、轮换和停用行为保持一致。
- VC 签发、完整验证、暂停、恢复、替换、过期和撤销行为保持一致。
- 密钥轮换后，旧 VC 仍使用对应历史公钥完成验证。
- 选择性披露和 SD-JWT 正常生成、验证并检测篡改。
- 搜索、稳定排序和分页行为保持一致。

### 12.4 暴露与审计

- 所有公开 API 均不返回私钥、主密钥或 Disclosure 原始材料。
- 日志递归脱敏私钥、密码、Token、完整 VC、proof 和敏感声明。
- KMS 密钥生成、签名、轮换及失败事件产生不含秘密的审计记录。

## 13. 完成定义

只有在应用完全不依赖 JSON 运行时存储、MySQL 配置缺失会阻止启动、规定的敏感字段均以 AES-256-GCM 密文保存、业务层无法导出私钥，并且全部安全测试及现有功能回归测试通过后，本次改造才算完成。
