# DID/VC 生命周期测试设计

## 1. 目标

为“信证台”计划扩展后的 DID 与 VC 能力建立可执行的测试与验收基线，覆盖：

- DID 的创建、使用、信息更新、密钥轮换和停用；
- VC 的签发、验签、暂停、恢复、更新/重新签发、过期和撤销；
- DID、VC 与验签记录列表的模糊搜索、分页和默认倒序；
- 正常流程、异常输入、安全校验、边界条件和非法状态转换。

该设计面向计划能力，不受当前 MVP 仅支持 DID 创建、VC 签发/验签/过期/撤销的限制。后续详细测试用例将同时作为开发验收标准。

## 2. 测试策略

采用分层测试矩阵，并补充完整端到端业务旅程。

### 2.1 服务/API 层

作为主要自动化测试层，覆盖业务规则、状态转换、签名与密钥版本、时间判定、搜索匹配、稳定排序和分页计算。

### 2.2 前端交互层

验证状态对应的按钮、提示和展示内容，以及搜索、翻页、页大小切换、空状态、失败状态和刷新后的状态保持。

### 2.3 端到端层

验证 DID 创建至 VC 最终失效的跨模块主流程，确认页面、API、持久化状态及验签结论一致。

## 3. DID 生命周期模型

DID 的业务阶段为：

`active -> updated -> key_rotated -> deactivated`

- 创建成功后 DID 为可用状态。
- `updated` 表示名称、服务地址等可变信息发生过更新，DID 仍然可用。
- `key_rotated` 表示验证密钥已产生新版本，DID 仍然可用。
- 更新和密钥轮换可以按业务需要发生多次，必须递增版本并保留审计信息。
- `deactivated` 为不可逆终态。停用后禁止再次更新、轮换密钥，以及作为 Issuer 或 Holder 参与新的 VC 签发。
- DID 停用后，历史 VC 仍使用签发时对应的历史公钥版本进行密码学签名验证；整体验证结果必须独立展示 DID 当前状态，不得把“签名有效”等同于“凭证当前可接受”。

## 4. VC 生命周期模型

VC 的状态转换为：

`active <-> suspended -> replaced | expired | revoked`

- VC 签发后，在生效时间范围内且关联 DID 可用时为 `active`。
- `suspended` 表示临时不可接受，签名检查可以通过，但整体校验不通过；允许恢复为 `active`。
- VC 更新采用重新签发：生成新 VC ID，旧 VC 标记为 `replaced`，并保留新旧 VC 的关联关系。
- `expired` 根据有效期自动判定，属于不可恢复终态。
- `revoked` 为主动撤销的不可逆终态。
- 已替代、已过期或已撤销的 VC 不允许暂停、恢复或再次更新。
- 验签至少分别返回格式完整性、Issuer DID 解析、DID 当前状态、密钥版本、密码学签名、有效期和凭证状态检查，不以单一布尔值掩盖具体失败原因。

## 5. 列表规则

DID、VC 和验签记录列表统一遵循以下约定：

- 默认按业务时间降序。DID 使用创建时间，VC 使用签发时间，验签记录使用验签时间。
- 业务时间相同时，以记录 ID 降序作为稳定排序条件。
- 首次进入、刷新、新增记录或完成状态操作后，仍保持最新记录在前。
- 搜索忽略英文字母大小写并去除输入首尾空格。
- 匹配规则为：任一可搜索字段包含完整搜索文本。不实现拼音、分词或多关键词表达式。
- DID 可搜索字段：名称、完整 DID、角色和状态。
- VC 可搜索字段：VC ID、学员姓名、课程名称、Issuer DID、Holder DID 和状态。
- 验签记录可搜索字段：VC ID、验证结果和失败检查项。
- 默认每页 10 条，可切换 10、20、50 条。
- 搜索条件或每页数量改变后返回第 1 页，并按筛选结果重新计算总数与总页数。
- 数据变化导致当前页超界时，返回最后一个有效页；无记录时停留第 1 页。
- 页面展示总记录数、当前页和总页数，并在边界页正确禁用上一页或下一页。
- 原始数据为空与搜索无结果使用不同的空状态提示。
- 搜索、排序和分页由服务端/API 执行，避免大数据量下全量传输。

重点分页边界数据量为 0、1、10、11、20、21、50 和 51 条。

## 6. 测试用例模块

### 6.1 DID 正常功能

- 创建 Issuer 和 Holder DID；
- DID Document 生成、解析和公开字段校验；
- 公钥验签和私钥隔离；
- 更新名称、服务地址等可变信息；
- 密钥轮换，以及新旧密钥版本的正确解析；
- DID 停用及其对后续操作的约束。

### 6.2 DID 异常与边界

- 必填项缺失、非法角色和重复请求；
- 更新、轮换或停用不存在的 DID；
- 停用后再次停用、更新或轮换；
- 使用旧密钥冒充当前密钥签名；
- 并发更新或轮换引起的版本冲突。

### 6.3 VC 正常功能

- 有效 Issuer DID 向有效 Holder DID 签发 VC；
- 未被修改的完整 VC 验签通过；
- VC 暂停、恢复、更新/重新签发；
- 有效期到达后自动过期和主动撤销；
- 历史 VC 使用对应历史密钥验签。

### 6.4 VC 异常与安全

- 篡改 VC 正文、proof、Issuer、Holder 或有效期；
- 使用错误私钥或未知 Issuer 签发；
- 使用已停用 DID 参与签发；
- 尚未生效、已暂停、已替代、已过期或已撤销 VC 的验证；
- 非法状态跳转、重复操作和并发状态操作；
- 必要字段缺失、错误 JSON、超过请求大小限制和特殊字符输入。

### 6.5 列表通用能力

- 默认倒序和相同时间下的稳定排序；
- 各业务字段的模糊搜索；
- 搜索结果与生命周期状态变化联动；
- 分页边界、页大小切换和超界页码修正；
- 空列表、无搜索结果、加载失败和刷新恢复。

### 6.6 端到端业务流程

- DID 创建 -> VC 签发 -> 验签成功；
- DID 更新/密钥轮换 -> 新 VC 签发 -> 新旧 VC 分别验签；
- VC 暂停 -> 验签失败 -> 恢复 -> 验签成功；
- VC 更新 -> 旧证被替代 -> 新证有效；
- VC 过期或撤销 -> 进入不可恢复终态；
- DID 停用 -> 禁止新签发 -> 历史 VC 显示签名有效但 DID 已停用。

## 7. 用例格式与优先级

每条详细用例包含：

- 用例编号；
- 模块；
- 优先级；
- 测试层级；
- 前置条件；
- 测试数据；
- 操作步骤；
- 预期结果；
- 自动化建议。

优先级定义：

- P0：核心签发/验签、私钥保护、不可逆状态约束和关键状态转换；
- P1：更新、轮换、暂停/恢复、重新签发、搜索和分页主路径；
- P2：极端边界、可用性提示和低概率并发冲突。

## 8. 完成标准

- 每个允许的生命周期状态转换至少有一条正向用例。
- 每个禁止的状态转换至少有一条反向用例。
- 每个验签检查项同时具有通过和失败场景。
- 每个列表的全部约定搜索字段均有匹配与无匹配场景。
- 分页边界数据量全部覆盖，默认排序可稳定复现。
- 六条端到端流程全部通过。
- P0 用例全部自动化且全部通过；P1 的服务/API 层用例原则上全部自动化。

## 9. 明确不在本设计范围内的事项

- 拼音搜索、复杂分词、多关键词布尔表达式；
- 区块链共识、真实链上费用和跨链解析；
- 生产级 HSM、KMS 或密钥托管集成；
- 第三方 DID Method 和外部 VC 生态互操作认证；
- 性能压测的具体并发指标。性能基线应在容量目标确定后单独设计。
