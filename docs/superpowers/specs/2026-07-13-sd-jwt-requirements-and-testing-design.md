# SD-JWT 需求与测试扩展设计

## 1. 目标

将 GitHub 新增的 SD-JWT 选择性披露能力纳入现有 DID/VC 产品需求基线，并在 Excel 测试用例、API 测试、服务集成测试和 UI 端到端测试中建立可追溯覆盖，最后形成实际执行结果。

## 2. 范围与定位

- 本期能力定位为 RFC 9901 核心格式的 SD-JWT 教学与演示实现。
- 不宣称完整 SD-JWT VC 生态、Key Binding JWT、Holder Binding 或跨钱包互操作。
- 保留现有教学版加盐摘要选择性披露；页面允许在教学版格式与 SD-JWT 格式之间切换。
- 可披露字段限定为学员姓名、课程、完成日期和成绩/成果。

## 3. 需求文档设计

修改 `docs/产品需求说明书-DID-VC生命周期.md`，完成以下调整：

1. 在产品目标、本期范围与术语中加入 SD-JWT 核心选择性披露。
2. 修正“不在本期范围”中对正式 SD-JWT VC 的表述，明确本期只覆盖 RFC 9901 核心演示能力及其限制。
3. 新增独立的 SD-JWT 功能需求组，使用稳定需求编号，覆盖：
   - 签发时生成 Issuer JWT、披露摘要和内部 disclosure 材料；
   - 普通 API 与页面不得暴露盐值、未披露值或内部材料；
   - 持有者至少选择一个白名单字段生成以 `~` 结尾的 presentation；
   - 验证 Issuer DID、当前或历史密钥、EdDSA JWS、摘要、时间和台账状态；
   - 拒绝空选择、非法字段、重复 disclosure、摘要篡改、格式错误和旧凭证缺少材料；
   - 记录生成与验证审计动作及披露验证日志。
4. 在 UI、API、日志动作和验收标准中补充相应契约。

## 4. Excel 测试用例设计

更新 `outputs/test-cases-20260712/DID-VC系统测试用例表-高覆盖版.xlsx`，沿用现有字段、编号和样式，在合适的功能用例工作表追加 SD-JWT 用例。用例至少覆盖：

- 正常签发并标记 `sdJwtAvailable`；
- 单字段与多字段选择性披露；
- 未披露姓名不出现在 presentation 中；
- 空字段、非白名单字段和旧凭证缺少 SD-JWT 材料；
- presentation 格式、结尾分隔符和重复 disclosure；
- disclosure 内容或摘要篡改；
- Issuer JWT 签名、`alg`、`typ` 和密钥版本；
- Issuer 密钥轮换后使用历史公钥验证；
- Issuer 停用、VC 暂停/撤销/替代/过期；
- 普通状态与列表 API 不泄露 `sdJwtMaterial`、盐值或隐藏声明；
- UI 格式切换、生成、验证、隐私摘要和篡改反馈；
- 审计动作与披露验证日志。

每条用例记录关联需求编号、前置条件、步骤、预期结果、测试层级和自动化映射，便于从 PRD 追踪到执行证据。

## 5. 自动化测试设计

### 5.1 API 层

扩展 `test/api/selective-disclosure.test.js`：验证公开投影不泄露内部材料、生成接口契约、验证接口契约、篡改失败，以及非法输入的 HTTP 错误行为。

### 5.2 服务集成层

扩展 `test/integration/selective-disclosure.test.js`：覆盖单字段/多字段生成、隐藏未披露值、历史密钥验证、重复或篡改 disclosure、凭证状态、Issuer 状态和旧凭证兼容错误。

### 5.3 UI 端到端层

扩展 `test/ui/selective-disclosure.spec.js`：覆盖格式切换、字段选择、生成、隐私摘要、验证结果、逐项检查和模拟篡改。

新增自动化测试遵循测试先行：先写能因缺失校验或行为而失败的用例，确认失败原因，再做必要的最小实现或修正，最后运行相关测试与全量回归。

## 6. 执行与证据

1. 运行 SD-JWT 相关 API 和集成测试。
2. 启动本地服务并运行 Playwright SD-JWT UI 用例。
3. 运行项目全量测试命令，区分代码失败、环境依赖失败和已有非相关失败。
4. 将实际结果回填 Excel 的执行结果相关字段；如现有工作簿没有适合的结果字段，则追加最少必要字段并保持现有样式。
5. 保留命令输出与项目既有测试记录机制产生的证据，不伪造执行结果。

## 7. 错误处理与兼容性

- 新签发凭证包含 SD-JWT 内部材料；历史凭证没有材料时明确提示重新签发。
- 内部 `sdJwtMaterial` 永不进入普通公开投影。
- 验证失败返回逐项检查，而不是将业务失败误报为服务器异常。
- 生成请求的非法选择直接拒绝，避免产生不可验证 presentation。
- 任何文档或测试更新不改变现有教学版选择性披露的既有契约。

## 8. 完成标准

- PRD 中存在可唯一引用的 SD-JWT 需求编号和完整验收规则。
- Excel 中每条新增用例可追踪到需求编号，并覆盖正常、异常、安全、状态和 UI 场景。
- API、集成和 UI 自动化均包含 SD-JWT 用例，且实际执行结果有证据。
- 全量回归结果被如实报告；若存在环境或既有失败，明确列出，不以部分通过宣称全部通过。
