# 签发与验证页面台账设计

## 目标

提升 VC 历史记录与验证结果的可发现性：在“凭证签发”页面底部展示完整的 VC 凭证台账，在“凭证验证”页面底部展示验证记录台账，并为失败记录提供可读的中文原因备注。

## 页面设计

### 凭证签发

签发表单和本次签发的 VC JSON 预览保持不变。在页面底部新增“凭证台账”，展示与运行总览相同的列、搜索、分页和生命周期操作：查看、暂停、恢复、更新和撤销。

总览与签发页面是同一组记录的两个视图。任何一个入口完成生命周期操作后，统一刷新应用数据，使两个视图保持一致。

### 凭证验证

验签输入和逐项报告保持不变。在页面底部新增“验证记录台账”，每行展示：

- 凭证 ID；
- 验证结果；
- 失败原因备注；
- 验证时间。

成功记录的备注显示“全部检查通过”。失败记录根据 `failedChecks` 映射为中文原因；多个原因以中文顿号或逗号分隔。未知检查代码显示稳定的兜底文案，避免空白备注。

## 数据与组件边界

后端继续以 `verificationLogs[].failedChecks` 保存失败检查代码，不重复持久化可本地化的展示文本。前端维护集中式检查代码到中文原因的映射。

VC 台账抽取为可接受容器标识的渲染逻辑，使总览和签发页复用相同的行模板与动作绑定。两个 VC 台账共享现有 VC 搜索条件和分页状态，避免同一数据在不同页面显示不同页码或过滤结果。

验证记录台账复用现有 `/api/verification-logs` 的搜索和分页状态。总览继续显示轻量的“最近验证”摘要；验证页展示完整表格及失败原因。执行一次验证并保存日志后，刷新列表，最新记录按现有倒序规则出现在首行。

## 状态与异常处理

- 无 VC 时，两个 VC 台账显示一致的空状态。
- 无验证记录时，验证记录台账显示明确的空状态。
- 缺少凭证 ID 时显示“未知凭证”。
- `failedChecks` 缺失但结果为失败时显示“未提供具体失败原因”。
- 搜索、翻页或生命周期请求失败时沿用现有 toast 错误提示，不清空当前已展示数据。

## 测试与验收

自动化测试覆盖：

1. 总览与签发页均存在 VC 台账，并展示同一凭证记录。
2. 从签发页台账执行生命周期操作后，状态正确更新。
3. 验证页记录台账展示成功记录及“全部检查通过”。
4. 篡改、撤销或其他失败验签产生失败记录，并显示对应中文原因。
5. 多项检查失败时完整展示多个原因。
6. VC 与验证记录的搜索、分页继续正常工作。

完成标准是相关 UI 测试、完整测试套件和静态差异检查全部通过。
