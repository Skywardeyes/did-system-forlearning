# Agent 可执行的完整测试体系设计

## 目标

基于现有 DID/VC 本地演示项目，补齐可由 Agent 无人值守执行的单元测试、集成测试、UI 测试、功能测试、接口测试和安全测试。测试必须隔离运行数据、提供稳定退出码和可诊断产物，并保持生产运行零第三方依赖。

## 已确认约束

- 保留现有 Node.js 20 与 `node:test` 测试，不迁移到其他单元测试框架。
- 仅新增 `@playwright/test` 开发依赖，用于 Chromium UI 自动化。
- UI 自动化只覆盖 Chromium，不下载 Firefox 或 WebKit。
- 优先复用本地依赖缓存；缺少依赖或浏览器时允许联网下载。
- 测试发现业务或安全缺陷时，不修改业务代码；保留可稳定复现的失败用例并记录缺陷。
- 测试基础设施自身的问题应修正，不能作为已知缺陷保留。
- 测试不得读取、修改或删除项目实际的 `data/store.json` 和 `data/logs.json`。
- 安全测试仅针对测试期间启动的本地临时实例，不扫描公网，不执行破坏主机或工作区数据的操作。

## 技术方案

采用分层混合方案：单元、集成、接口、功能和安全测试继续使用 Node.js 内置 `node:test` 与 `assert`；真实浏览器 UI 测试使用 Playwright Test 驱动 Chromium。共享辅助器统一管理临时目录、随机端口、服务启停、测试数据工厂和清理。

该方案保留已有 57 条测试资产，避免引入 Vitest、Supertest 等非必要依赖。生产依赖仍为空，Playwright 仅写入 `devDependencies`。

## 目录与职责

```text
test/
  unit/          纯函数、加密、查询、状态转换、脱敏和 UI 渲染函数
  integration/   存储、日志及 DID/VC 服务间协作
  api/           真实 HTTP 服务的路由、输入、状态码与响应契约
  functional/    跨层业务旅程与 DID/VC 生命周期
  security/      本地非破坏性安全回归测试
  ui/            Playwright Chromium 用户界面与浏览器旅程
  helpers/       临时环境、服务控制、请求与数据工厂
playwright.config.js
```

现有测试根据其实际职责迁移或纳入对应目录，避免同一行为在同一层级重复验证。迁移后测试名称继续使用清晰的中文场景描述。

## 分层覆盖范围

### 单元测试

- 稳定 JSON 序列化、Ed25519 签名与验签。
- `did:key` 生成、编码与解析边界。
- 搜索规范化、稳定排序、分页和超界修正。
- DID/VC 状态机的合法及非法转换。
- 日志递归脱敏、截断和筛选条件。
- HTML 转义、日志级别渲染、分页和 UI 状态更新。

### 集成测试

- JSON 存储首次初始化、读写、替换、损坏文件行为与留存上限。
- DID 创建、更新、密钥轮换、历史公钥解析和停用。
- VC 签发、验签、暂停、恢复、替换、过期和撤销。
- 日志服务与日志存储组合、关联 ID 与敏感字段脱敏。
- 临时文件上的写入隔离和项目约定支持的并发行为。

### 接口测试

- 全部 API 路由的成功响应及响应结构。
- 缺失字段、非法字段、未知资源、版本冲突和状态冲突。
- 非法 JSON、错误 Content-Type、非法 HTTP 方法和未知路由。
- 搜索、组合筛选、分页、详情和清空确认。
- 状态码、Content-Type、错误对象和公开数据不含私钥。

### 功能测试

- `did:example`、`did:key` 及两种混合 Issuer/Holder 组合。
- DID 创建、VC 签发、完整验证和页面刷新后的持久化。
- 姓名、课程、Issuer、Holder、有效期和 proof 篡改。
- VC 暂停、恢复、替换、过期与撤销。
- DID 密钥轮换后的新旧凭证验证与 DID 停用影响。
- 演示数据重置和审计日志业务闭环。

### UI 测试

- 首页加载、五个导航入口和主要空状态。
- DID 创建、DID Document 查看和能力按钮展示。
- VC 表单签发、JSON 预览、加载最新凭证和完整验证。
- 篡改操作与失败反馈。
- DID/VC 生命周期操作按钮及确认交互。
- DID、VC、验证记录与结构化日志的搜索、筛选和分页。
- 日志详情、清空确认、敏感内容不可见。
- 窄屏布局无横向溢出，导航仍可操作。

### 安全测试

- API、HTML、日志和错误响应均不泄露私钥。
- 用户可控名称、课程、搜索词和日志内容被安全转义，不能形成可执行 XSS。
- 日志不记录私钥、`proofValue`、token、完整 VC 或超大对象。
- URL 路径穿越不能读取工作区或主机任意文件。
- 畸形 JSON、超大请求体和异常查询参数被有限且稳定地处理。
- `__proto__`、`constructor` 等原型污染载荷不能改变进程对象行为。
- 错误响应不暴露堆栈、本地绝对路径或内部实现细节。
- 未支持或危险的 HTTP 方法不能改变状态。
- 检查适用于本地 Web 应用的基础安全响应头；若缺失则形成缺陷记录。

## 执行与隔离

提供以下确定性命令：

```text
npm run test:unit
npm run test:integration
npm run test:api
npm run test:functional
npm run test:security
npm run test:ui
npm run test:all
```

每个 Node 测试环境创建独立临时目录，通过环境变量或服务工厂注入临时 `DATA_FILE`、`LOG_FILE` 和随机可用端口。测试结束时无论成功或失败都关闭服务并清理临时目录。

Playwright 配置自动启动测试服务并等待可用状态。首次执行若缺少 Chromium，安装步骤为 `npx playwright install chromium`。UI 测试失败时保留截图、trace 和 HTML 报告；这些运行产物加入 `.gitignore`。

`test:all` 先运行 Node 分层测试，再运行 Chromium UI 测试。任何失败返回非零退出码，不使用重试掩盖不稳定测试。测试数据使用固定工厂值；时间相关断言使用注入时间或明确的容差边界。

## 缺陷处理

能够在隔离环境稳定复现的业务或安全问题保留为失败测试，并记录到 `docs/测试缺陷报告.md`。每项记录包含：测试命令、用例名称、复现条件、实际结果、期望结果、影响、风险级别和修复建议。

失败分类规则：

- 测试辅助器、选择器、启停逻辑或依赖配置错误：修正测试基础设施。
- 业务行为违反已确认需求或现有文档：保留失败测试并登记功能缺陷。
- 存在数据泄露、注入、越权读取或不安全错误暴露：保留失败测试并登记安全缺陷。
- 当前项目明确声明不支持的能力：不创建错误预期；在测试边界文档中说明。

## 验收标准

- 六类测试均有独立目录、独立命令和实际用例，不以文档清单代替自动化。
- `npm run test:all` 可由 Agent 无交互执行，并能自动启动、等待和关闭本地服务。
- 所有测试使用临时数据路径，不污染项目演示数据。
- Playwright 仅运行 Chromium，并在 UI 失败时生成诊断产物。
- 测试基础设施全部可运行；现有实现缺陷以稳定失败用例和缺陷报告呈现。
- README 和测试文档包含后续 Agent 所需的完整执行说明。
