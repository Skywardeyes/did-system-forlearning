# DID 创建结果前端展示设计

## 目标

修复创建 DID 请求成功后前端因访问失效的 `event.currentTarget` 而报错的问题，并让新创建的身份立即显示在 DID 注册表中。

## 范围

- 创建成功后安全重置 DID 表单。
- 刷新前端状态并在 DID 注册表中显示新身份。
- 展示身份名称、角色、完整 DID、公钥 JWK。
- 保留 DID Document 的详情查看入口。
- 成功提示包含新创建的 DID。
- 私钥继续只保存在服务端本地数据文件，不通过 API 返回，也不在浏览器中展示。

不新增独立结果面板或创建结果弹窗，不修改 DID、密钥或 VC 的后端数据模型。

## 交互与数据流

1. 用户提交 DID 创建表单。
2. 事件处理器在第一个异步操作之前保存表单元素引用。
3. 前端向 `POST /api/dids` 发送名称和角色。
4. 后端创建 Ed25519 密钥对和 `did:key` 身份，将完整身份写入本地存储，并返回经过私钥脱敏的身份对象。
5. 前端用已保存的表单引用重置表单。
6. 前端刷新 `/api/state`，新身份立即进入 DID 注册表。
7. 注册表显示完整 DID 和公钥 JWK，并继续支持查看 DID Document。
8. 页面显示包含新 DID 的成功消息。

## 前端设计

现有 DID 注册表作为唯一的创建结果展示区域，避免增加重复面板。创建处理器将 `event.currentTarget` 先保存为局部变量，后续跨越 `await` 时只使用该稳定引用。

身份卡片显示：

- 名称与 Issuer/Holder 角色；
- 不截断的完整 DID；
- 格式化后的公钥 JWK；
- DID Document 查看按钮。

页面不得读取或渲染 `privateJwk`。

## 错误处理

- API 创建失败时保留表单内容并显示后端错误。
- 创建成功但刷新失败时显示实际刷新错误；后端已经创建的数据不会被回滚。
- 只有创建及刷新流程都完成后才显示成功提示。

## 测试与验收

- 添加前端回归测试，证明异步请求完成后表单仍能被安全重置。
- 验证创建成功后会刷新状态，并使用返回的 DID 显示成功信息。
- 验证身份注册表展示完整 DID、公钥 JWK 和 DID Document 入口。
- 验证 API 状态和创建响应不包含 `privateJwk`。
- 运行现有 `npm test`，确认 DID/VC 业务测试无回归。
- 在浏览器中创建一个 Issuer DID，确认无 `reset` 报错且新身份立即出现。
