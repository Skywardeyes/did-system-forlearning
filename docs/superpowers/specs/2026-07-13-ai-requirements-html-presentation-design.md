# AI 需求生成过程 HTML 演示稿设计

## 目标

将 `docs/AI生成需求过程总结.md` 改编为适合项目答辩的交互式 HTML 演示稿，重点讲清 AI 如何通过对话、项目分析、方案比较、用户确认、需求固化、TDD 实施和真实页面验证，将初始想法转化为可执行需求体系。

## 内容结构

1. 封面：AI 如何把初始想法转化为可实施需求。
2. 初始需求：DID/VC、生命周期、搜索、排序和分页。
3. 需求生成方法：检查现有代码，区分已实现与计划能力。
4. DID 与 VC 生命周期规则的逐项确定。
5. DID Method 冲突发现与 `did:example` / `did:key` 选型。
6. 搜索、稳定排序、分页与空状态的精细化。
7. 三种测试组织方式比较及最终“分层矩阵 + 端到端主流程”选择。
8. 日志需求从“查看记录”演进为完整审计与系统日志规范。
9. 需求到 TDD、专项测试、全量回归、Git 提交和页面验收的转换。
10. UI 错乱案例：通过真实浏览器诊断并反向修订需求与测试。
11. 完整闭环、主要文档产物和方法总结。

## 交互与视觉

- 使用 11 个纵向滚动吸附的全屏章节。
- 支持鼠标滚轮、方向键、Page Up/Page Down、Home/End、侧边导航和全屏切换。
- 页内使用原生 `details/summary` 展开决策依据、规则和真实问题。
- 主色沿用深蓝、青色和绿色；流程决策增加紫色，风险和反馈使用黄色/红色。
- 视觉主线为“用户目标 → 项目检查 → 方案比较 → 用户确认 → 需求基线 → TDD 实现 → 真实验收 → 反馈修订”。
- 桌面使用侧边导航，移动端改为顶部页码与进度。

## 内容原则

- 所有内容以源 Markdown 为主，不虚构对话、数字或产物。
- 区分“AI 生成需求”与“AI 独立决定需求”：关键业务规则由用户逐项确认。
- 强调需求、设计、测试、实现和验收的可追踪关系。
- 将 UI 问题作为“验收反馈修正需求”的真实案例，而不只是 CSS 细节。
- 最后结论强调：产物不是孤立 AI 文本，而是可指导开发、测试和验收的交付体系。

## 验收标准

- 交付单文件、离线可打开的 HTML。
- 精确包含 11 个主章节和对应导航。
- 不依赖远程图片、字体、脚本或样式。
- 至少包含 18 个可键盘操作的折叠证据区。
- 完整覆盖 DID/VC 生命周期、Method 选型、列表规则、测试方案、日志演进、TDD、UI 反馈、文档产物和方法总结。
- 桌面和移动端不产生不可控的整页横向滚动。
- 主要文字在投影尺寸下可读，详细规则通过折叠卡片展开。
