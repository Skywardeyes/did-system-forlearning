# 测试用例 Excel 覆盖设计

## 目标

重建 DID/VC 系统测试用例表，使其既能反映现有自动化验证，又能作为面向用户操作的正式功能验收清单。用例以业务覆盖为中心，不再机械对应测试代码文件。

## 测试类型

“用例类型”仅允许以下六种值：接口测试、单元测试、集成测试、功能测试、兼容性测试、安全测试。

- 接口测试验证 HTTP 边界、参数、状态码和响应数据。
- 单元测试验证独立函数、类或服务规则。
- 集成测试验证服务、存储、日志和 DID Method 等组件协作。
- 功能测试采用用户视角，对独立操作和业务判断逐项建例。
- 兼容性测试只覆盖当前 Chromium 环境中的桌面/移动视口、溢出、导航、表格和表单布局，不扩展至其他浏览器。
- 安全测试验证数据暴露、注入、遍历、协议加固和敏感日志脱敏。

## 功能覆盖

功能用例至少覆盖：运行总览；DID 创建、查看、更新、密钥轮换和停用；VC 签发、查看、搜索和分页；VC 暂停、恢复、替代更新、过期和撤销；VC 验证、篡改检测和七项检查；验证记录台账及失败原因；日志搜索、组合筛选、详情和清空；演示数据重置；空状态、错误提示和不可逆操作确认。

每条功能用例只描述一个可识别的用户目标或业务判断。综合旅程可保留为集成测试，但不得代替独立功能用例。

## 工作簿结构

“测试用例”工作表严格包含七列：用例标题、用例类型、功能模块、前置条件、用例步骤、预期结果、测试结果。所有标题、步骤和结果说明使用中文，DID、VC、API、JSON 等行业缩写可以保留。

“覆盖总览”工作表按测试类型和功能模块统计用例总数、通过数、未执行数和通过率，以便识别覆盖缺口。

## 结果规则

- 已有直接对应的自动化测试且最近一次执行成功：通过。
- 新增的功能覆盖用例、尚无直接自动化对应项或尚未人工执行：未执行。
- 测试结果支持通过、失败、阻塞、未执行四个可编辑值。

不得因为相关底层单元测试通过，就把尚未执行的用户功能用例标记为通过。

## 验收标准

- 测试类型列只出现六个允许值。
- 面向用户的功能点均有独立功能用例。
- 兼容性范围不包含 Firefox、WebKit 或其他浏览器。
- 每条用例七个字段均非空。
- 覆盖总览与测试用例主表数量一致。
- 工作簿无公式错误，筛选、冻结窗格、换行、列宽和结果下拉列表可用。
