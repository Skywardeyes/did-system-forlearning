# AI 生成需求过程总结

## 1. 文档目的

本文总结“信证台 DID/VC Learning Lab”项目中，AI 如何通过持续对话、项目分析、方案比较、用户确认、文档固化、测试实施和真实页面验证，将初始想法逐步转化为可指导开发 Agent 实施的系统需求。

该过程并非由 AI 一次性生成完整需求，而是采用迭代式方法逐步完善，并在实施和验收过程中持续修正需求、设计与测试标准。

## 2. 初始需求

用户首先提出以下目标：

- 覆盖 DID 和 VC 的正常创建、签发与验签；
- 覆盖 DID 和 VC 的生命周期管理；
- 列表类页面支持模糊搜索；
- 数据量较多时支持分页；
- 默认按照时间倒序展示，最新记录位于最前方。

AI 随后读取项目源码、接口、测试和既有文档，确认项目当时已经支持的功能，并区分：

- 当前已实现能力；
- 计划扩展能力；
- 可以继续沿用的实现；
- 需要补充的新状态、新接口和新测试。

## 3. 生命周期需求设计

### 3.1 DID 生命周期

通过逐项讨论和用户确认，DID 生命周期被定义为：

```text
创建 → 正常使用 → 信息更新 → 密钥轮换 → 停用
```

主要规则包括：

- DID 停用为不可逆操作；
- 停用后不得继续更新或轮换密钥；
- 停用后不得参与新的 VC 签发；
- 信息更新和密钥轮换必须增加版本号；
- 历史公钥需要保留，以验证密钥轮换前签发的 VC；
- 当前及历史私钥不得通过普通 API 或页面公开。

### 3.2 VC 生命周期

VC 生命周期被定义为：

```text
active ↔ suspended
active/suspended → replaced
active/suspended → expired
active/suspended → revoked
```

主要规则包括：

- 暂停后的 VC 可以恢复；
- 被替代、过期和撤销后的 VC 不可恢复；
- VC 更新采用重新签发方式，不得原地修改已签名的 VC JSON；
- 新旧 VC 必须保存替代关系；
- 暂停、停用等业务状态不得与密码学签名结果混为一谈；
- 签名有效不代表 VC 当前可以被业务接受。

## 4. 列表、搜索、排序和分页

AI 根据用户要求，将所有列表统一设计为：

- 支持主要业务字段的模糊搜索；
- 自动去除搜索条件首尾空格；
- 英文匹配忽略大小写；
- 默认按业务时间降序排列；
- 业务时间相同时按记录 ID 降序，保证稳定排序；
- 默认每页 10 条；
- 支持切换 10、20、50 条；
- 搜索条件或页大小变化后返回第 1 页；
- 当前页超过总页数时自动返回最后一个有效页；
- 区分原始数据为空和搜索无结果两种状态。

分页边界测试覆盖：

```text
0、1、10、11、20、21、50、51 条
```

## 5. 测试方案选择

AI 提出了三种测试组织方式：

1. 分层测试矩阵；
2. 完整业务场景测试；
3. 按页面功能划分测试。

最终根据用户确认，采用“分层测试矩阵 + 端到端主流程”的方案：

- 服务/API 层：验证业务规则、状态机、签名、搜索、排序和分页；
- 前端交互层：验证按钮状态、表单、筛选、分页、空状态和错误提示；
- 端到端层：验证 DID 创建至 VC 最终失效的完整业务旅程。

每条测试用例统一包含：

- 用例编号；
- 模块；
- 优先级；
- 测试层级；
- 前置条件；
- 测试数据；
- 操作步骤；
- 预期结果；
- 自动化建议或自动化映射。

## 6. DID Method 选型修正

### 6.1 发现问题

在设计 DID 密钥轮换时，AI 发现原项目使用的 `did:key` 无法满足完整生命周期要求。

`did:key` 的 DID 标识符直接由公钥派生，因此：

- 不支持在保持 DID 不变时轮换密钥；
- 不支持更新 DID Document；
- 不支持发布标准停用状态；
- 根据新公钥重新计算后得到的是一个新 DID，而不是原 DID 的新版本。

### 6.2 方案确认

经过方案比较和用户确认，第一阶段同时支持：

| DID Method | 主要用途 | 生命周期能力 |
|---|---|---|
| `did:example` | 教学演示完整生命周期 | 支持更新、轮换和停用 |
| `did:key` | 演示无注册表、自认证身份 | 不支持更新、轮换和停用 |

系统同时增加：

- DID Method 创建选项；
- Method 能力声明；
- Example/Key Adapter；
- 根据 DID 前缀选择解析适配器；
- 不同 Method 作为 Issuer、Holder 的混合签发；
- 未知 Method 和 Method/DID 不一致校验。

## 7. 权威需求文档形成

为了使后续 Agent 可以准确实施需求，AI 将对话结果整理为权威产品需求说明书，详细定义：

- 产品目标和范围；
- 明确不包含的能力；
- 角色与术语；
- DID Method 能力矩阵；
- DID、VC 状态机；
- DID、VC 数据模型；
- 签发和验签规则；
- API 契约；
- 页面行为；
- 搜索、排序和分页；
- 错误码建议；
- 安全与隐私要求；
- 旧数据兼容规则；
- 测试与完成标准；
- 实现 Agent 的禁止事项。

同时修订原有 MVP 文档和开发方案，明确：当旧文档与权威产品需求说明书冲突时，以新需求说明书为准。

## 8. 日志模块需求演进

用户后续提出需要查看各种操作成功和失败记录。AI 进一步澄清日志范围，并确定同时支持：

1. 操作审计日志；
2. 系统运行日志。

日志需求最终包括：

- DID、VC、验签和重置等业务操作日志；
- 非法 JSON、未知接口、请求超限和系统异常日志；
- 成功和失败均记录；
- `info`、`warn`、`error` 三级日志；
- INFO 绿色、WARN 黄色、ERROR 红色，并同时显示文字；
- 使用关联 ID串联同一次请求；
- 支持文本、类型、结果、级别、模块和时间范围组合筛选；
- 使用独立 `data/logs.json`；
- 最多保留最新 5,000 条；
- 重置演示数据时不删除日志；
- 清空日志后保留清理摘要；
- 私钥、proof、完整 VC、请求体和令牌递归脱敏。

日志模块随后被同步到：

- 权威需求文档；
- 开发设计；
- 日志专项设计；
- 测试用例；
- 实施计划；
- README；
- 人工验收说明；
- 交付总结。

## 9. 需求到实现的转换

需求确认后，AI 将需求拆分为可执行计划，并采用测试驱动开发方式：

```text
编写失败测试
→ 确认测试因功能缺失而失败
→ 编写最小实现
→ 运行专项测试
→ 运行全量回归
→ 分阶段提交
→ 真实页面验收
→ 合并主分支
```

实施过程使用独立 Git 工作树隔离修改，完成后由用户决定是否合并到主分支。

## 10. UI 兼容性问题与需求反馈

功能实现完成后，用户发现页面 UI 出现错乱。AI 没有直接尝试随机修改 CSS，而是先进行真实浏览器诊断。

诊断覆盖视口：

```text
1440、1280、1024、768、390px
```

发现的根因包括：

- 隐藏单选框继承全局 `input { width: 100% }`，导致 DID 页面横向溢出；
- 日志筛选栏固定八列最小宽度，导致中等桌面宽度下面板溢出；
- 单元测试能够验证 HTML 和状态逻辑，但不能证明最终浏览器布局正确。

最终修订为：

- 隐藏单选框使用明确的小尺寸，不再撑开页面；
- 日志筛选栏采用桌面四列、平板两列、手机一列；
- 移动端导航为五个入口提供等宽列；
- 增加 UI 布局防回归测试；
- 使用真实浏览器逐页检查页面、视图和面板的横向溢出。

## 12. 完整需求生成流程

```text
用户提出业务目标
        ↓
AI 检查现有代码和文档
        ↓
区分当前能力与计划能力
        ↓
提出多种设计方案及权衡
        ↓
用户逐项确认关键规则
        ↓
AI 固化产品需求、设计和测试标准
        ↓
将需求拆分为可执行实施计划
        ↓
通过 TDD 实现并运行自动化测试
        ↓
通过真实浏览器验证页面行为
        ↓
发现问题后回溯并修订需求或设计
        ↓
同步更新代码、测试、需求和交付文档
```

## 13. 方法总结

本项目中 AI 生成需求的核心特点是：

1. **基于现有项目，而非脱离代码生成通用需求。**
2. **通过用户逐项确认，避免 AI 擅自决定关键业务规则。**
3. **同时维护需求、设计、测试、实现和验收之间的可追踪关系。**
4. **在实施中发现规范冲突时，先修订需求和技术方案。**
5. **使用自动化测试证明业务行为，使用真实浏览器证明页面行为。**
6. **将失败场景、安全要求和禁止事项作为正式需求，而非补充说明。**
7. **持续检查旧文档，防止多个文档产生相互冲突的口径。**

最终形成的不是一份孤立的 AI 文本，而是一套可以直接指导开发 Agent、测试 Agent 和验收人员工作的需求与交付体系。
