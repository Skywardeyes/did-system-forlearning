# AI 生成需求过程总结

## 1. 项目背景与核心结论

“信证台 DID/VC Learning Lab”是一个面向教学与本地演示的去中心身份和可验证凭证系统。项目从 DID 创建、VC 签发与验证出发，逐步扩展到 DID/VC 生命周期、搜索分页、结构化日志、安全测试、选择性披露和永久测试证据链。

本项目中的“AI 生成需求”，并不是让 AI 一次性输出一份完整文档，更不是由 AI 代替用户决定关键业务规则。真实过程是：AI 读取现有代码和文档，识别缺口与冲突，提出多个方案及其权衡；用户逐项确认关键决策；随后再将结论固化为需求、设计、测试和实施计划，并在真实实现与验收中继续修正。

最终形成的不是一份孤立的 AI 文本，而是一套可以指导开发 Agent、测试 Agent 和验收人员协同工作的需求与交付体系。

## 2. 从初始想法到需求边界

项目初始目标主要包括：

- 覆盖 DID 和 VC 的正常创建、签发与验签；
- 补充 DID 和 VC 的生命周期管理；
- 列表页面支持模糊搜索和分页；
- 默认按时间倒序展示，让最新记录优先出现。

面对这些目标，AI 没有直接生成一套通用 DID/VC 需求，而是先检查了项目源码、HTTP 接口、测试和既有文档，将现状拆分为四类：

1. 已经实现且可以保留的能力；
2. 已有雏形，但与新需求存在差距的能力；
3. 需要新增的状态、接口、页面与安全约束；
4. 必须通过用户决策才能确定的业务规则。

这一步非常关键：需求不是从空白页面开始，而是建立在可验证的项目现状之上。

## 3. 生命周期需求的逐项确定

### 3.1 DID 生命周期

经过方案讨论和用户逐项确认，DID 生命周期被定义为：

```text
创建 → 正常使用 → 信息更新 → 密钥轮换 → 停用
```

对应的工程规则包括：

- DID 停用为不可逆操作；
- 停用后不得继续更新、轮换密钥或参与新的 VC 签发；
- 信息更新和密钥轮换必须增加版本号，并使用 `expectedVersion` 防止并发覆盖；
- 旧公钥、密钥版本和历史 `verificationMethod` 需要保留，以验证轮换前签发的 VC；
- 当前及历史私钥不得通过普通 API、DID Document 或页面公开。

这些规则后续被映射为状态检查、版本冲突错误、历史密钥解析和敏感数据脱敏测试。

### 3.2 VC 生命周期

VC 状态机被确定为：

```text
active ↔ suspended
active/suspended → replaced
active/suspended → expired
active/suspended → revoked
```

其中，暂停是可恢复状态；替代、过期和撤销是终止状态，不能恢复为有效。VC 更新不允许原地修改已签名的 JSON，而是重新签发一张具有新 ID 的凭证，并使用 `replaces` 与 `replacedBy` 保存新旧关系。

该过程还确立了一个重要的答辩口径：**密码学签名有效，不等于凭证当前可以被业务接受。** 被暂停、撤销、替代或过期的 VC，其历史签名仍可能正确，但完整验证结果必须失败。

## 4. 第一个真实纠偏：DID Method 选型冲突

在设计 DID 密钥轮换时，AI 通过对照 Method 特性发现，原项目中的 `did:key` 无法同时满足“DID 保持不变”和“更换公钥”。原因是 `did:key` 标识符直接由公钥派生：公钥变化后，得到的是新 DID，而不是原 DID 的新版本。

这不是一个可以用局部代码技巧解决的问题，而是需求与标识方法语义冲突。经过方案比较和用户确认，系统最终同时支持两种 Method：

| DID Method | 教学用途 | 生命周期能力 |
|---|---|---|
| `did:example` | 演示可管理的完整生命周期 | 支持更新、密钥轮换和停用 |
| `did:key` | 演示无注册表、自认证身份 | 不支持原 DID 的更新、轮换和停用 |

工程上增加了 Method Registry、`ExampleDidAdapter`、`KeyDidAdapter` 和 Method 能力声明。VC 服务根据 DID 前缀选择解析适配器，未知 Method 必须失败，不得静默回退到默认实现。

同时，测试覆盖了四种 Issuer/Holder 组合：`example/example`、`key/key`、`example/key` 和 `key/example`。这证明两种 Method 不仅能独立工作，也能在同一签发与验证流程中组合使用。

## 5. 将搜索和分页转化为可测试契约

“列表支持模糊搜索和分页”如果只保留为一句功能描述，开发和验收仍会产生歧义。因此，项目将它细化为一组统一查询规则：

- 对主要业务字段进行模糊搜索；
- 自动去除搜索条件首尾空格，英文匹配忽略大小写；
- 默认按业务时间降序排列，时间相同时再按记录 ID 降序，保证稳定排序；
- 默认每页 10 条，支持 10、20 和 50 条切换；
- 搜索条件或页大小改变后回到第 1 页；
- 当前页超过总页数时，自动回到最后一个有效页；
- 区分“原始数据为空”与“搜索无结果”两种页面状态。

分页边界使用 `0、1、10、11、20、21、50、51` 条数据进行测试，分别覆盖空集、单页上限、刚好跨页和页大小上限。DID 列表、VC 台账、验证记录和日志中心都复用这套契约。

## 6. 测试方案从“功能验证”演进为“分层证据”

针对如何组织测试，AI 协助比较了三种方案：

1. 按技术层组织的分层测试矩阵；
2. 按完整用户旅程组织的业务场景测试；
3. 按页面区域组织的功能验收测试。

经过用户确认，项目采用“分层测试矩阵 + 端到端主流程”：

- 单元层验证稳定序列化、Base58、Ed25519、查询分页和日志脱敏等独立规则；
- 集成层验证 DID Method、VC 服务、JSON 存储和审计日志的协作；
- API 层验证路由、请求参数、状态码、错误码和敏感数据边界；
- 功能与 UI 层验证用户真正能否完成创建、签发、验证、状态操作、搜索和查看日志；
- 端到端层将 DID 创建、VC 签发、密钥轮换、暂停恢复与最终失效连成完整旅程。

每条验收用例统一记录编号、模块、优先级、测试层级、前置条件、测试数据、操作步骤、预期结果与自动化映射，使需求能够追踪到具体断言。

## 7. 日志需求从一句话演进为完整模块

用户最初只提出“需要查看各种操作成功和失败记录”。AI 协助澄清后，将日志范围区分为：

1. 记录 DID、VC、验证和重置等业务操作的审计日志；
2. 记录非法 JSON、未知路由、请求超限和内部异常的系统运行日志。

最终需求被细化为：

- 成功和失败都要记录，并区分 `info`、`warn` 和 `error` 三级；
- 每次 HTTP 请求生成 `correlationId`，使系统日志与业务审计日志可以串联；
- 支持按文本、类型、结果、级别、模块和时间范围组合查询；
- 日志独立保存在 `data/logs.json`，最多留存最新 5,000 条；
- 重置演示数据不删除日志，确认清空日志后保留 `LOG_CLEAR` 摘要；
- 私钥、令牌、`Authorization`、proof、完整 VC 和请求体必须递归脱敏，统一显示为 `[REDACTED]`。

日志需求不仅进入代码，还同步到权威需求、日志专项设计、实施计划、测试用例、README、人工验收和交付总结，体现了“一个需求必须在所有产物中保持一致”的原则。

## 8. 从需求到可验证成果

需求是否有价值，最终要看它能否落地为可执行、可解释和可复核的系统行为。本项目将几个重要需求转化为了具体验证结果。

### 8.1 七项完整验证

项目没有将 VC 验证简化为一次数字签名检查，而是分为：

1. **格式完整性**：凭证必要字段和 proof 是否存在；
2. **Issuer DID 解析**：是否能够找到签发方 DID Document；
3. **DID 当前状态**：Issuer 是否存在且未停用；
4. **密钥版本**：当前或历史公钥是否与 `verificationMethod` 匹配；
5. **Ed25519 签名**：凭证是否由对应私钥签发且内容未被修改；
6. **有效期**：当前时间是否位于 `validFrom` 和 `validUntil` 之间；
7. **凭证当前状态**：本地签发台账是否存在，且状态是否为 `active`。

只有七项全部通过，系统才返回 `valid=true`。每项检查都保留独立失败原因，用户可以区分“凭证被篡改”、“DID 已停用”、“密钥版本不匹配”、“凭证已过期”和“凭证已撤销”等不同情况。

### 8.2 教学版选择性披露

项目还将“凭证验证时不应披露无关字段”转化为了教学版选择性披露功能。签发时，系统为可披露声明生成随机盐，计算 SHA-256 声明摘要，并由 Issuer 使用 Ed25519 对摘要清单签名。持有者可以只披露课程、完成日期等指定字段；验证方通过盐值与摘要检查披露内容是否被篡改。

这项实现用于解释选择性披露的核心思想，并不等同于正式 SD-JWT VC、BBS+ 或零知识证明协议。

### 8.3 安全与可信证据

安全需求被转化为了可执行攻击测试，包括私钥和错误堆栈泄露、日志递归脱敏、存储型 XSS、原型污染、目录穿越、非 JSON 请求、1 MB 请求体上限、不支持方法和安全响应头。

测试结果也不仅停留在终端中。全量执行会保留 Node 和 UI 源日志、测试环境、Git 提交、结果摘要、Playwright 报告与失败附件，并生成 `evidence-manifest.json` 和 `checksums.sha256`。后者可以发现证据文件被修改、删除或额外添加，使“测试通过”能够被事后复核。

## 9. 从需求基线到 TDD 实施

为了让后续 Agent 能够准确执行，已确认的对话结论被整理为权威产品需求说明书，统一定义产品范围、角色与术语、Method 能力矩阵、DID/VC 状态机、数据模型、签发和验证规则、API 契约、页面行为、搜索分页、错误码、安全隐私、旧数据兼容和完成标准。

当旧文档与新的权威需求冲突时，以新需求为准，并同步修订 MVP 说明和开发方案，避免多份文档同时指导实施却给出不同口径。

实施阶段采用测试驱动开发：

```text
编写失败测试
→ 确认测试因功能缺失而失败
→ 编写最小实现
→ 运行专项测试
→ 运行全量回归
→ 分阶段提交
→ 真实页面验收
→ 由用户决定合并方式
```

对需要隔离的功能开发，使用独立 Git 工作树避免污染主工作区。实施完成后，由用户决定本地合并、创建 PR、保留分支或放弃修改。

## 10. 两个真实纠偏案例

### 10.1 并发写入丢失

集成测试曾发现，两个并发 DID 创建请求完成后，最终文件中只保留了一条记录。进一步追踪发现，原实现只串行化了最终写文件动作，但两个请求可能同时读取旧状态，各自修改后再互相覆盖。

该问题最终通过将整个“读取—修改—写入”事务加入同一队列解决，并增加并发回归测试。这个案例说明：测试不仅用于证明实现符合需求，也会暴露原需求中未明确表达的数据完整性边界。

### 10.2 UI 逻辑通过但真实页面错乱

功能实现后，用户发现页面存在横向溢出和排版错乱。AI 没有随机试改 CSS，而是在 `1440`、`1280`、`1024`、`768` 和 `390px` 视口下进行真实浏览器诊断。

根因包括：

- 隐藏单选框继承了全局 `input { width: 100% }`，虽然肉眼不可见，仍会撑开 DID 页面；
- 日志筛选栏固定为八列最小宽度，在中等桌面宽度下无法收缩；
- 单元测试可以证明 HTML 结构和状态逻辑，但不能证明最终浏览器布局正确。

修正后，隐藏单选框使用明确小尺寸，日志筛选栏分别使用桌面四列、平板两列和手机一列，移动端导航保持等宽且可操作。项目同时增加布局防回归测试，并将“真实浏览器逐页验收”纳入正式完成标准。

这个案例说明：**自动化逻辑测试证明系统如何计算，真实浏览器验收证明用户实际看到了什么。**

## 11. 需求到交付的追踪链

为防止需求、实现和验收口径逐步分离，项目建立了从需求基线到交付证据的文档链：

| 阶段 | 主要产物 | 作用 |
|---|---|---|
| 需求基线 | `产品需求说明书-DID-VC生命周期.md` | 统一产品范围、状态机、数据模型、页面行为和完成标准 |
| 方法选型 | `DID方法选型说明.md` | 解释 `did:example` 与 `did:key` 的能力差异和选型边界 |
| 技术设计 | `开发方案与任务拆解.md` 与专项设计 | 将业务需求转换为模块边界、数据流和异常处理 |
| 实施计划 | `docs/superpowers/plans/` | 将设计拆成可执行、可验证的 TDD 步骤 |
| 测试与验收 | `测试用例-DID-VC生命周期.md`、`测试与人工验收.md` | 将需求映射为自动化断言和人工检查步骤 |
| 运行与使用 | `README.md` | 说明实际启动方式、演示流程与实现边界 |
| 交付与复核 | `交付总结.md`、`test-records/` | 记录已完成能力、限制、测试日志、报告和校验和 |

当某项需求发生变化时，不能只修改代码，还需检查需求、设计、计划、测试、README 和交付材料是否需要同步，从而保持各阶段可追踪。

## 12. 完整需求生成闭环

```text
用户提出业务目标
        ↓
AI 检查现有代码、接口、测试和文档
        ↓
区分当前能力、可复用能力与待扩展能力
        ↓
提出多种方案并说明权衡
        ↓
用户逐项确认关键业务规则
        ↓
AI 协助固化需求、设计、测试与完成标准
        ↓
将需求拆分为可执行的 TDD 实施计划
        ↓
运行专项测试与全量回归
        ↓
通过真实浏览器和人工步骤验证用户行为
        ↓
发现问题后回溯并修订需求、设计或测试
        ↓
同步更新代码、测试、需求和交付文档
```

这一闭环使需求工程不再是开发前的一次性阶段，而是贯穿分析、决策、实施、测试和验收的持续过程。

## 13. 实现边界

本项目的成果应当在正确的范围内解读。系统参考 W3C DID Core 和 VC Data Model 2.0 的核心数据结构，并实现了 Ed25519 签名、完整生命周期、教学版选择性披露和本地测试证据链，但其定位仍是**本地教学演示系统**。

项目中的 `EducationalEd25519Signature2026` 和 `EducationalSelectiveDisclosureProof2026` 是为了明确表达教学实现，不等同于正式注册的 Data Integrity cryptosuite、SD-JWT VC、BBS+ 或零知识证明。系统也没有提供生产级 KMS/HSM 密钥托管、真实身份核验、链上注册、标准状态列表、正式互操作认证或公网压力验证。

明确这些边界并不会削弱项目价值，反而表明需求、实现和答辩口径保持了一致。

## 14. 答辩总结

本项目中 AI 协助需求工程的核心价值，可以概括为以下七点：

1. **需求基于现有项目证据，而不是脱离代码的通用想象。**
2. **AI 负责分析、比较和追踪，关键业务规则由用户逐项确认。**
3. **当 `did:key` 与密钥轮换需求冲突时，先修正产品与技术方案，而不是在代码中伪造能力。**
4. **需求、设计、实施计划、测试、验收和交付材料保持可追踪关系。**
5. **自动化测试证明业务与安全规则，真实浏览器验收证明页面行为。**
6. **失败场景、隐私边界、并发完整性和禁止事项被视为正式需求，而不是事后补充。**
7. **测试结果通过日志、环境、Git 版本、报告和校验和形成可复核证据。**

因此，这次实践不是“让 AI 替人写需求”，而是将 AI 作为一个能够持续读取项目、暴露冲突、组织方案、维护追踪关系并辅助验证的工程协作者。在用户承担决策与最终验收责任的前提下，AI 将需求分析、实施和证据链连接成了一个可执行、可验证、可回溯的完整闭环。
