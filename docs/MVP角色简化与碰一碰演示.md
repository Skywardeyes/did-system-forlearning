# MVP 角色简化与“碰一碰”演示

## 一、角色与产品边界

| 角色 | 唯一入口 | 主要操作 | 私钥位置 |
|---|---|---|---|
| Holder | 信证钱包 `http://127.0.0.1:5176` | 创建 DID、领取 VC、选择字段、碰一碰出示 | 浏览器 IndexedDB 中的不可导出 CryptoKey |
| Issuer | 信证台 `http://127.0.0.1:5173` | 创建 Issuer DID、模板、签发和管理 VC | 组织侧 KMS |
| Verifier | 信证台 `http://127.0.0.1:5173` | 接收碰一碰证明、执行验证、查看台账 | 不需要 Holder 私钥 |

信证台注册的是组织账号，一个账号只绑定一个组织。系统不再创建个人空间，不提供空间切换、组织成员邀请或跨组织兼职。数据库中的 `tenant_id` 仍保留，用于隔离不同组织的数据，而不是让一个账号切换多个身份。

## 二、启动

```powershell
npm run migrate
npm start
npm run frontend:dev
npm run wallet:dev
```

分别打开信证台和信证钱包。首次使用在信证台注册组织账号；Holder 在信证钱包注册独立的钱包账号。两类账号不共享角色和登录入口。

钱包注册时可以选择“本地自托管”或“平台代托管（MVP 模拟）”。当前自托管私钥仍保存在浏览器 IndexedDB；代托管选项现阶段只记录账号偏好并用于演示，尚未实现生产级 HSM、恢复签名协议和密钥风控，因此不能把它描述为已经完成的安全托管产品。

## 三、最短完整演示

1. Holder 在信证钱包创建本地 DID，点击“发布公开 Holder DID”。钱包对登记包签名，平台仅保存 DID Document 和公钥。
2. Holder 从下拉框选择一个已注册且审核通过的组织，发送 Holder 关联申请；Issuer 在信证台“DID 身份”中接受申请，系统自动关联该 Holder DID。
3. Issuer 创建并发布凭证模板，选择 Holder 后签发 VC。
4. Holder 钱包收到提醒，在收件箱点击“领取并导入”；VC 保存到本地钱包。
5. Holder 进入“选择性披露”，搜索并添加一张或多张 VC，只勾选本次需要的字段。
6. 点击“本地生成组合披露证明”。后台自动创建五分钟有效的一次性 Challenge，钱包用 Holder 私钥绑定整组披露内容。
7. 点击醒目的“碰一下，就验证！”，或者点击旁边的“出示二维码”。两种方式投递的是同一份一次性证明；二维码只携带随机投递标识和过期时间。
8. Verifier 打开信证台“钱包证明验证”，页面自动出现 Holder、凭证数和到达时间。
9. Verifier 点击“验证”，右侧立即显示逐项和逐凭证结果，底部生成组合验证台账。

## 四、讲解口径

“碰一碰”简化的是交互，不是密码学检查。界面不再要求观众理解和复制 Challenge，但后台仍执行：

1. 生成高熵一次性随机数；
2. Holder 私钥签署组合证明、Challenge 和验证方标识；
3. 验证时检查 Issuer 签名、模板、有效期、动态状态、Holder DID、所有 VC 主体一致性和 Holder 组合签名；
4. 首次通过时原子消费 Challenge，旧证明不能再次验证；
5. 待验证证明在 MySQL 中使用 AES-256-GCM 信封加密保存。

当前实现是单机 MVP 的 NFC 模拟通道，不声称已经接入实体 NFC 标签。生产版本应让 NFC/二维码携带验证站点标识，把投递定向到具体 Verifier，并增加设备认证、近场会话和超时确认。

## 五、演示前检查

- MySQL schema 为 V12；
- 信证台组织账号已经注册；
- Holder DID 已从钱包发布并被 Issuer 关联；
- 至少一张有效 VC 已领取到钱包；
- 验证页处于“等待 Holder 碰一碰”状态；
- 不在界面中复制 Challenge、域名或证明 JSON。
