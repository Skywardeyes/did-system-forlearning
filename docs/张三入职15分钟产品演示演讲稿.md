# 张三入职场景：15 分钟产品演示演讲稿

> 适用范围：同事完成开场、DID/VC 通俗介绍和“张三入职”流程图讲解后，由演示人接续完成产品介绍、现场操作和价值总结。
>
> 使用方式：`【操作】`是演示动作，不需要念；“口播”部分可以直接照读，也可以按现场节奏删减。

## 一、接过流程图，进入真实产品

【操作】打开“张三入职场景”下滑式演示页第一页。

各位老师，刚才我的同事通过流程图介绍了 DID 和 VC，也说明了张三入职时面临的问题。接下来我不再只讲概念，而是把刚才这张流程图放进我们实际开发的系统里，完整走一遍。

这次演示包含四个参与方：上海大学、华山医院、张三，以及作为招聘企业的格尔软件。

上海大学负责证明张三的学历，华山医院负责证明张三的体检结果，张三本人负责持有和出示这些凭证，格尔软件负责在入职时完成验证。

最终要实现的不是把毕业证照片和体检报告扫描件再次上传，而是形成一条完整的数字信任闭环：**组织分别签发，个人统一持有，个人按需组合，企业独立验证。**

---

## 二、先介绍两个产品和三个角色

【操作】下滑到“一个组织工作台，一个个人钱包”。

我们把系统明确拆成了两个产品。

第一个产品叫“信证台”，面向组织。上海大学和华山医院在这里扮演 Issuer，也就是签发方；格尔软件在这里扮演 Verifier，也就是验证方。

第二个产品叫“信证钱包”，面向个人。张三是 Holder，也就是凭证持有人。他只在钱包里管理自己的 DID、领取凭证和生成证明，不进入组织工作台。

这三个角色不能混为一谈。签发方只对自己掌握的事实负责，持有人决定向谁出示什么，验证方根据自己的业务规则判断是否接受。

【操作】下滑到“私钥各归其主”。

这里还有一个最重要的边界：**私钥各归其主，公开资料用于验证。**

上海大学和华山医院使用各自的机构私钥签发凭证；张三的 Holder 私钥在自己的钱包本地生成，并以不可导出的 `CryptoKey` 保存在浏览器 IndexedDB 中，不上传给信证台。张三发给组织的是公开 DID Document 和控制权签名，不是私钥。

DID 主要回答“这是谁、谁控制对应密钥”，VC 主要回答“哪个机构对什么事实作出了签名证明”。

## 三、说明为什么当前用数据库模拟注册表

【操作】下滑到“为什么当前没有把区块链作为主流程”。

在正式演示前，我先说明一个范围边界：这次我们没有把区块链放进现场主流程。

受结业任务周期、团队经验、设备条件和多节点部署环境限制，当前 MVP 使用 MySQL 实现 DID 注册表、凭证状态、定向消息和验证台账。我们的目的，是先证明身份建立、凭证签发、个人持有、选择性披露和验证这条业务链确实能运行。

因此，我们不是用数据库冒充区块链。当前数据库是一个可替换的注册与状态实现。未来如果进入多机构共同治理场景，可以把公开 DID、公钥版本和停用状态迁移到联盟链或多方可信注册表，而钱包、签发和验证的核心业务逻辑仍然可以保留。

## 四、进入现场演示：展示已经准备好的参与方

【操作】切换到信证台登录页。现场可快速展示注册入口，但不重复注册全部账号。

现在进入现场操作。

为了让演示集中在核心业务，而不是把时间花在重复录入账号和模板上，我们已经提前注册好了上海大学、华山医院、格尔软件和张三，并提前创建了本科毕业证明、A 类体检报告和 B 类体检报告模板。

这里的组织并不是写死在代码里的。注册时可以选择学校、医院、企业等组织类型；模板字段也是由组织动态定义的。也就是说，学校可以定义学历字段，医院可以定义体检字段，其他机构也可以创建完全不同的业务凭证，不需要开发人员为每种证书重新修改系统。

【操作】登录上海大学，进入“DID 身份”，展示上海大学的 Issuer DID；再进入“凭证签发”，展示“本科毕业证明”模板。

这是上海大学的 Issuer DID。系统对外公开 DID Document 和公钥，机构私钥则由服务端 KMS 边界管理。数据库中的 Issuer 私钥 JWK 使用 AES-256-GCM 加密保存，只有执行授权签名时才在 KMS 内部解密，不会通过 API 返回给操作员。

这是上海大学创建的“本科毕业证明”模板，其中包含学生姓名、学历层次、专业、毕业状态、毕业日期等字段。

【操作】切换到华山医院，展示 Issuer DID 和 A、B 类体检报告模板。

华山医院也有自己独立的 Issuer DID，并维护 A 类和 B 类两套体检模板。两套模板可以对应不同检查范围和服务内容。这里体现的是动态模板能力，而不是把所有行业都套进固定的三个字段。

## 五、张三在个人钱包掌握自己的身份

【操作】打开信证钱包，登录张三账号，进入“我的身份”，展示“张三·求职身份”。

现在切换到张三的信证钱包。

这个 Holder DID 是张三在自己的钱包中创建的。创建时，浏览器使用 Web Crypto 生成 Ed25519 密钥对：公钥进入 DID Document，私钥保存为不可导出的本地 `CryptoKey`。

这里要特别强调，`did:key` 字符串和 DID Document 都不是秘密，它们本来就用于公开解析和验签。真正必须保密的是私钥。**公钥负责让别人验证，私钥负责证明“确实是我”。**

【操作】在“向组织发送 DID”区域，展示张三已向上海大学、华山医院发送的申请记录；如环境已重置，则现场选择 DID、组织并发送一次。

张三选择自己的求职 DID，再选择上海大学或华山医院并填写申请说明，就可以把公开身份资料定向发送给相应组织。

发送时，钱包会用 Holder 私钥对登记内容做 Ed25519 签名。组织收到后，从公开 DID Document 中取得 Holder 公钥并验签，因此组织不仅收到了一个 DID 字符串，还确认了申请人确实控制这个 DID 对应的私钥。

---

## 六、上海大学和华山医院分别签发 VC

【操作】切换到上海大学信证台，进入“凭证签发”；选择上海大学 Issuer、张三 Holder 和“本科毕业证明”模板。若凭证已提前签发，可展示签发日志；若现场签发，则按准备数据填写后点击“使用 Issuer KMS 签发”。

上海大学接受张三的 Holder 申请后，为他签发一张本科毕业证明。

这张凭证可以包含“张三、本科、计算机科学与技术、已毕业、毕业日期”等内容。点击签发时，KMS 在内部使用上海大学的 Issuer 私钥生成 Ed25519 签名，同时生成支持选择性披露的 SD-JWT 材料。

签发后的 VC 正文、披露盐值和相关材料在服务端使用 AES-256-GCM 加密落库。日志中仍然保留高可读性的签发时间、签发组织、持有人和模板名称，方便组织审计，但操作员不需要靠阅读整份原始 JSON 来管理凭证。

【操作】切换到华山医院，展示或现场签发“A 类体检报告”。

华山医院用同样的方式为张三签发 A 类体检报告，但这张凭证由华山医院自己的 DID 和私钥签名。也就是说，学历的真实性由学校负责，体检的真实性由医院负责，平台不会伪装成所有事实的统一签发者。

---

## 七、张三领取凭证：Challenge 在后台完成

【操作】回到信证钱包“凭证收件箱”，展示两条待领取消息，依次点击“领取并导入”；随后进入“本地凭证库”，展示两行凭证。

张三的钱包收到两家机构发来的凭证通知。收件箱不会只显示难以理解的 UUID，而会显示签发组织、模板名称和签发日期。

现在我点击“领取并导入”。大家在界面上不会看到复杂的 Challenge 操作，这是正常的，因为它由钱包自动完成。

实际过程是：后端先下发一次性 Challenge；钱包使用张三本地私钥签名；后端根据 Holder DID Document 中的公钥验签，并原子消费这个 Challenge；验证通过后，服务端才授权解密交付材料，凭证随后进入张三本地钱包。这样可以防止其他人只凭一个凭证编号冒领，也能防止同一个领取请求被重复使用。

现在本地凭证库中已经出现两张凭证：**上海大学·本科毕业证明**和**华山医院·A 类体检报告**。这两张凭证来自不同组织，但都由张三自己的 Holder DID 持有。

## 八、从两张凭证中只挑必要字段

【操作】进入“选择性披露”，在验证组织下拉框中搜索并选择“格尔软件股份有限公司”。

接下来是整个演示最关键的部分。

张三入职时，格尔软件真正需要知道的不是完整学号、证书编号、体检报告编号、血常规和影像明细，而是：他是否已经本科毕业，专业是什么，以及当前是否适合入职。

因此，张三先明确选择接收方为格尔软件。系统生成的 Challenge 和 Domain 会绑定本次验证目标，避免这份证明被复制给另一个组织冒用。

【操作】搜索并添加“上海大学·本科毕业证明”，勾选：学生姓名、学历层次、专业、毕业状态、毕业日期；明确不勾选学号和证书编号。

从学历凭证中，我只选择学生姓名、学历层次、专业、毕业状态和毕业日期，不选择学号和证书编号。

【操作】搜索并添加“华山医院·A 类体检报告”，勾选：体检类别、综合结论、入职适岗结论、报告有效期；不勾选报告编号、检查明细和服务价格。

从体检凭证中，我只选择 A 类、综合结论、入职适岗结论和有效期，不披露报告编号、血常规、胸部影像、传染病筛查明细和服务价格。

两张凭证由同一个 Holder DID 持有，而且整个组合证明还会由张三本地私钥再次签名，所以体检报告中不需要重复披露姓名，也可以证明两部分材料属于同一个持有人。

【操作】点击“本地生成组合披露证明”。

现在点击生成。系统在钱包本地把两张 VC 的选定字段组合成一份 Presentation。

我们的选择性披露路线使用随机盐和 SHA-256 摘要：签发时，每个字段都形成一个带随机盐的披露项，摘要进入签名载荷；出示时只交出被选择字段的“盐值、字段名和字段值”。验证方重新计算摘要，并与 Issuer 已签名的摘要比对。没有选择的字段原值不会进入这份 Presentation。

最后，钱包把两张 SD-JWT、Holder DID、Challenge 和 Domain 绑定在一起，再用张三的 Ed25519 私钥对整个组合证明签名。

## 九、模拟“碰一下”或二维码投递

【操作】先在另一个窗口登录格尔软件信证台，进入“钱包证明验证”，让页面处于等待状态；再回到钱包。

在真实移动端场景中，这一步可以通过 NFC 碰一碰或者扫描验证方二维码完成。当前项目是网页 MVP，没有真实手机 NFC 环境，因此我们用一个明确选择了接收组织的按钮来模拟传输对象和触发动作。

【操作】点击“碰一下，就验证！”。如需展示二维码，则重新生成一份证明后点击“出示二维码”，不要对同一个 Challenge 重复投递。

现在张三点击“碰一下，就验证！”，证明被定向发送给格尔软件。这里发送的不是两张完整原始 VC，而是一份包含已选字段、必要签名材料、Holder 证明和防重放参数的组合 Presentation。

二维码是另一种交互入口，底层传递的是同类证明。为了防止重放，同一个 Challenge 只能消费一次。

## 十、格尔软件逐项验证，而不是“看起来像真的”

【操作】切回格尔软件“钱包证明验证”，等待收到证明后点击“验证”，展示右侧逐项验证报告和披露字段。

格尔软件已经收到张三投递的组合证明。现在点击“验证”。

这里不是简单地判断 JSON 能不能打开，也不是看到签名字段就直接相信。验证程序会逐项完成以下检查：

第一，检查 Challenge 和 Domain，确认这份证明是为格尔软件本次请求生成的，不是从别处复制来的旧证明。

第二，解析张三的 Holder DID Document，用其中的公钥验证组合签名，证明当前出示者确实控制张三钱包中的私钥。

第三，分别解析上海大学和华山医院的 Issuer DID Document，用两家机构各自的公钥验证两张凭证上的 Issuer 签名。

第四，重新计算每个已披露字段的 SHA-256 摘要，确认字段值没有在签发后被修改。

第五，检查两张 VC 的状态、有效期以及相关 DID 是否仍然有效，防止已经暂停、撤销或过期的材料继续通过认证。

【操作】指向逐项披露结果。

格尔软件最终可以看到的是“张三、本科、计算机科学与技术、已毕业、A 类、合格、适合入职、有效期”等入职所需事实。

它看不到张三的学号、毕业证书编号、体检报告编号、检查明细、服务价格，更看不到任何私钥。这就是“够用即可”的最小披露。

验证完成后，系统还会写入验证台账，记录何时验证、验证结果、涉及几张凭证和披露了哪些路径，而不是再保存一份张三的完整原始材料。

## 十一、解释验证方到底信任什么

【操作】可停留在验证结果页，或切回演示页“验证通过背后实际检查了什么”。

这里需要强调：验证方不能因为收到一份选择性披露证明就无条件接受。**签名有效，只能说明内容没有被篡改并且对应私钥参与了签名；它不能单独证明上海大学在现实中确实是一所被认可的大学。**

当前 MVP 的信任起点，是平台已注册的组织资料、MySQL 中的 DID 与状态注册表、机构 KMS 保存的签名密钥，以及验证方自己的业务规则。也就是说，当前演示仍然信任平台数据库、后端和 KMS。

进入真实生产环境后，格尔软件还需要掌握或查询：哪些 Issuer 是可信的、它们被允许签发哪类凭证、Issuer 和 Holder 的 DID Document、DID 与 VC 的当前状态，以及企业自己的入职准入规则。

所以完整验证不是一句“验签通过”，而是：**信任名单通过、DID 可解析、Issuer 签名通过、Holder 签名通过、状态和期限有效、披露摘要一致，最后还要满足格尔软件自己的业务政策。**

---

## 十二、说明哪些是加密，哪些不是加密

【操作】下滑到“安全不等于所有数据都变成密文”。

老师在前次答辩中提醒我们，安全边界必须讲清楚。因此我们重新审查并整改了数据保护方案。

账号密码使用 scrypt 加随机盐做不可逆哈希；Issuer 私钥、服务端 VC 正文和披露材料使用 AES-256-GCM 加密落库；签发与出示使用 Ed25519 签名；字段承诺使用随机盐加 SHA-256 摘要。

但安全不等于所有数据都必须显示成密文。

DID Document 中的公钥必须公开，验证方才能验签；张三主动选择的“学历：本科”和“体检结论：合格”也必须让格尔软件读取。它们的安全目标不是保密，而是通过签名和摘要保证来源真实、内容未被篡改。

JSON 和 Base64URL 只是结构与编码，不是加密。当前本地演示使用 HTTP，也不能称为传输加密；生产部署必须使用 HTTPS/TLS。当前钱包中的 Holder 私钥已是不可导出 `CryptoKey`，但领取后的 VC 包还没有额外做本地内容加密，生产移动端还需要进一步使用系统安全区、硬件密钥库和生物识别解锁。

## 十三、产品价值与可扩展场景

【操作】下滑到“把材料流转变成可验证的信任流转”。

通过刚才的流程，我们把传统的材料流转变成了可验证的信任流转。

对个人来说，多份证件可以由自己统一持有，并且只披露当前场景需要的信息。

对签发方来说，学校只对学历负责，医院只对体检负责，不需要维护一个跨机构的总档案。

对验证方来说，不再只是人工看图片、盖章和复印件，而是自动检查来源、签名、期限和动态状态。

对整个生态来说，不同机构独立签发的多个事实，可以由个人按场景组合成一次证明。

所以我们的产品价值可以概括为：**一次签发、个人持有、按需组合、跨组织验证、状态可查。**

【操作】下滑到应用场景页。

这个能力不只适用于入职。它也可以用于外协工程师准入，把承包商派工、安全培训、职业资格和区域授权组合验证；可以用于教育培训，只出示学历或结业结果；可以用于医疗健康，只披露适岗结论和有效期；未来还可以用于智能体身份、能力证明和可撤销委托。

## 十四、未来最佳生产方案与收束

【操作】下滑到最后的“最佳生产方案”。

最后，我们也清楚当前 MVP 还没有天然击败传统 CA，也没有因为使用 DID 就自动实现去中心化。真正的生产方案需要把现实身份背书、公共状态治理、密码学凭证和个人控制权组合起来。

例如，教育主管部门或 CA 审核并确认上海大学的 Issuer DID，卫生主管部门或 CA 审核并确认华山医院的 Issuer DID；联盟链或多方可信注册表登记经过审核的 Issuer DID、公钥版本、密钥轮换和停用状态；学校和医院分别掌握机构私钥并签发 VC；张三的钱包掌握 Holder 私钥和凭证；格尔软件根据权威信任名单、DID、Issuer 签名、Holder 签名、VC 状态、有效期、披露摘要和自己的准入规则完成验证。

这里的信任是分工完成的：主管部门或 CA 回答“它是否真是这家机构”；多方注册表回答“公钥和状态能否被单一平台静默修改”；DID/VC 回答“谁签发了什么”；个人钱包回答“由谁持有、向谁出示什么”。

我们本阶段完成的是一条可运行、可验证、可继续替换底层信任设施的 MVP 闭环。下一步不是盲目增加页面，而是继续强化移动端钱包安全、机构密钥治理、权威 Issuer 准入、多方可信注册表和标准互操作能力。

## 十五、演示异常时的备用口播

### 1. 钱包暂时没有刷新出凭证

当前钱包使用定时轮询模拟消息提醒，正常情况下会自动出现；如果现场网络或浏览器调度延迟，我先展示已经领取到本地凭证库的结果。领取过程本身仍然包含一次性 Challenge、Holder 私钥签名、后端验签和授权交付。

### 2. “碰一下”后验证方暂时没有收到

当前是网页端对 NFC 场景的模拟，底层仍然按照选定组织定向投递。现场如果轮询出现延迟，我使用已准备好的待验证记录继续展示逐项验证报告，不影响密码学验证链路的说明。

### 3. 现场验证失败

验证失败本身也说明系统不是无条件放行。我们可以直接根据逐项报告判断是 Challenge 已消费、目标组织不匹配、凭证过期、生命周期异常、Issuer 签名失败，还是披露摘要不一致。

### 4. 老师追问“为什么不直接展示完整 JSON”

JSON 是机器交换格式，不是最终用户体验。我们保留技术数据用于审计和学习，但产品界面优先显示“谁在什么时候给谁签发了什么”以及“验证了哪些字段、结果是什么”。真正的安全来自密钥、签名、摘要、状态与访问边界，而不是让页面看起来像一串复杂代码。
