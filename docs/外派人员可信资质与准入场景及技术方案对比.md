# 外派人员可信资质与准入场景及技术方案对比

## 1. 文档目的

本文用于说明“信证台”在外派人员、外协人员和供应商驻场人员准入场景中的业务价值，并比较 DID/VC 与传统 PKI/CA 实现路径的优缺点。

本文不主张“PKI/CA 无法实现该场景”，也不主张 DID 天然可信或 DID/VC 必然比 PKI 更安全。两种方案都以非对称密码、数字签名、密钥管理和信任治理为基础，差异主要在于业务建模、凭证流转方式和跨组织协作模式。

## 2. 业务场景

### 2.1 场景概述

外派工程师、承包商员工或供应商驻场人员进入园区、工厂、数据中心或施工现场前，通常需要证明多项资格：

- 本人与现场出示者一致；
- 当前受雇于或受派遣于指定承包商；
- 具备电工、焊工、高处作业等职业资格；
- 已完成园区或项目要求的安全培训；
- 已获得当前项目、区域和时间段的作业授权；
- 上述资格未过期、未暂停、未撤销。

这些事实并不由同一个机构掌握：承包商掌握劳动或派遣关系，资格机构掌握职业资格，培训机构掌握安全培训记录，项目方掌握项目授权，园区运营方负责最终准入判断。

### 2.2 当前常见流程

当前流程往往依赖身份证复印件、证书图片、Excel 名单、盖章文件、邮件和多个业务系统：

1. 外派人员向承包商提交材料；
2. 承包商整理后发送给项目方；
3. 项目方或园区分别联系签发机构核实；
4. 各园区保存一套人员和证件副本；
5. 门岗根据名单、图片或人工审核结果放行；
6. 人员离职、资格撤销或项目结束后，再由各方人工同步状态。

### 2.3 核心痛点

#### 重复提交与重复存储

同一人员进入不同项目时需要反复提交相同材料，各验证单位又分别保存个人信息和证件副本。

#### 多来源材料难以一次核验

在职关系、职业资格、安全培训和项目授权来自不同机构，验证人员需要在多个系统和联系人之间切换。

#### 静态材料无法反映当前状态

证书图片仍然存在，不代表资格仍然有效；人员可能已经离职，培训可能已经过期，项目授权也可能已经取消。

#### 责任边界不清

综合证明由谁出具、谁为哪一项事实负责不清晰。承包商不应替资格机构证明职业资格，资格机构也不应替项目方证明区域授权。

#### 个人信息过度披露

为了证明一项作业资格，人员经常被要求提交包含身份证号、住址等无关字段的完整材料。

#### 跨组织接口成本高

每个园区分别对接承包商、培训机构和资格机构，会形成大量点对点接口和数据同步关系。

## 3. 目标业务结果

目标不是简单地把纸质证书变成带签名的电子文件，而是形成以下闭环：

> 多个机构分别签发业务凭证，个人统一持有并按场景组合出示，验证方依据自身准入规则一次验证来源、持有人、有效期和状态，最终给出明确的业务准入结论。

典型结果如下：

```text
准入结论：允许进入 A 区执行电气维修

✓ 当前属于获准承包商
✓ 电工作业资格有效
✓ 园区安全培训有效
✓ A 区项目授权有效
✓ 所有凭证绑定同一持有人
✓ 本次出示绑定当前验证请求，未发生重放
```

## 4. 使用 PKI/CA 的实现路径

PKI/CA 可以完成该场景，但完整方案不仅包含 CA。通常还需要：

1. 建设统一 CA、行业 CA 或多 CA 交叉信任体系；
2. 为人员、机构或设备签发 X.509 身份证书；
3. 用证书扩展、X.509 属性证书、CMS 签名文件或自定义令牌表达业务属性；
4. 使用 CRL、OCSP 和业务状态接口检查证书与资格状态；
5. 建设可信机构目录，规定谁能证明哪类业务事实；
6. 统一人员标识、属性编码、证书策略和数据格式；
7. 建设持证客户端、准入规则引擎、验证工具和审计系统。

如果把单位、工种、培训和项目权限全部写入一张身份证书，任何属性变化都可能导致整张证书重签。如果将它们拆成多张属性证书或签名令牌，方案会更灵活，但其业务结构已经接近“多个签发者分别签发业务凭证”的 VC 模型。

## 5. 使用 DID/VC 的实现路径

### 5.1 标识与控制权

外派人员在个人钱包中控制 Holder DID 及对应私钥；机构分别控制自己的 Issuer DID 和签发密钥。Holder 私钥控制只能证明当前出示者控制相应 DID，不能自动证明姓名、资格或雇佣关系。

### 5.2 多机构分别签发

各机构只对自己掌握的事实负责：

| 凭证 | 建议签发者 | 证明内容 |
|---|---|---|
| 外派/在职关系 VC | 承包商 | 人员当前属于该机构并被派往指定项目 |
| 职业资格 VC | 资格认证机构 | 工种、等级、证书编号和有效期 |
| 安全培训 VC | 培训机构或园区 | 已完成指定培训且仍在有效期内 |
| 项目授权 VC | 项目业主 | 允许在指定项目、区域、时段执行指定作业 |

### 5.3 个人持有和组合出示

凭证签发后交付个人钱包。园区发起包含业务目的、所需声明、challenge、domain 和有效期的验证请求，个人确认后由钱包组合多份凭证或其最小披露结果，生成一次准入出示。

### 5.4 验证和业务决策

验证方依次检查：

- 数据格式和凭证类型；
- Issuer DID、签名和密钥版本；
- Issuer 是否有权签发该类凭证；
- Holder DID 与持有人证明；
- challenge、domain、有效期和防重放；
- VC 是否过期、暂停、撤销或被替代；
- 多份 VC 是否属于同一主体；
- 业务字段是否满足当前准入策略。

密码学验证通过不等于业务必然准入，最终结论由验证方的业务策略决定。

## 6. DID/VC 相比 PKI/CA 的重点优势

### 6.1 业务声明与身份密钥解耦

DID/密钥用于证明控制权，VC 用于表达在职、资格、培训和授权。不同业务事实拥有独立签发者、有效期和状态；项目授权变化不必重新签发职业资格或人员身份。

### 6.2 天然适配多签发者责任边界

多个机构可以独立签发凭证，每个机构只承担相应声明的责任，不需要一个中心 CA 理解并背书所有业务事实。

这里的“独立签发”不等于“无需治理”。平台仍需规定哪些机构可信，以及它们有权签发哪些凭证类型。

### 6.3 个人成为凭证流转中心

机构将凭证交付个人，个人可在不同项目中重复使用有效凭证。验证方不必默认连接每个签发机构的人员数据库，也不必长期保存完整证件副本。

### 6.4 支持多凭证组合满足一个业务目的

一次准入可以组合在职关系、职业资格、安全培训和项目授权四类 VC，并将它们绑定到同一 Holder 和同一次验证请求。这比把所有易变属性塞进一张身份证书更符合各类事实的实际生命周期。

### 6.5 更适合最小披露和用户授权

验证请求明确说明请求方、目的、字段范围和有效期；个人可以确认或拒绝。结合 SD-JWT VC 等机制，可只披露当前准入所需的字段。

若产品仅实现字段选择和授权流程，而未实现标准密码学选择性披露，应明确称为“最小披露流程”，不能宣称实现零知识证明或完全不可关联。

### 6.6 降低点对点业务接口依赖

签发者按照统一 Schema 签发，验证方按照统一格式和信任规则验证。新增认可的签发机构时，主要调整信任目录和策略，而不是让每个验证方分别建设一套人员数据查询接口。

### 6.7 更容易形成完整的签发—持有—出示—验证模型

传统 PKI 的核心关系是 CA、证书主体与依赖方；VC 的核心角色直接对应 Issuer、Holder、Subject 和 Verifier，更适合个人携带多机构业务证明跨组织使用。

## 7. DID/VC 的不足与实施风险

### 7.1 不会消灭信任治理

DID 只能帮助解析验证方法，Issuer 签名只能证明某机构签发过声明。平台仍需建设机构准入、Schema 管理、签发权限、信任目录和验证规则。

### 7.2 钱包密钥恢复和设备迁移复杂

个人自主控制私钥会带来手机丢失、设备更换、备份恢复和多设备同步问题。托管钱包易用但控制权较弱，非托管钱包控制权强但恢复成本更高。

### 7.3 标准和生态成熟度低于传统 PKI

DID Method、VC 安全格式、Holder Binding、状态机制和选择性披露方案需要明确技术剖面，不能只写“遵循 DID/VC 标准”而回避互操作细节。

### 7.4 状态服务和隐私仍需设计

已交付的 VC 无法从个人设备中物理收回，只能通过状态服务标记失效。逐条在线查询还可能泄露验证行为，需要权衡实时性、可用性和隐私。

### 7.5 纯内部场景收益可能不足

若所有签发者、人员和验证者都属于同一家企业，且已有成熟 CA、IAM 和门禁体系，DID/VC 的新增复杂度可能高于收益。其优势通常随参与机构、凭证类型、跨场景复用和隐私要求增加而增强。

## 8. PKI/CA 的主要优势

- X.509、HSM、CRL、OCSP、智能卡和 USB Key 等技术与产品成熟；
- 在 TLS、VPN、设备认证、企业登录、代码签名和电子签章中具有明显优势；
- 企业已有 CA、IAM 和目录服务时，内部部署成本较低；
- 合规、密钥保护、证书策略和运维体系更成熟。

因此，本项目不应把 DID/VC 定位为 PKI 的全面替代品。机构签发密钥仍可由成熟 PKI、KMS 或 HSM 保护，DID/VC 主要解决其上的业务凭证协作问题。

## 9. 对比结论

| 维度 | PKI/CA 方案 | DID/VC 方案 |
|---|---|---|
| 核心强项 | 主体/名称与公钥绑定 | 多方业务声明的持有与出示 |
| 多机构业务属性 | 需要扩展或额外系统 | 原生建模为多份 VC |
| 属性生命周期 | 易与证书重签耦合 | 每份 VC 独立管理 |
| 个人持有与跨域使用 | 需要额外客户端和协议 | 核心角色模型 |
| 多凭证组合 | 需要定制 | 适合组合出示 |
| 最小披露 | 需要额外密码协议 | 有 SD-JWT 等专门方案 |
| 信任治理 | 根 CA、证书策略及业务规则 | Issuer 目录、Schema 及业务规则 |
| 技术成熟度 | 高 | 相对较低 |
| 密钥设备与合规 | 成熟 | 需要结合现有安全基础设施 |

## 10. 产品定位与答辩表述

推荐产品定位：

> 信证台面向外派人员和供应商人员的跨组织可信资质与准入场景，让不同机构签发的业务资格由个人安全持有、按目的组合出示，并由园区或项目方依据自身规则即时验证。

推荐差异化表述：

> 我们不替代 PKI。PKI 提供成熟的密钥、证书和数字签名基础；信证台解决 PKI 之上的多机构业务凭证如何分别签发给个人、由个人携带和组合出示，并按场景规则完成跨组织验证。

若被问“PKI 能否实现”，建议回答：

> 可以。用 PKI 完成整个业务，还需要在证书体系之外增加属性凭证、持证客户端、跨机构信任目录、业务状态接口、披露授权和准入规则引擎。我们采用 DID/VC，不是因为 PKI 不能验签，而是因为 DID/VC 将多方业务声明、个人持有和场景化出示作为一等模型，更符合该场景的数据流和责任边界。

## 11. 对外表达禁区

产品材料不应宣称：

- DID/VC 天然比 PKI 更安全；
- DID 本身可以证明真实身份；
- 去中心化意味着不再需要可信机构；
- 使用区块链即可保证业务事实真实；
- VC 已签名就一定应被业务接受；
- PKI/CA 无法实现跨组织资质验证；
- 字段选择等同于密码学选择性披露或零知识证明。
