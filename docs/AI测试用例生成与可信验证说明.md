# AI 测试用例生成与可信验证说明

## 1. AI 生成测试用例的思路

本项目没有让 AI 只根据代码表面生成少量正常流程，而是先向 AI 提供产品需求、DID/VC 生命周期、接口约束和页面操作流程，再由 AI 从多个角度拆分测试点。

AI 首先识别正常业务闭环，例如创建 DID、签发 VC、执行七项验证以及暂停、恢复、更新和撤销凭证；随后补充异常输入、非法状态转换、篡改检测、敏感信息保护、搜索分页和页面交互等场景。测试用例最终按照接口测试、单元测试、集成测试、功能测试、兼容性测试和安全测试分类，使底层规则、组件协作和用户操作都能得到验证。

生成用例时遵循“一条用例验证一个明确目标”的原则。对于用户功能，测试不仅检查接口返回成功，还会通过 Chromium 页面执行点击、输入、搜索和状态操作，并检查用户实际看到的结果。

## 2. 人工参与和后续补充

测试用例并不是由 AI 单方面决定。人工负责提出业务目标、判断系统边界、审核用例是否符合实际操作，并决定哪些结果可以作为验收依据。

项目初期的用例主要来自需求文档、生命周期规则和已有代码。后续人工检查时发现，功能测试数量较少，部分用例虽然验证了底层服务，却没有形成面向用户操作的独立功能用例。因此又明确要求补充凭证签发页台账、验证记录台账、中文失败原因、搜索分页、空状态、错误提示、移动视口和日志操作等场景，并要求新增用例在真正执行前标记为“未执行”，不能因为相关底层测试通过就直接写成“通过”。

在人工确认覆盖范围后，AI 才继续生成相应的 Playwright 自动化脚本并执行。人工参与的重点包括：提出遗漏场景、确认测试分类、审核步骤和预期结果、判断失败是否为真实缺陷，以及决定是否接受 AI 提出的修改方案。

## 3. 如何验证 AI 执行结果可信

本项目不以 AI 回复中的“测试通过”作为最终依据，而以测试工具和本地证据文件为依据。执行 `npm run test:evidence` 后，系统会运行全部 Node 测试和 Chromium UI 测试。即使 Node 阶段失败，UI 阶段仍会继续执行，最终由两个测试进程的退出码和解析结果统一判断批次是否通过。

每次执行都会在本机永久生成独立的 `test-records` 批次目录，其中保存：

- Node 和 UI 的完整原始日志；
- 测试开始时间、结束时间、执行命令和退出码；
- Git 分支、提交编号及工作区是否存在未提交修改；
- 本次实际发现的测试用例标题和执行结果；
- Playwright HTML 报告，以及失败时的截图、视频和 Trace；
- 所有证据文件的 SHA-256 哈希清单。

只有测试结果能够正常解析、退出码为零、失败数为零、跳过和 todo 为零，并且证据文件完整生成时，批次才会被判定为通过。可以使用以下命令重新验证证据是否被修改：

```bash
npm run test:evidence:verify -- <测试批次目录>
```

如果日志或报告被修改、删除或额外加入文件，验证命令会返回非零退出码，并指出对应文件。历史批次可以使用 `npm run test:evidence:list` 查看，系统不会自动删除旧记录。

因此，本项目中的测试结论不是由 AI 自己声明，而是形成了“测试用例—自动化脚本—测试框架结果—原始执行日志—Git 提交—哈希证据”的本地追踪链。成功和失败都可以找到对应的执行记录和判断依据。

## 4. 人机协作结论

本项目采用的原则是：**人确定目标、覆盖范围和验收标准，AI 辅助生成用例、编写脚本和分析失败，测试框架产生客观结果，本地证据链证明结果是否可信。** AI 提高了整理和执行效率，但最终测试责任仍由人工承担。
