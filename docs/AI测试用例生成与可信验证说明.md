# AI 测试用例生成、人工参与与可信验证说明

## 1. 背景与目标

本项目使用 AI 辅助整理测试用例、生成自动化脚本并执行测试，但没有把 AI 回复中的“测试通过”直接作为验收结论。原因是 AI 可能遗漏业务边界，也可能误解测试输出；即使自动化脚本显示通过，也需要确认测试是否真正执行、断言是否有效、执行环境和代码版本是否明确。

因此，本项目把测试工作拆成三部分：人工确定目标和验收标准，AI 辅助分析、生成与执行，测试框架和本地证据文件提供客观依据。最终目标不是证明“AI 会测试”，而是让每条成功或失败结果都能找到对应的用例、脚本、日志和代码版本。

## 2. AI 生成用例时使用的项目输入

AI 并不是脱离项目背景直接编写测试。生成用例前，AI读取并对照了以下内容：

- 产品需求和 MVP 范围，确认 Issuer、Holder、Verifier 和演示管理员的职责；
- DID 生命周期，包括创建、更新、密钥轮换和不可逆停用；
- VC 生命周期，包括 active、suspended、replaced、expired 和 revoked；
- 七项验证规则：格式、Issuer、DID 状态、密钥版本、签名、有效期和凭证状态；
- HTTP 接口、请求限制、错误码、搜索分页和日志接口；
- 页面上的 DID、签发、验证、台账和日志操作；
- 私钥保护、日志脱敏、XSS、目录穿越和安全响应头等约束；
- 已有 Node、接口、集成、安全和 Playwright 测试。

AI 根据这些材料建立“需求—测试点—预期结果”的关系。例如，“撤销后的 VC 不允许恢复”会被拆成状态转换测试、接口拒绝测试和用户页面操作测试；“验证失败原因可见”既要检查后端返回的失败项，也要检查验证记录台账中的中文备注。

## 3. 测试点的拆分方法

本项目将用例分为接口测试、单元测试、集成测试、功能测试、兼容性测试和安全测试。

| 测试类型 | 主要验证内容 | 项目示例 |
|---|---|---|
| 单元测试 | 独立函数和规则 | Ed25519、Base58、稳定序列化、查询分页、失败原因映射 |
| 接口测试 | HTTP 边界和响应契约 | DID/VC 创建、状态码、非法 JSON、请求大小限制 |
| 集成测试 | 多组件协作 | VC 服务与 DID Method、历史密钥、JSON Store、审计日志 |
| 功能测试 | 用户可见操作和业务结果 | 创建 DID、签发 VC、暂停恢复、查看验证失败原因 |
| 兼容性测试 | 当前 Chromium 环境中的布局适配 | 桌面/移动视口、导航、表单和横向溢出 |
| 安全测试 | 数据保护和攻击面 | 私钥泄漏、日志脱敏、XSS、原型污染、目录穿越 |

生成用例时尽量遵循“一条用例验证一个明确目标”。例如，暂停和恢复最初可能出现在一条生命周期旅程中，但面向用户的功能表中会分别列出“暂停 active VC”和“恢复 suspended VC”，这样失败时能直接定位具体操作。

## 4. 人工参与和后续补充的真实过程

初版测试主要依据需求、状态机和已有代码生成，底层规则与接口覆盖较多。但人工审查 Excel 用例表后发现，“功能测试”数量偏少，一些测试虽然验证了服务和接口，却没有站在用户角度执行页面操作。例如，接口能够暂停凭证，并不等于用户在签发页底部台账点击“暂停”后一定能看到正确状态。

针对这个问题，人工后续提出了更明确的要求：

1. 测试类型统一为六类，不再使用含义模糊的“组件/服务测试”；
2. 功能测试必须包含面向用户操作的独立用例；
3. 补充签发页底部 VC 台账和总览台账同步验证；
4. 补充验证记录台账的成功备注、单项失败原因、多项失败原因和未知原因兜底；
5. 补充搜索、清除、分页、空状态、错误提示和不可逆操作确认；
6. 兼容性只验证当前 Chromium 的桌面和移动视口，不扩大到其他浏览器；
7. 尚未实际执行的新用例必须写“未执行”，不能因为同模块的单元测试通过就推断成功。

AI 根据这些人工要求重新整理高覆盖用例表，并生成对应的 Playwright 自动化脚本。新增脚本按总览与空状态、DID 用户操作、凭证台账、验证记录、日志与重置、响应式兼容性等功能域拆分。新增的 33 条 Chromium UI 自动化首先单独执行，33 条全部通过，然后再并入全量测试。

这个过程说明，AI 可以快速扩展测试，但“哪些功能必须覆盖、怎样才算用户操作、什么时候可以写通过”由人工决定。

## 5. proof 篡改随机失败案例

全量执行时曾出现一条篡改测试失败：测试期望修改 proof 后验证失败，但实际结果仍然有效。仅看一次结果，可能会误认为签名验证存在严重漏洞；随后单独复跑多次又全部通过，也可能被简单解释为偶发环境问题。

进一步检查测试代码后发现，旧脚本把签名首字符固定改成 `A`：

```js
vc.proof.proofValue = `A${vc.proof.proofValue.slice(1)}`;
```

如果原签名本来就以 `A` 开头，测试数据实际没有发生变化，因此凭证仍然有效。这不是业务验签错误，而是测试构造存在概率性缺陷。

修正后，脚本根据原字符在 `A` 和 `B` 之间切换，保证 proof 一定变化：

```js
const first = vc.proof.proofValue[0] === 'A' ? 'B' : 'A';
vc.proof.proofValue = `${first}${vc.proof.proofValue.slice(1)}`;
```

修改后目标测试连续执行 20 次，20 次全部通过，再执行完整 Node 和 Chromium UI 回归。这个案例体现了人工判断的重要性：不能因为 AI 复跑成功就忽略第一次失败，也不能在未分析原因时直接修改业务代码。

## 6. 自动化执行和结果判定

项目通过以下命令执行带证据的本地测试：

```bash
npm run test:evidence
```

统一执行器依次运行 Node 和 Chromium UI。即使 Node 阶段失败，UI 阶段仍继续执行，使同一批次能够保留两个阶段的完整信息。

一个批次只有同时满足以下条件才被判定为通过：

- 两个阶段的进程退出码都是 0；
- 测试框架输出能够被正常解析；
- failed、skipped 和 todo 都是 0；
- 用例清单、元数据、日志和报告成功写入；
- SHA-256 证据清单成功生成。

工作区存在未提交修改时仍允许执行，但 `metadata.json` 会明确记录 `clean: false`。这表示测试验证了“当前提交加本地修改”，不能把结果解释为某个纯 Git 提交的结果。

## 7. 本地永久证据包

每次运行都会创建互不覆盖的本地批次，并永久保留：

```text
test-records/
└── 2026-07-12/
    └── 2026-07-12T20-14-19+08-00/
        ├── metadata.json
        ├── result.md
        ├── node.log
        ├── ui.log
        ├── test-cases.json
        ├── playwright-report/
        ├── test-results/
        ├── evidence-manifest.json
        └── checksums.sha256
```

各类证据的作用如下：

- `node.log`、`ui.log`：测试框架的完整原始输出，包含通过数、失败断言、文件位置和耗时；
- `metadata.json`：记录执行时间、命令、退出码、Node/Playwright 版本、Git 分支和提交；
- `test-cases.json`：记录本次实际发现的测试标题、阶段、结果和可解析的文件位置；
- `playwright-report/`：成功和失败均可打开的 HTML 报告；
- `test-results/`：UI 失败时保存截图、视频、Trace 和错误上下文；
- `evidence-manifest.json`、`checksums.sha256`：记录文件大小和 SHA-256，用于发现证据被修改、删除或额外加入。

`test-records/` 已加入 `.gitignore`，不会上传 GitHub，也不会被自动清理，所有批次只在本机永久保存。

## 8. 人工如何复核结果

人工可以按以下步骤检查一次 AI 触发的测试是否可信：

1. 查看 `metadata.json`，确认 Git commit、分支、工作区状态和执行命令；
2. 查看 `result.md`，确认总数、通过、失败、跳过和退出码；
3. 在 `node.log` 或 `ui.log` 中查找具体测试标题和原始输出；
4. 在 `test-cases.json` 中确认 Excel 用例对应的自动化测试确实被发现；
5. UI 失败时打开 HTML 报告、截图、视频或 Trace，复核浏览器实际执行过程；
6. 使用哈希命令检查证据是否被修改：

```bash
npm run test:evidence:verify -- <测试批次目录>
```

验证结果中的 `valid` 必须为 `true`，且 `missing`、`extra`、`modified` 均为空。历史批次可通过以下命令查看：

```bash
npm run test:evidence:list
```

如果需要更强的确认，可以在相同代码和环境下重新执行，比较两次批次的测试数量、失败项和 Git commit。关键随机性测试还应进行多次重复运行，而不是只看一次全绿。

## 9. 人、AI 和测试工具的职责

| 参与方 | 主要职责 | 不能替代的部分 |
|---|---|---|
| 人工 | 提出目标、补充遗漏、审核业务边界、判断失败性质、承担最终验收责任 | 不能把不理解的 AI 输出直接作为结论 |
| AI | 分析需求、拆分测试点、生成脚本、执行命令、整理和解释证据 | 不能自行决定业务是否正确，也不能只凭描述宣告通过 |
| 测试框架 | 执行代码和浏览器操作、计算断言、产生退出码与原始报告 | 不能判断需求本身是否合理或覆盖是否充分 |
| 本地证据链 | 固化提交、环境、日志、报告和哈希，支持事后复核 | 不能替代人工对测试设计质量的判断 |

## 10. 结论

本项目形成的协作方式是：**人工确定测试目标和验收标准，AI 辅助生成用例、编写脚本和分析失败，测试框架产生客观结果，本地永久证据链证明结果是否可信。**

AI 的价值主要体现在提高测试点整理、脚本生成和问题定位效率；人工的价值体现在发现覆盖不足、提出后续补充、识别随机测试缺陷和决定最终结论。通过这种分工，成功和失败都不再只是 AI 的一句话，而是可以追踪到具体用例、执行记录、代码版本和验证依据。
