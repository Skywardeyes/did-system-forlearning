### 00:40–02:00　A：项目背景与问题

#### 本段目标

从真实结业证书场景引出项目，不直接堆叠技术名词。

#### 建议讲述

在传统结业证书场景中，验证者通常只能查看纸质证书、图片或向发证机构人工查询。这种方式存在几个问题：证书内容可能被修改，验证过程依赖中心机构，验证者很难仅凭一份数据判断签发者是谁、内容是否被篡改、证书是否已经失效。

因此，我们把问题具体化为：培训机构如何签发一份机器可读、可验证的电子结业凭证？学员怎样持有它？第三方怎样独立检查凭证内容、签发者身份、有效期和当前状态？

项目没有追求生产级身份平台，而是希望在本地环境中把这些概念变成可操作、可观察、可测试的完整流程。

#### 建议强调

- “独立验证”不等于完全脱离所有信任，而是验证依据由身份文档、公钥、签名和状态共同组成。
- 项目选择本地演示，是为了降低区块链、网络和外部身份服务对学习过程的干扰。
- 本系统不宣称已经实现正式生产级的 W3C 互操作方案。

### 02:00–03:20　A：DID 与 VC 基本概念

#### 本段目标

让没有相关背景的听众能够理解后续演示中的四个角色和两个核心对象。

#### 建议讲述

DID 是去中心化标识符。在本项目中，一条 DID 记录不仅有标识符，还关联名称、角色、状态、公钥、验证方法和 DID Document。它相当于后续解析签发者公钥和判断身份状态的入口。

VC 是可验证凭证。它包含签发者 Issuer、持有人 Holder、证书声明、有效时间和密码学证明 proof。在我们的场景中：

- Issuer 是“可信学习中心”；
- Holder 是学员“张晓明”；
- VC 是“数字身份与可信凭证训练营”的结业证书；
- Verifier 是任何拿到 VC 并执行验证的人或系统。

正常流程是：Issuer 创建 DID，Holder 创建 DID，Issuer 使用自己的私钥对 VC 签名，Verifier 通过 Issuer DID Document 中的公钥检查签名和其他状态。

#### PPT/画面建议

```text
Issuer DID ──私钥签名──> VC ──交付──> Holder
                            │
                            └──提交──> Verifier
                                      │
                                      └──解析 Issuer DID、公钥、状态并验证
```

#### 代码依据

- DID 创建和注册：`src/vc-service.js` 中的 `createDid()`。
- VC 签发：`issueCredential()` 和 `issueCredentialIntoState()`。
- VC 验证：`verifyCredential()`。
- DID Document 构造：`src/crypto.js` 与 `src/did-methods.js`。

### 03:20–04:30　A：项目目标、MVP 与实现边界

#### 本段目标

说明做了什么，也主动说明没有做什么。

#### 建议讲述

我们为项目确定了三层目标。

第一层是核心业务闭环：创建 Issuer 和 Holder 身份、签发 VC、查看 VC、验证正常凭证，并能观察篡改和撤销后的失败结果。

第二层是生命周期：`did:example` 支持更新、密钥轮换和不可逆停用；VC 支持暂停、恢复、替换、过期和撤销。

第三层是工程完整性：提供本地存储、结构化日志、搜索分页、安全响应头以及单元、集成、API、功能、安全和 UI 测试。

同时我们明确了边界：系统不接入真实区块链，不进行真实身份核验，不提供生产级密钥托管，也没有实现正式注册的 W3C Data Integrity cryptosuite。项目中的 proof 名称是 `EducationalEd25519Signature2026`，用于明确这是教学实现。

#### 建议强调

主动说明边界不是项目缺陷，而是 MVP 范围管理和技术诚实性。

### 04:30–06:00　A：系统架构与运行方式

#### 本段目标

用一张图讲清楚系统组成和一次请求如何流转。

#### 建议讲述

项目采用轻量级本地架构。前端是原生 HTML、CSS 和 JavaScript 单页应用；后端使用 Node.js 内置 HTTP 模块；核心业务集中在 VC Service；数据和日志分别写入本地 JSON 文件。

浏览器发起 `/api/*` 请求后，`server.js` 负责路由、请求体检查和错误映射，再调用 `VcService` 完成业务操作。业务数据通过 `JsonStore` 持久化，审计信息通过独立的 `LogService` 和 `LogStore` 保存。普通页面和脚本则由同一个 Node 服务提供，因此启动后只需要访问一个本地地址。

项目的正常启动命令是 `npm start`，默认监听 `http://127.0.0.1:4173`。运行时不依赖数据库和第三方 npm 包；Node.js 版本要求为 20 或更高。

#### 架构图建议

```text
浏览器 UI（public/）
       │ HTTP / JSON
       ▼
server.js：静态资源、API 路由、输入限制、安全响应头
       │
       ▼
vc-service.js：DID/VC 业务规则、状态机、七项验证
    │                         │
    ▼                         ▼
store.js                  log-service.js
业务 JSON                 日志脱敏、查询
    │                         │
    ▼                         ▼
data/store.json           data/logs.json
```

#### 实际代码依据

- `src/server.js` 使用 Node.js `createServer()`。
- 默认端口来自 `PORT`，默认值为 `4173`。
- 业务数据路径来自 `DATA_FILE`，日志路径来自 `LOG_FILE`。
- `readJson()` 限制请求必须是 `application/json`，且请求体不超过 1 MB。
- 静态资源和 API 都返回 CSP、`no-referrer`、`nosniff` 等基础安全响应头。

### 06:00–07:25　A：两种 DID Method 与生命周期

#### 本段目标

解释项目为什么同时实现 `did:example` 和 `did:key`，以及二者的差异。

#### 建议讲述

系统实现了两种教学用 DID Method。

`did:example` 是本地注册型身份。它支持修改名称和服务地址、轮换密钥以及停用。更新时使用 `expectedVersion` 做版本冲突检查；密钥轮换后，旧公钥会进入历史密钥记录，所以轮换前签发的 VC 仍能找到对应版本的公钥完成验签；身份停用后不可恢复，也不能继续参与新凭证签发。

`did:key` 则直接由 Ed25519 公钥指纹生成。代码使用 Ed25519 multicodec 前缀 `0xed01` 和 Base58btc 编码形成指纹。由于标识符与公钥绑定，它被设计为不可更新、不可轮换、不可停用。

实现两种 Method 的目的，是让学习者看到 DID 并不是只有一种统一行为，不同 Method 会定义不同的解析和生命周期能力。

#### 对比表

| 能力 | `did:example` | `did:key` |
|---|---|---|
| 生成方式 | 本地随机标识并注册 | 由 Ed25519 公钥指纹生成 |
| 更新名称/服务 | 支持 | 不支持 |
| 密钥轮换 | 支持 | 不支持 |
| 停用 | 支持且不可逆 | 不支持 |
| 历史密钥验证 | 支持 | 无轮换场景 |

#### 代码依据

- `src/did-methods.js`：`ExampleDidAdapter`、`KeyDidAdapter`、`DidMethodRegistry`。
- `src/vc-service.js`：`updateDid()`、`rotateDidKey()`、`deactivateDid()`。
- `src/crypto.js`：`base58Encode()` 与 Ed25519 密钥生成。

### 07:25–08:50　A：VC 签名、七项验证与状态机

#### 本段目标

为后半段现场验证建立理论基础。

#### 建议讲述

签发 VC 时，系统先构造凭证主体和 proof 之外的字段，再通过稳定键序 JSON 序列化得到确定的字节内容，最后用 Issuer 的 Ed25519 私钥签名。签名结果以 Base64URL 形式写入 `proofValue`，proof 还记录验证方法和密钥版本。

验证不是只调用一次密码学验签，而是依次完成七项检查：

1. 格式完整性；
2. Issuer DID 能否在本地注册表中解析；
3. Issuer DID 当前是否可用；
4. proof 指定的密钥版本和验证方法能否解析；
5. Ed25519 签名是否正确；
6. 当前时间是否在有效期内；
7. 本地凭证状态是否为 active。

只有七项全部通过，最终的 `valid` 才为 true。这也说明“签名正确”和“凭证整体有效”是两个不同层次。例如凭证被撤销后，原始签名仍可能正确，但凭证状态检查会失败，因此整体无效。

#### VC 状态机

```text
active ──暂停──> suspended ──恢复──> active
   │                 │
   ├──替换──> replaced（终态）
   ├──撤销──> revoked（终态）
   └──到期──> expired（终态）

suspended 也可以被替换或撤销。
```

#### 代码依据

- `VC_TRANSITIONS` 明确合法状态转换。
- `signCredential()` 使用 Ed25519 私钥签名稳定序列化后的凭证。
- `verifyCredentialSignature()` 删除 proof 后重建签名载荷并用公钥验签。
- `verifyCredential()` 汇总七项检查，并使用 `checks.every()` 计算最终结果。

### 11:50–12:45　B：签发结业 VC

#### 操作步骤

1. 进入“凭证签发”。
2. 选择 Issuer 和 Holder。
3. 确认学员姓名、课程名称和结业日期。
4. 点击签发。
5. 在右侧 VC JSON 中指出关键字段。

#### 建议讲述

现在由 Issuer 为 Holder 签发结业凭证。前端通过 `POST /api/credentials` 提交 Issuer DID、Holder DID、学员姓名、课程名称和结业日期。

后端首先确认两个 DID 存在、角色正确且没有停用，然后构造 VC。请大家关注五部分：

1. `issuer` 指向发行方 DID；
2. `credentialSubject.id` 指向持有人 DID；
3. credentialSubject 中保存姓名、课程和完成日期；
4. `validFrom` 和 `validUntil` 定义有效期；
5. `proof` 中保存验证方法、密钥版本、证明用途和 `proofValue`。

`proofValue` 不是对某一个字段单独签名，而是对稳定序列化后的完整未签名凭证进行 Ed25519 签名。因此后续修改姓名或课程都会破坏签名一致性。

#### 代码依据

- API：`POST /api/credentials`。
- 业务：`issueCredentialIntoState()`。
- 签名：`signCredential()`。
- 默认有效期：签发时间至系统设置的到期时间。

### 12:45–13:40　B：执行完整验证

#### 操作步骤

1. 进入“凭证验证”。
2. 点击“载入最新凭证”。
3. 点击“执行完整验证”。
4. 按页面顺序解释七项检查。

#### 建议讲述

现在提交原始 VC。系统不会只返回一个成功或失败，而是返回七项检查明细。

- 格式完整性：必要字段是否存在；
- Issuer DID 解析：本地是否能找到签发方；
- DID 当前状态：签发方是否已经停用；
- 密钥版本：proof 指定的历史或当前密钥能否解析；
- Ed25519 签名：内容是否与签名一致；
- 有效期：当前时间是否处于生效区间；
- 凭证当前状态：本地签发记录是否为 active。

这张原始凭证七项全部通过，所以最终结果为有效。验证完成后，系统还会保存验证时间、凭证 ID 和失败检查项，形成最近验证记录。

#### 时间控制

### 13:40–14:20　B：模拟篡改并观察失败

#### 操作步骤

1. 点击“模拟篡改姓名”。
2. 指出 JSON 中姓名后新增“（已修改）”。
3. 再次执行验证。
4. 指向“Ed25519 签名”失败。

#### 建议讲述

我们现在只修改 credentialSubject 中的学员姓名，不修改 proof。再次验证后，可以看到格式、Issuer 和密钥仍可能正常，但 Ed25519 签名失败。

原因是验证端会删除 proof，按照同样的稳定序列化规则重建凭证字节，再用 Issuer 公钥验证原签名。姓名变化导致字节内容变化，原签名自然无法匹配。

这说明数字签名能发现内容篡改，但它不会阻止别人修改文件；它提供的是修改后可被检测的能力。

#### 关键学习点

“防篡改”更准确的表达是“篡改可检测”，不是让数据物理上无法被编辑。

### 14:20–15:30　B：暂停与恢复演示

#### 操作步骤

1. 返回运行总览中的凭证台账。
2. 找到原始凭证，点击“暂停”。
3. 打开该原始凭证或把提前保存的原始 VC 载入验证页，执行验证。
4. 指出 Ed25519 签名仍通过，但凭证状态为 suspended，整体失败。
5. 返回凭证台账，点击“恢复”。
6. 使用同一张原始 VC 再次验证，确认凭证状态恢复为 active，七项重新通过。

#### 建议讲述

系统使用明确的状态机限制转换。active 可以暂停、替换或撤销；suspended 可以恢复、替换或撤销；replaced、expired 和 revoked 都是终态，不能再次转换。

现在我们先暂停原始凭证。重新验证时，原始数据没有被修改，因此 Ed25519 签名仍然通过；但“凭证当前状态”显示 suspended，最终结果无效。恢复之后再次验证，同一份 VC 又可以通过。

这个过程证明暂停是可逆状态，也再次体现两个层次：签名回答“内容是否由对应私钥签发且没有被修改”，状态回答“这张凭证现在是否仍被认可”。

#### 时间控制

不要重新讲七项验证，只对比签名检查和凭证状态检查。暂停、验证、恢复、验证必须连续完成。

### 15:30–16:30　B：替换凭证演示

#### 操作步骤

1. 回到凭证台账，在已经恢复为 active 的原始凭证上点击“更新”。
2. 输入一个简短的新课程名称，例如“数字身份与可信凭证训练营（进阶）”。
3. 确认台账出现一张新 ID 的 active 凭证。
4. 指出原始凭证已经变为 replaced。
5. 打开新旧凭证，展示 `replaces` 与 `replacedBy` 关系；若时间不足，只展示台账状态和 ID。

#### 建议讲述

凭证一旦签名，就不应该直接修改原始内容。因此系统中的“更新”实际采用替换模式：使用修正后的课程信息签发一张新 ID 的 VC，再把旧凭证标记为 replaced。

新凭证保存 `replaces`，旧记录保存 `replacedBy`，这样既能使用新凭证，也保留了完整历史关系。旧凭证的签名仍能证明它曾经被签发，但状态检查不再通过。

#### 代码依据

`replaceCredential()` 调用 `issueCredentialIntoState()` 创建新凭证，然后将旧记录改为 replaced。`VC_TRANSITIONS` 把 replaced 定义为终态。

### 16:30–17:20　B：撤销凭证并再次验证

#### 操作步骤

1. 在凭证台账找到刚生成的新 active 凭证。
2. 点击“撤销”并确认不可逆提示。
3. 打开新凭证，将其载入验证页。
4. 执行验证，指出签名通过、凭证状态 revoked、整体失败。

#### 建议讲述

最后，我们撤销替换后生成的新凭证。撤销是不可逆终态。再次验证时，原始数据未被修改，所以签名仍能通过；但状态已经是 revoked，凭证整体无效。

现在一条完整业务链已经形成：原始凭证先经历正常、暂停、恢复，再被新凭证替换；新凭证最终被撤销。系统保留整个过程，而不是覆盖历史数据。

### 18:40–19:35　B：AI 协作过程与学习反思

#### 建议讲述

在开发阶段，AI 主要帮助我们把设计拆成可执行任务、生成实现草案、补充异常场景和测试用例，并在测试失败后分析可能原因。项目提交记录可以看到，我们经历了“先设计和计划，再实现，再测试，再修复”的多轮迭代，例如补充完整业务旅程、安全回归、测试记录、JSON 请求限制、Base58 修正、存储并发修正，以及搜索分页体验优化。

AI 提高了信息整理和试错速度，但也带来三个需要人工把关的问题：

1. AI 可能把教学实现表述成正式标准实现；
2. AI 生成的代码可能在正常路径可用，但遗漏并发、错误输入或生命周期终态；
3. 测试数量多不等于结论一定可靠，还需要检查测试是否真的覆盖需求，并进行现场人工验收。

因此，我们逐渐形成了一套协作闭环：人提出问题和验收目标，AI 辅助分析与实现，自动化测试提供反馈，人再阅读代码、判断结果并修正需求或实现。

#### AI 协作流程

```text
提出业务问题
→ AI 辅助梳理概念和方案
→ 人工确定 MVP 与技术边界
→ AI 辅助拆解任务和生成草案
→ 人工审核、集成和运行
→ AI 辅助补充测试与分析失败
→ 自动化测试 + 人工演示验收
→ 修正代码、文档和理解
```

#### 学习收获建议

- 理解 DID 不是单一产品，而是标识符、DID Document 和 Method 规则的组合。
- 理解 VC 有效性需要同时考虑签名、签发方、密钥版本、时间和状态。
- 理解数字签名提供的是来源和完整性验证，而不是让文件无法被编辑。
- 学会把需求转化为状态机、接口约束和可自动验证的测试用例。
- 学会在 AI 输出与最终结论之间加入代码审查、测试和人工判断。

## 七、问答准备

### 1. 这个项目真的去中心化吗？

建议回答：

这是 DID/VC 的本地教学实现，展示了标识符、DID Document、公私钥签名和凭证验证等核心思想，但 DID 注册表和凭证状态仍保存在本地文件中，并不等同于生产环境的去中心化网络。我们在项目边界中明确说明了这一点。

### 2. 为什么没有使用区块链？

建议回答：

项目目标是理解 DID/VC 的数据结构、签名和生命周期。区块链只是某些 DID Method 的一种基础设施，不是 DID 和 VC 的必要条件。本地实现更适合在有限时间内观察完整流程。

### 3. 为什么撤销后签名还可能通过？

建议回答：

签名证明原始内容由对应私钥签发且没有改变。撤销改变的是凭证当前是否被认可，不会重写历史签名。因此系统必须把签名和状态作为两个独立检查项，最后综合判断。

### 4. 密钥轮换后旧 VC 为什么还能验证？

建议回答：

proof 保存了签发时的 `keyVersion` 和验证方法。轮换时，系统把旧公钥保存到 `keyHistory`，验证旧 VC 时可以根据版本找到历史公钥。历史私钥不会通过公开 API 返回。

### 5. 私钥保存在哪里？安全吗？

建议回答：

教学版本将私钥保存在本地 `data/store.json` 中，并通过 `publicDid()` 保证普通 API 返回时移除当前和历史私钥。这不属于生产级密钥托管；正式系统应使用硬件安全模块、密钥管理服务或安全钱包。

### 6. 如何证明不是只做了页面？

建议回答：

页面操作会调用真实 HTTP API，后端实际执行 Ed25519 签名、验签、状态转换和 JSON 持久化。项目还有分层自动化测试；当前 Node 测试实际运行 120 项并全部通过。

### 7. AI 在项目中占了多大作用？

建议回答：

AI 参与了概念整理、需求拆解、方案草拟、代码建议、测试补充和缺陷分析，但项目边界、功能取舍、代码集成、测试执行和最终验收由我们完成。我们也保留了需求、计划、测试和提交记录作为过程证据。

### 8. 项目与正式 W3C 标准有什么差异？

建议回答：

数据结构参考 DID Core 和 VC Data Model 2.0 的核心概念，但 proof 使用教学名称 `EducationalEd25519Signature2026`，没有实现正式注册的 Data Integrity cryptosuite，也没有生产级解析网络、状态列表、密钥托管和互操作认证。
