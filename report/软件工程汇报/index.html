01
REQUIREMENTS ANALYSIS

需求分析

传统电子证书可以展示，却很难独立回答“谁签发、是否被改、身份是否有效、凭证是否已失效”。本项目把这些疑问转化为可执行、可测试的软件需求。

问题定义

从“看证书”升级为“验证证书”

验证方不能只看一张图片，而要检查签发者身份、公钥、密码学签名、有效时间以及当前生命周期状态。

项目目标

构建本地可信凭证闭环

在不依赖数据库、区块链和外部网络的条件下，完整演示 DID 创建、VC 签发、验证、篡改检测和状态失效。

did:example

注册型、可变化

支持修改名称和 HTTPS 服务地址，使用 expectedVersion 检查并发冲突；轮换时保留历史公钥；停用后不可恢复。

did:key

公钥衍生型、不可变

Ed25519 公钥添加 multicodec 前缀后进行 Base58btc 编码，标识与密钥绑定，因此不支持更新、轮换和停用。

VC 生命周期状态机

人工操作由 VC_TRANSITIONS 限制；过期由有效期自动推导
stateDiagram-v2
  [*] --> active: 签发
  active --> suspended: 暂停
  suspended --> active: 恢复
  active --> replaced: 替换
  suspended --> replaced: 替换
  active --> revoked: 撤销
  suspended --> revoked: 撤销
  active --> expired: validUntil 到期（自动推导）
  suspended --> expired: validUntil 到期（自动推导）
  replaced --> [*]
  revoked --> [*]
  expired --> [*]
关键含义：暂停是可逆状态；replaced、revoked 和 expired 是终态。expired 不是单独的人工操作，而是验证或读取时根据 validUntil 推导出的有效状态。密码学签名可能仍然正确，但只要凭证状态不是 active，最终验证结果仍然失败。

七项验证规则

01

格式完整性

02

本地注册表能否找到签发方。

03

DID 状态

签发方是否存在且未停用。

04

密钥版本

能否解析当前或历史公钥。

05

Ed25519 签名

验证来源并检测内容篡改。

06

有效期

当前时间是否处于有效区间。

07

本地记录是否仍为 active。

RESULT

七项全部通过才返回 valid=true。

IMPLEMENTATION

编码实现

代码实现不只追求“功能能跑”，还处理确定性签名、并发更新、私钥公开边界、日志脱敏和 HTTP 输入约束。

src/crypto.js稳定签名载荷
export function stableStringify(value) {
  if (value === null || typeof value !== 'object')
    return JSON.stringify(value);
  if (Array.isArray(value))
    return `[${value.map(stableStringify).join(',')}]`;
  return `{${Object.keys(value).sort()
    .map(key => `${JSON.stringify(key)}:${stableStringify(value[key])}`)
    .join(',')}}`;
}

为什么不能直接 JSON.stringify？

  1. JSON 对象的键顺序可能不同，但业务内容相同。
  2. 签名针对字节序列；字节顺序不同会导致验签失败。
  3. stableStringify 递归排序对象键，使相同内容得到确定的签名字节。
  4. 数组保持原顺序，因为数组顺序通常具有业务含义。
src/store.js原子写入
async writeAtomic(state) {
  await mkdir(path.dirname(this.filePath), { recursive: true });
  const temporaryPath = `${this.filePath}.tmp`;
  await writeFile(temporaryPath,
    `${JSON.stringify(state, null, 2)}\n`, 'utf8');
  await rename(temporaryPath, this.filePath);
}

存储可靠性如何保证？

  1. 先写临时文件，完成后再整体替换正式文件，降低半写 JSON 风险。
  2. Promise 事务队列串行执行“读取—修改—保存”。
  3. 避免两个并发请求读取同一旧状态后互相覆盖。
  4. 这适合单机教学场景，但不能替代生产数据库事务。
公开投影

publicDid()

从身份对象中移除当前 privateJwk，并递归移除 keyHistory 中的历史私钥。浏览器只获得验签所需的公开数据。

日志脱敏

redact()

private、secret、token、password、authorization、cookie、proofValue、credential 和请求体在持久化前统一替换。

src/crypto.js加盐声明摘要
createHash('sha256')
  .update(stableStringify({ path, salt, value }))
  .digest('base64url');

为什么必须有随机盐？

  1. 只对姓名、状态等低熵字段直接哈希，容易被枚举猜测。
  2. 每个声明使用 16 字节随机盐，同值也得到不同摘要。
  3. 披露时只提交选中字段的盐；未披露字段的盐留在本地。
  4. Issuer 对完整摘要清单签名，防止替换摘要。

从 BDD、SDD、TDD 反向审视项目

学习反思：老师上课说过 BDD、SDD、TDD。我们从这个角度反向思考项目：系统以 SDD（规格驱动开发）为主线，用需求表、状态机、数据模型和接口契约先定义“系统应当怎样”；测试设计采用 BDD（行为驱动）思维，把用户可观察的业务规则写成场景；在新增功能和修复缺陷时，补充失败用例、实现代码并回归，因而具有局部 TDD（测试驱动开发）特征，但不宣称全项目严格按 TDD 从零开发。
方法本项目如何体现可复核证据
SDD：规格驱动先定义 DID/VC 角色、七项验证、VC 状态机、字段结构和 REST 接口契约。需求表、ER 图、时序图、VC_TRANSITIONS、API 测试。
BDD：行为驱动用“暂停后验证失败、恢复后成功”“篡改后签名失败”“撤销后不可恢复”等用户可观察结果描述需求。生命周期旅程、端到端场景、选择性披露验证台账。
局部 TDD：测试驱动迭代例如修复“暂停 VC 仍可生成披露”时，先增加教学版与 SD-JWT 都应拒绝的测试，再加入状态校验并回归。集成/API 回归用例；当前 Node 自动化测试 134/134 通过。
