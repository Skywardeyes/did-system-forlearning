SECURITY REMEDIATION

重新定义“管理员能看到什么”

DID Document 中的公钥本来就应公开；真正需要保护的是私钥、VC 个人声明、披露盐和敏感访问轨迹。整改围绕最小权限和可追责展开。

对象默认展示保护方式原因
DID 与公钥可以公开DID Document 解析验证方必须取得公钥才能验签
Holder DID 私钥禁止公开与上传个人钱包 IndexedDB + 不可导出 Web Crypto CryptoKey已自托管；平台仅登记公开 DID Document
VC 非敏感摘要按权限展示专用投影查询支持状态、时间和生命周期管理
VC 完整正文默认隐藏独立角色 + 用途 + 审计含姓名、课程等个人声明
披露盐与材料禁止公开加密存储防止反推未披露声明

职责分离

`tenant_admin` 负责组织和审计，但不再自动拥有 Issuer、Holder、Verifier 或完整正文读取权限。

重点验证的不是“成功路径”

失败时也必须保持安全

越权读取

没有 `credential_data_reader` 时必须返回 403，租户管理员不能绕过权限查看正文。

审计失败

台账无法写入时不允许先返回明文，确保访问行为与证据记录原子一致。

生命周期失效

暂停或撤销的凭证不能继续生成披露证明，并在验证阶段明确失败。

06
SELF-CUSTODY MVP

答辩反馈已落地：Holder 私钥回到个人钱包

当前 MVP 已把 Holder DID 的生成与出示签名移到独立钱包：浏览器 Web Crypto 生成不可导出私钥，平台只登记公开 DID Document，不能读取或代签 Holder 私钥。

已落地的边界与尚未夸大的部分

机构 Issuer 私钥仍由 KMS 管理；钱包交付包只含 VC 与 Issuer 签名材料；最小披露由钱包本地签名并绑定 Challenge 和验证方域名。验证方只把 Challenge 的 SHA-256 哈希写入台账，并在首次成功验证时原子消费，重放会失败。正式 DID Registry、跨设备恢复和标准 SD-JWT Holder Binding 仍是下一阶段，绝不保存个人私钥、VC 明文或个人信息到链上。

接下来三项工作

按风险和迁移依赖排序

个人钱包 MVP（已完成）

本地生成 Holder 私钥和 DID；私钥不上传，钱包本地保存 VC、选择字段并生成 Holder 签名的披露证明。

签发、验证平台拆分

Issuer 使用机构 HSM/KMS；Verifier 只验证最小披露并默认不留存完整 VC。

DID 注册与信任治理

抽象 DID Registry，先做本地/联盟账本适配，再锚定版本、状态和受信任机构目录。
